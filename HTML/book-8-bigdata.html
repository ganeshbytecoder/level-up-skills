SPARK INTERNALS: DAG → STAGES → TASKS → OUTPUT
ARCHITECTURE — WHAT RUNS WHERE: Driver: your spark-submit process. Builds DAG. Coordinates. 1 per application. SparkContext → DAG Scheduler → Task Scheduler Cluster Manager: YARN / Kubernetes / Standalone — allocates Executors Executor: JVM processes on worker nodes. Run Tasks. Store cached data. Each Executor: N cores (N tasks in parallel) + configurable memory FROM spark.sql() TO CLUSTER EXECUTION — every step: 1. Your code: spark.read.parquet("s3://datalake/orders/") .filter(col("amount") > 100) .groupBy("country").agg(sum("amount")) 2. Unresolved Logical Plan: parsed AST (column names not yet resolved) 3. Analyzed Logical Plan: Catalog resolves column names, types, table schemas 4. CATALYST OPTIMIZER (rule-based + cost-based): - Predicate pushdown: push .filter() DOWN into Parquet reader → read fewer row groups - Column pruning: only read "amount", "country" columns from Parquet → 10× less I/O - Join reordering: smaller table first (cost-based using table statistics) - Constant folding: 1 + 1 evaluated at plan time, not per-row 5. Physical Plan: chosen execution strategy (BroadcastHashJoin vs SortMergeJoin) 6. TUNGSTEN: generates optimized JVM bytecode (whole-stage code generation) 10-100× faster than interpreted code. Operates on binary format, avoids GC pressure. 7. DAG cut into Stages at SHUFFLE BOUNDARIES: Stage 1: read parquet → filter → partial aggregation (map-side combiner) --- SHUFFLE --- ← rows with same country go to same partition Stage 2: reduce aggregation → write output SHUFFLE — THE PERFORMANCE KILLER: All rows must be partitioned by key → network transfer across all executors. 200 tasks × 200 tasks = 40,000 network connections. Disk I/O + network + re-read. Cost: minutes to hours for large datasets. How to MINIMIZE shuffles: a) Broadcast join (small table <10MB): spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10485760") small_df broadcast to ALL executors. Large table never moves. Zero shuffle. b) Partition on join key before joining: pre-bucket tables by customer_id. Two tables bucketed same way? → zero shuffle join (bucket pruning + local join) c) Reduce spark.sql.shuffle.partitions: default=200. For small data → 200 empty tasks. With AQE: automatically coalesces small post-shuffle partitions. DATA SKEW — The Most Common FAANG Interview Problem: Problem: 99% of orders belong to country="US". 1 partition gets all of them. 199 tasks finish in 5 min. Task 200 runs for 6 HOURS. Job blocked on 1 task. Solution 1: SALTING Add random salt: country_salted = country + "_" + FLOOR(RAND() * 10) Now "US" splits into ["US_0","US_1",...,"US_9"] → 10× more even distribution Final aggregation: strip salt → sum up sub-totals. 2-pass but 10× faster end-to-end. Solution 2: ADAPTIVE QUERY EXECUTION (AQE — Spark 3.0+): spark.sql.adaptive.enabled = true AQE re-plans at runtime based on actual partition sizes after shuffle: - Skew join optimization: detects skewed partitions → splits them automatically - Coalesces small partitions → reduces task count (200 → 20 for small data) - Dynamically switches BroadcastJoin ↔ SortMergeJoin based on actual size Solution 3: SKEW HINT df.hint("skew", "country") → Spark AQE splits the US partition into sub-tasks MEMORY TUNING: Total Executor memory 8GB breakdown: 300MB: JVM overhead (reserved for OS, metaspace) Remaining 7.7GB × 0.6 (spark.memory.fraction = 0.6) = 4.6GB Spark managed: 4.6GB × 0.5 (spark.memory.storageFraction): 2.3GB Storage (cached RDDs) 4.6GB × 0.5: 2.3GB Execution (shuffle, sort, join) (Execution can borrow from Storage if idle, and vice versa — unified memory model) Remaining 7.7GB × 0.4: 3.1GB User memory (your data structures, UDFs) GC Tuning: G1GC for large heaps. spark.executor.extraJavaOptions=-XX:+UseG1GC Off-heap: project Tungsten stores shuffle data off-heap → no GC pressure on binary data
🔄
Transformations vs Actions
Transformations (lazy): map, filter, join, groupBy — build the DAG but don't execute. Actions (trigger execution): count(), show(), write, collect(). Only then does Spark submit stages to the cluster. This laziness enables Catalyst to optimise the full plan before any data moves.
💾
Caching Strategy
df.cache() or df.persist(StorageLevel.MEMORY_AND_DISK). Cache AFTER expensive shuffle operations that are reused. Never cache input data (only read once). Checkpointing: breaks long lineage (streaming) — truncates DAG and saves snapshot to HDFS. Use for iterative algorithms (GraphX, ML).
📊
Partitioning Rules
Default shuffle partitions: 200. Rule: target 128MB–256MB per partition. Too few partitions → OOM. Too many → scheduling overhead. Repartition: df.repartition(400, col("country")) — full shuffle, use before heavy operations. Coalesce: reduce partitions, no shuffle, use before writing.
📁
Small Files Problem
1000 tasks each write 1MB files = 1000 tiny Parquet files on S3. Next job reading them: 1000 S3 LIST operations + 1000 parallel reads = slow. Fix: df.coalesce(20).write.parquet(path) before writing. Alternatively: periodic compaction job (Iceberg OPTIMIZE, Delta OPTIMIZE) merges small files.
⚡
Broadcast Join
If one table < spark.sql.autoBroadcastJoinThreshold (default 10MB): driver sends it to EVERY executor's memory. Large table never moves. O(N) instead of O(N log N) sort-merge. WARNING: broadcast threshold should be <20% of executor memory. Too large → OOM on a 200-way broadcast.
🔬
Speculative Execution
spark.speculation = true: if a task runs 1.5× longer than median of its stage peers, Spark launches a duplicate on another node — whichever finishes first wins. Handles slow nodes (hardware issues, GC pause). Risk: for non-idempotent external writes, speculative tasks may double-write. Disable for output stages.

FLINK WATERMARKS + EXACTLY-ONCE — The Most Asked Flink Question
FLINK vs SPARK STREAMING — FUNDAMENTAL DIFFERENCE: Spark Structured Streaming: micro-batch. Collects events for 0.1–1s, processes as tiny batch. Latency: 100ms–1s. Good enough for most. Familiar batch semantics. Not "true" streaming: each micro-batch is a bounded Spark job. Apache Flink: record-at-a-time streaming. Each event triggers computation immediately. Latency: <10ms. Critical for: fraud detection, algorithmic trading, real-time recommendations. True streaming: operators run continuously. State persisted between events. EVENT TIME vs PROCESSING TIME: Event A: sensor fires at 12:00:00 PM (event time) Event A arrives at Flink at 12:00:07 PM (processing time) — 7s network delay Processing time window (12:00 - 12:01): would MISS event A entirely (arrived before window check). Event time window (12:00 - 12:01): correctly placed in 12:00 window based on when it HAPPENED. Event time is always correct. Processing time is always wrong for out-of-order events. WATERMARKS — How Flink decides "the window is complete": Watermark = timestamp below which we believe all events have arrived. Watermark(T) = max_event_time_seen_so_far - max_out_of_orderness Example: max_out_of_orderness = 10 seconds Events seen with timestamps: 12:00:05, 12:00:12, 12:00:18 Watermark after 12:00:18 = 12:00:18 - 10s = 12:00:08 Window 12:00:00-12:00:10 closes when watermark passes 12:00:10 → Watermark = 12:00:11 → window FIRES → computes aggregate, emits result Setting max_out_of_orderness too low → events arrive after window closed = data loss Setting it too high → huge state stored, result latency increases LATE DATA policy (after watermark passes): allowedLateness(Duration.ofMinutes(5)) → window stays open for 5 more minutes Events in that window: trigger UPDATES to already-emitted result (retract streams) After 5min: events go to side output stream (you decide: reprocess or DLQ) WINDOW TYPES: Tumbling Window: |--5min--|--5min--|--5min--| Fixed, no overlap. Count events per 5min bucket. Sliding Window: |--10min--| sliding by 2min. Adjacent windows overlap. Session Window: |--activity--|gap >30min|--activity--| Closes after 30min of inactivity. Perfect for: user session analytics, shopping cart abandonment. Global Window: One window for all time. Must define custom trigger to emit results. EXACTLY-ONCE — Flink + Kafka (2-Phase Commit Protocol): Flink achieves exactly-once via CHECKPOINTING + Kafka Transactions: Checkpoint N starts: Flink injects BARRIER into each Kafka partition (a special marker message) Each operator: when barrier from ALL input partitions received → snapshot its state to S3 Kafka sink: begin Kafka transaction (txn_id = "checkpoint-N") During checkpoint: All outputs buffered in Kafka transaction (not yet committed = invisible to consumers) Operators continue processing (non-blocking checkpoint → low latency impact) Checkpoint N completes (all operators confirmed snapshot): Flink calls notifyCheckpointComplete() Kafka sink: COMMIT transaction → records become visible to downstream consumers ATOMICALLY CRASH during checkpoint N: Transaction never committed → Kafka automatically aborts → records invisible Flink restores from checkpoint N-1 → re-reads Kafka from committed offset N-1 Result: each record processed exactly once end-to-end. No duplicates. No data loss. STATE BACKENDS: MemoryStateBackend: state in JVM heap. Max ~1GB. Dev/test only. FsStateBackend: state in heap, checkpoints to HDFS/S3. Good for medium state. RocksDBStateBackend: state on LOCAL SSD (RocksDB LSM tree). Checkpoints to S3. Supports TERABYTES of state per TaskManager. Used by Uber, LinkedIn, Alibaba at PB scale. Trade-off: 2-5× slower than heap state (serialization + disk I/O). Tunable. Incremental checkpoints: only changed state blocks uploaded, not full snapshot. 10× faster.
🎭
CEP — Complex Event Processing
Flink CEP detects patterns across event sequences in real-time. Example fraud rule: Pattern: 3 transactions > $1000 within 5 minutes from same user. Flink CEP uses NFA (non-deterministic finite automaton) internally. Sub-10ms latency per event. Used by: PayPal fraud, Alibaba risk engine, Uber trip anomaly detection.
⚖️
Backpressure Handling
Flink's network stack is credit-based: downstream operator grants upstream "credits" (buffer space available). If downstream slow → upstream stops sending → backpressure propagates up to source (Kafka consumer slows down read rate). Natural throttling. No data loss. Monitor: Flink UI shows task backpressure % per operator.
🔀
KeyedStream Parallelism
stream.keyBy(event -> event.getUserId()) — all events for same key go to same subtask (consistent hashing). State is per-key, isolated. Auto-scaled: adding parallelism = redistribute keys across more subtasks. RocksDB: stores state as key-value, supports millions of distinct keys per subtask.

KAFKA INTERNALS: What Happens When You Produce a Message
PARTITION PHYSICAL LAYOUT (on broker disk): /kafka-logs/orders-0/ ← partition 0 of "orders" topic 00000000000000000000.log ← segment file: records as byte stream 00000000000000000000.index ← sparse offset index: offset → file position (every 4096 bytes) 00000000000000000000.timeindex ← timestamp → offset index for time-based seek 00000000000000012345.log ← next segment (created when active segment exceeds 1GB or 7 days) Seek to offset 50000: Binary search in .index for nearest entry ≤ 50000 → file position X Linear scan .log from position X until offset 50000 found = O(1)-ish PRODUCE FLOW (acks=all, exactly-once): Producer → finds leader broker for partition (from metadata cache) Producer sends batch: [{offset: auto, key: orderId, value: {…}, timestamp: now}] Leader broker: 1. Write to local log (sequential append, write-ahead) 2. Followers fetch from leader (replication) – all ISR replicas must ack 3. Leader updates HW (High Watermark) = min(all ISR leo) 4. Sends ack to producer ISR (In-Sync Replicas): followers within replica.lag.time.max.ms (default 10s) of leader If follower falls behind → removed from ISR → no longer blocks acks acks=all with min.insync.replicas=2: need at least 2 ISR replicas to ack. Durability guarantee. CONSUMER GROUP REBALANCING — The Pain Point: 3 consumers, partition 0,1,2 assigned: C0→P0, C1→P1, C2→P2 Consumer C0 dies → rebalance triggered EAGER (old default): all consumers STOP. Drop all assignments. Reassign from scratch. Called "stop-the-world rebalance." 30-60 seconds of processing halt. Bad for production. COOPERATIVE STICKY (modern): incremental rebalance. Only partitions from dead consumers are reassigned. Other consumers keep processing their partitions during rebalance. 2-round protocol: Round 1: consumers report capabilities. Round 2: minimal reassignment. Config: partition.assignment.strategy=CooperativeStickyAssignor Group coordinator (a broker) manages assignment via heartbeat protocol (session.timeout.ms) EXACTLY-ONCE WITH KAFKA TRANSACTIONS: Producer creates transactional session: transactional.id = "order-processor-1" BEGIN_TRANSACTION producer.send("order-events", orderId, payload) ← message in transaction producer.send("inventory-updates", orderId, delta) consumer.commitOffset(topicPartition, offset) ← consumed offset included in transaction COMMIT_TRANSACTION Consumers with isolation.level=read_committed: only see committed transaction messages. If crash before COMMIT: transaction expires (transaction.timeout.ms) → auto-aborted. Idempotent producer (enable.idempotence=true): deduplicates within single producer session. Combining: transactional + idempotent = exactly-once end-to-end within Kafka. SCHEMA REGISTRY + AVRO — Schema Evolution Rules: Producer serializes: Avro bytes + magic byte + schema-id (4 bytes) = Kafka message value Consumer: reads schema-id → fetches schema from Registry → deserializes Registry enforces compatibility rules: BACKWARD: new schema can read data written by old schema (add fields with defaults) FORWARD: old schema can read data written by new schema (remove fields with defaults) FULL: both BACKWARD + FORWARD BREAKING (reject): rename field, change type, remove required field → registry rejects commit Result: zero coordination needed between producer and consumer teams for safe schema changes.

PARQUET INTERNALS — How 1TB Scan Becomes 10GB Read
ROW-BASED (CSV/JSON) vs COLUMNAR (Parquet/ORC): Query: SELECT SUM(amount) FROM orders WHERE country = 'US' 100 columns, 10B rows. Row-based (CSV): Read ALL 100 columns × 10B rows = 100% of data = 10TB read. What you need: 2 columns × 10B rows = 0.2TB reading 2 columns. Columnar (Parquet): Read ONLY "amount" + "country" columns = 2% of data = 200GB read. 50× less I/O. Queries complete 50× faster (or use 50× less cost on cloud). PARQUET PHYSICAL LAYOUT: File: ├── Row Group 1 (128MB default) │ ├── Column Chunk: "country" (all country values for rows 1-1M) │ │ ├── Page 1 (1MB): [US,US,DE,US,UK,...] compressed + encoded │ │ ├── Page 2 (1MB): ... │ │ └── Page stats: min="AU", max="ZW", null_count=0 │ ├── Column Chunk: "amount" (all amount values for rows 1-1M) │ └── Column Chunk: "timestamp" (all timestamp values for rows 1-1M) │ └── Page stats: min=1672531200, max=1672617600 ← row group min/max ├── Row Group 2 (128MB) ... └── Footer (file metadata, row group stats, schema, offsets) PREDICATE PUSHDOWN — 3 Levels of Skip: Level 1: Row group skipping (file footer stats): Query: WHERE timestamp > 2024-01-15 Row Group 1 stats: max=2024-01-10 → ALL rows in group < filter → skip entire 128MB block Row Group 2 stats: min=2024-01-14, max=2024-01-20 → might contain matches → read Level 2: Page skipping (page stats within column chunk): Row Group 2, "timestamp" column, Page 1: max=2024-01-14 → skip page Row Group 2, "timestamp" column, Page 2: min=2024-01-15 → read this page Level 3: Bloom filters (optional, per column): Pre-computed: "does this row group contain country='JP'?" Bloom filter answer: NO (with 0% false negatives) → skip entire row group. INSERT into bloom filter at write time. O(1) lookup at read time. Result: 10TB file, reading only 50MB. 200× I/O reduction. ENCODING STRATEGIES (automatic in Parquet): RLE (Run-Length Encoding): [US,US,US,US,US,DE,DE] → [(US,5),(DE,2)] → 70% smaller Best for: low-cardinality columns (country, status enum) Dictionary encoding: build dictionary {0:US, 1:DE, 2:UK}. Store indices instead of strings. [US,DE,UK,US,US] → [0,1,2,0,0]. Integer indices compress much better than strings. Best for: string columns with <1000 unique values. Parquet default for strings. Delta encoding: timestamps [1000,1001,1002,1003] → [1000, Δ=1,1,1]. Nearly free with sorted data. FORMAT DECISION GUIDE: Parquet: OLAP queries, S3/HDFS storage. Columnar. Best for: Spark/Trino analytics. Avro: Message serialisation (Kafka events), row-based. Best for: streaming, schema evolution. ORC: Hive-optimised columnar. Built-in ACID (Hive transactions). Best for: Hive workloads. Delta/Iceberg: adds ACID + time travel ON TOP of Parquet. Best for: data lakehouse. COMPRESSION COMPARISON: Snappy: Fast compress + fast decompress. CPU cheap. Ratio: 2-3×. Default Parquet choice. Gzip: High compression ratio: 5-8×. Slow compress. Medium decompress. Use: cold archive. Zstd: Best of both: ratio like Gzip, speed like Snappy. Recommended for new pipelines. LZO: Splittable (Hadoop can split mid-file). Use only if Snappy splitting not supported.

ICEBERG METADATA HIERARCHY — How ACID Works on Object Storage
THE PROBLEM WITH HIVE-STYLE PARTITIONS: Hive table: s3://lake/orders/year=2024/month=04/day=19/*.parquet Hive Metastore: RDBMS storing partition locations (single point of failure) Problems: - No ACID: two writers → partially written partitions → corrupt reads - Schema changes: rename column → all old files still use old name → ETL breaks - No time travel: deleted data is gone. Compliance: "show me data from 3 months ago" → impossible - Bad filtering: WHERE order_date = '2024-04-19' requires knowing partition structure (SUBSTR tricks) - Small files: many appends → thousands of tiny files → slow listing on S3 ICEBERG METADATA STRUCTURE (on S3, no external metastore needed): Catalog (REST/Glue/Hive): points to current metadata file for table ↓ Metadata File (JSON): snapshot history, schema versions, partition specs ├── Current snapshot-id: 987654321 ├── Schema versions: [v1: {id,name,amount}, v2: {id,name,amount,country}] └── Snapshots: snapshot_987654321, snapshot_876543210, ... ↓ Snapshot File (JSON): list of manifest files for this version ├── manifest_001.avro (summary: 50M rows, data_file_count=100) └── manifest_002.avro (added 5M new rows via INSERT) ↓ Manifest File (Avro): list of data files with column-level statistics ├── s3://lake/orders/data/00001.parquet {min_date:2024-01-01, max_date:2024-01-31, rows:500K} ├── s3://lake/orders/data/00002.parquet {min_date:2024-02-01, max_date:2024-02-28, rows:480K} └── s3://lake/orders/data/00003.parquet {added in latest snapshot} ↓ Data Files: actual Parquet files (content never mutated after write) HOW ACID WORKS ON S3: S3 is eventually consistent (historically). Iceberg uses OPTIMISTIC CONCURRENCY: Writer 1: creates new data files → creates new snapshot → atomic PUT catalog pointer Writer 2: creates new data files → tries to PUT catalog pointer If catalog pointer changed since W2 started: CONFLICT → retry (exponential backoff) If catalog unchanged: W2 commits successfully Readers: always read a snapshot. Snapshot is immutable. Reading is always consistent. MERGE/UPDATE: copy-on-write (rewrite affected data files) vs merge-on-read (write delete files). TIME TRAVEL: Snapshots are kept (configurable retention). Query past state: SELECT * FROM orders FOR VERSION AS OF 987654321 -- specific snapshot SELECT * FROM orders FOR TIMESTAMP AS OF '2024-04-01' -- point-in-time Use cases: audit, rollback bad data, debugging, A/B comparison, reproducible ML training sets. ROLLBACK: UPDATE catalog pointer to point to past snapshot = instant table rollback. HIDDEN PARTITIONING (Iceberg innovation): Hive: WHERE order_date = '2024-04-19' requires query to know partition structure User must write: WHERE year=2024 AND month=4 AND day=19 → partition pruning works Else: full table scan. Users constantly make mistakes. Iceberg hidden partitions: partition_spec says "partition by DAYS(order_date)" Query: WHERE order_date = '2024-04-19' Iceberg: automatically applies DAYS() transform → prunes correct date partitions User never thinks about partitioning. Correct performance automatically. COMPACTION (mandatory maintenance): Streaming writes: 1000 tasks × 10 writes/hr = 10,000 tiny Parquet files/hr After 24hrs: 240,000 files. Each Spark task: list 240K files + open + read = 10 min setup. Iceberg OPTIMIZE / bin-packing: Scans all data files, rewrites into target-size (512MB) Parquet files. New snapshot created pointing to compacted files. Old files expire after retention period. Run as daily/weekly Spark job. Transparent to readers (snapshot-based isolation).
🆚
Iceberg vs Delta vs Hudi
Iceberg: Open standard (Apache). Multi-engine (Spark, Trino, Flink, Athena). Hidden partitioning. Best multi-engine interop. Netflix, Apple, LinkedIn. Delta Lake: Databricks-first. Best Spark integration. Z-ordering. Hudi: Uber-first. Upsert-optimised (record-level updates). Best for CDC workloads. All three support ACID + time travel.
🗑️
Row-Level Deletes
GDPR "right to be forgotten": DELETE FROM orders WHERE user_id = 12345. Without Iceberg: rewrite every Parquet file that contained that user = expensive. Iceberg: adds a delete file (position delete file: "row N in file X is deleted"). Query engine merges data + delete files at read time. Schedule periodic compaction to physically remove deleted rows.
📈
Z-Ordering / Sort Order
SORT ORDER BY(country, date) — data files physically sorted by these columns. Query WHERE country='US' AND date>'2024-01-01': skip entire files where min/max shows no overlap. Similar to data clustering in Snowflake. Iceberg: defined at table level, applied during compaction. Max skipping benefit for high-cardinality filters.

AIRFLOW ARCHITECTURE + PRODUCTION DAG PATTERNS
COMPONENTS: Webserver: Flask app. UI + REST API. Read-only view of Metadata DB. Scheduler: Parses DAG files every 30s. Creates TaskInstances in Metadata DB. Single point. Workers: Execute tasks. CeleryWorker (persistent) or KubernetesWorker (ephemeral pod per task). Metadata DB: PostgreSQL / MySQL. Stores DAG definitions, task states, logs, connections, variables. Executor: How tasks are dispatched: LocalExecutor (dev), CeleryExecutor (scale-out), KubernetesExecutor (cloud-native). SCHEDULE INTERVAL GOTCHA: schedule_interval="@daily" → runs at midnight. But for what day? Airflow schedule: data_interval_start + data_interval_end. A DAG scheduled for 2024-04-19 ACTUALLY RAN at 2024-04-20 00:00:00 UTC. (End of the interval = trigger time) This trips up EVERY beginner. Logical date != execution date. Modern (Airflow 2.2+): use data_interval_start and data_interval_end context variables. data_interval_start = "2024-04-19 00:00:00" ← what your task should process execution_date (deprecated) = logical_date = "2024-04-19" IDEMPOTENT DAG — Production Must-Have: Idempotent: running the same DAG run twice produces the same output. Safe to re-run. BAD (not idempotent): INSERT INTO daily_summary SELECT ... (runs twice = duplicate rows) GOOD: Option 1: DELETE + INSERT in same atomic transaction Option 2: INSERT ... ON CONFLICT (logical_date) DO UPDATE SET ... (upsert) Option 3: Overwrite partition in Parquet/Iceberg spark.write.mode("overwrite").partitionBy("date").parquet(path) Overwrites only the partition for this date. Re-running: same output. TASKFLOW API (Airflow 2.0+) — Modern Pattern: from airflow.decorators import dag, task from datetime import datetime @dag(schedule="@daily", start_date=datetime(2024,1,1), catchup=False) def orders_pipeline(): @task def extract() -> dict: return {"row_count": 1_000_000, "s3_path": "s3://lake/raw/orders/2024-04-19"} @task def transform(raw: dict) -> dict: # raw dict passed automatically via XCom return {"cleaned_path": "s3://lake/clean/orders/2024-04-19"} @task def load(clean: dict): # write to Iceberg pass load(transform(extract())) # dependencies declared by function call, not set_downstream() dag = orders_pipeline() BACKFILL — Processing historical data: New DAG deployed. Need to process last 90 days of data. airflow dags backfill --start-date 2024-01-01 --end-date 2024-04-18 orders_pipeline Airflow: creates 108 DAG runs (one per day). Executes in parallel (up to max_active_runs). Idempotency: running backfill on already-processed dates: upsert/overwrite = safe. KUBERNETES EXECUTOR (each task = isolated K8s pod): No persistent workers (vs CeleryExecutor with always-on workers). Scheduler submits task → creates K8s pod with task env → pod runs → terminates. Benefits: perfect isolation, custom Docker image per task, no resource contention between tasks. Overhead: pod startup time: 10-30s. Bad for tasks faster than 1 minute. KubernetesPodOperator: run ANY Docker image as a task. Spark, dbt, custom code. Maximum flexibility.

TRINO QUERY EXECUTION — From SQL to Distributed Result
ARCHITECTURE: Coordinator: single node. Parses SQL → optimises → creates distributed execution plan → dispatches to Workers. Workers: N nodes. Execute pipeline stages in parallel. Pull data from connectors. Connectors: pluggable. Hive (S3+Parquet/Iceberg), PostgreSQL, MySQL, Kafka, Elasticsearch, BigQuery. No persistent storage. Pure compute. All data read from connectors on demand. Key difference from Spark: Trino is pipeline-based (pipelined execution, no write-to-disk between stages). Spark: stage-by-stage with disk materialisation of shuffle. Trino: fully pipelined → 10-100× lower latency. Trade-off: Spark handles arbitrarily large datasets (spills to disk). Trino: memory-bound per query. FEDERATED QUERY — JOIN across different data sources: SELECT o.order_id, u.email, o.amount FROM iceberg.orders o ← Parquet files on S3 JOIN postgresql.users u ← live Postgres DB ON o.user_id = u.id WHERE o.amount > 1000 Execution plan: Stage 0 (Hive connector): Scan Parquet on S3 → filter amount > 1000 → partial table Stage 1 (PostgreSQL connector): Full scan users table from Postgres Stage 2 (Exchange): Hash join on user_id Build side (smaller): users table → distributed hash table across workers Probe side (larger): filtered orders → probe hash table per row Stage 3: Aggregate results → return to coordinator → send to client DYNAMIC FILTERING (Trino's killer optimisation): Query: WHERE user.country = 'US' AND orders JOIN users ON user_id Runtime: after reading users table, Trino knows all user_ids where country='US'. Bloom filter: built dynamically from actual user_id values at runtime. Pushed DOWN to the orders table scan (Hive connector). Parquet reader: checks if each row group's user_id range overlaps bloom filter. Skip 80% of Parquet row groups → orders scan reads 1TB instead of 10TB. This is pure runtime magic — no static predicate. Done automatically by Trino. TRINO vs SPARK — WHEN TO CHOOSE: Interactive analytics (<30s latency goal): → Trino (pipelined, no write overhead) ETL on 100TB+ (batch, latency doesn't matter): → Spark (handles huge shuffles via disk) Real-time SQL on Kafka stream: → Trino Kafka connector ML feature engineering: → Spark (MLlib, built-in model training) Federated query (multiple sources, one SQL): → Trino (connector model) Data lake UPSERT / MERGE INTO: → Spark + Iceberg

ZOOKEEPER INTERNALS — ZNodes, Watches and Leader Election
DATA MODEL — HIERARCHICAL KEY-VALUE STORE (like a filesystem): /kafka ← persistent ZNode /kafka/brokers ← persistent /kafka/brokers/ids ← persistent /kafka/brokers/ids/1 ← ephemeral! Vanishes when broker 1 disconnects. /kafka/controller ← ephemeral: whoever created this is the Kafka controller Ephemeral ZNodes: tied to client session. Session expires → ZNode deleted. Persistent ZNodes: survive client disconnection. Explicit delete required. Sequential ZNodes: /lock/lock-0000000001, /lock-0000000002 — monotonically increasing suffix. LEADER ELECTION RECIPE: 5 Kafka brokers want one of them to be Kafka Controller. All brokers attempt: CREATE /kafka/controller {brokerId=3} EPHEMERAL Only ONE succeeds (atomic in ZK). That broker is now leader. Others: set a WATCH on /kafka/controller (notify me if this ZNode changes) Leader broker crashes: 1. Session timeout detected by ZK (session.timeout.ms, default 18s) 2. ZK deletes /kafka/controller (ephemeral node) 3. ALL watchers notified: "/kafka/controller was deleted" 4. All brokers race to CREATE /kafka/controller again 5. One wins. New leader elected. Followers update watches. Herd Effect: 1000 brokers all wake on same event → thundering herd on ZK. Better: sequential lock-based election (each candidate only watches predecessor): Broker creates /election/lock-00000001, /lock-00000002, /lock-00000003 (sequential) Broker holding lowest number = leader. Others watch only the NEXT LOWER sequential node (not all watching same node). Lowest node deleted → only second-lowest woken up → it becomes leader → no herd. DISTRIBUTED LOCK (ZooKeeper recipe): Thread wants lock on "order-processor": 1. CREATE /locks/order-processor/lock-000000001 EPHEMERAL SEQUENTIAL 2. List children of /locks/order-processor/ 3. Is MY node the lowest sequence? YES → Lock acquired. Do work. NO → Watch the node with sequence-1 before mine. 4. WATCHED node deleted → Back to step 2 (re-check, don't assume lock is mine). 5. On work complete: DELETE my ephemeral node → next in queue wakes up. Fencing: if client crashes while holding lock → ephemeral node auto-deleted → next acquires. Problem: "zombie" process that was declared dead may resume. Use fencing tokens (monotonic number from ZK) to reject stale writes. KAFKA KRAFT — REMOVING ZOOKEEPER (Kafka 3.3+): Problem with ZK: separate 3/5 node ZK cluster to manage. Admin burden. ZK has 10ms+ write latency. KRaft: Kafka implements its own Raft-based metadata quorum. Controller quorum within Kafka brokers. Benefits: - No separate ZK deployment. Simpler operations. - Faster controller failover: ZK-based: 30s+. KRaft: sub-second. - Kafka can support millions of partitions (ZK was a bottleneck at ~200K partitions) Migration: Kafka 3.3+ fully ZK-free. Production migration tool provided.

MEDALLION ARCHITECTURE (Bronze → Silver → Gold) — Industry Standard at LinkedIn, Databricks, Netflix

RAW SOURCES          BRONZE (Raw)              SILVER (Cleaned)           GOLD (Business)
─────────────────    ──────────────────────    ─────────────────────────  ──────────────────────────
PostgreSQL CDC   ──► Kafka topic: orders       De-duplicated orders       orders_daily_summary (agg)
Kafka events     ──► Iceberg: bronze/orders   Parsed + typed              customer_360_view (joined)
S3 raw files     ──► (append-only, raw)       Validated + enriched        product_revenue (metrics)
API webhooks     ──► Avro format preserved    Conformed schema             ML feature store
                     No transformations        Parquet + Iceberg           Parquet + Iceberg
                     Ingested by Flink         Spark batch + Flink stream   Spark batch
                     Retention: 90 days        Retention: 5 years          Retention: 2 years

LATENCY:             < 5 seconds              < 15 minutes                  < 1 hour

LAMBDA ARCHITECTURE:   Batch Layer (Spark, 6hr latency) + Speed Layer (Flink, 1s latency) → Serving layer merges
  When to use: batch and streaming need different logic. Historical recomputation needed.
  Problem: TWO codebases to maintain. Complex merge logic. Often out of sync.

KAPPA ARCHITECTURE:    Single streaming path (Flink / Kafka Streams). Historical replay = reprocess from Kafka.
  When to use: same logic for batch and streaming. Kafka retention > computation window.
  Advantage: one codebase. Simpler. Kafka as immutable event log.
  Limitation: infinite Kafka retention = expensive. Large replay takes time (vs Spark's optimised batch).

CDC (Change Data Capture) — Debezium pattern:
  PostgreSQL WAL (Write-Ahead Log) → Debezium → Kafka topic: postgres.public.orders
  Every INSERT/UPDATE/DELETE becomes a Kafka event. No polling. Sub-second latency.
  Iceberg: CDC consumer updates Iceberg table (MERGE INTO = upsert). Near-real-time sync.
  Use: replace nightly database dumps. Power real-time analytics on operational data.

DATA MESH (Organizational Pattern, not a technology):
  Problem: central data team becomes bottleneck for 50 product teams.
  Data Mesh: each product domain owns its OWN data as a "data product."
    Orders team owns + publishes: orders_clean (SLA: <15min freshness, 99.9% uptime)
    Payments team consumes orders_clean without depending on central team.
  Enabling technology: data catalog (Apache Atlas, DataHub) + Iceberg (multi-team access)
    + unified governance (column-level access control, centralized quality checks)

10 CRITICAL PRODUCTION SCENARIOS — Cause + Diagnosis + Fix
SCENARIO 1: Spark job runs for 6 hours then fails at 99% Symptom: 1 task of 200 running for hours. Other 199 finished in 5min. Diagnosis: Spark UI → Stages → find hanging task → check shuffle read bytes. One partition: 50GB. Others: 200MB. Classic data skew. Fix: salting + AQE skew join hint. Check for NULL keys (all NULLs hash to same partition). SCENARIO 2: Spark OOM — java.lang.OutOfMemoryError in executor Symptom: "Lost task 5.0 in stage 3.0: ExecutorLostFailure (executor 2 exited)" Causes: a) join of two large tables in-memory, b) too many partitions too small (GC overhead) c) UDF using Python with large objects, d) collect() pulling 1B rows to driver Fix: a) increase spark.executor.memory or use disk spill. b) reduce shuffle partitions. c) use Pandas UDFs with Arrow (10× faster GC). d) NEVER collect() at scale. Use write(). SCENARIO 3: Kafka consumer lag growing — pipeline falling behind Symptom: consumer_group_lag metric 0 = healthy. Growing lag = pipeline can't keep up. Causes: a) slow processing (N+1 external API calls per message) b) too few partitions (can't add more consumers than partitions) c) GC pauses causing consumer to miss heartbeat → rebalance → processing stops for 30s Fix: a) batch external calls (group 100 messages, one API call). b) increase partitions (irreversible). c) tune G1GC: -XX:MaxGCPauseMillis=100. Increase heartbeat interval. Use CooperativeStickyAssignor. SCENARIO 4: Airflow DAG failing intermittently — flapping heartbeat Symptom: DAG runs complete but some tasks show "zombie task not running on worker" Cause: scheduler heartbeat timeout. Long-running tasks don't check in (job_heartbeat_sec = 5s default) Fix: increase [scheduler] scheduler_heartbeat_sec. Fix: ensure workers don't OOM. Better: KubernetesPodOperator with resourceLimits → isolated, immune to other task OOM. SCENARIO 5: Trino query OOM — "Query exceeded per-node memory limit" Symptom: trino.execution.MemoryRevokingSchedulerStatus: "running out of memory" Cause: large join, no spill to disk configured (Trino defaults: no spill) Fix: SET SESSION spill_enabled=true. SET SESSION max_revocable_memory_per_node='16GB'. Prevention: always define query_max_memory in cluster config. Reject queries above limit. SCENARIO 6: Iceberg compaction not running — 10 million tiny files Symptom: Trino/Spark queries on Iceberg table take 20 minutes just to plan (listing S3 files). Cause: streaming Flink job writing 1 file per task per checkpoint (every 5 minutes). Multiply. Fix: run Airflow compaction DAG nightly: spark.sql("CALL iceberg.system.rewrite_data_files(table=>'orders', options=>map('target-file-size-bytes','536870912'))") Prevention: Flink sink: checkpoint interval 15min (not 5), larger target file size. SCENARIO 7: Parquet query slow — predicate pushdown not working Symptom: WHERE status='COMPLETED' query scans all 10TB. Should scan 200GB. Cause: wrong partition column (partitioned by date, filtering by status). Row group stats useless if status values span all groups. Fix: z-order/sort on status column during compaction → data clustering → row group stats now useful. Or: add status as an Iceberg partition transform → separate data files per status value. SCENARIO 8: Flink job checkpoint failing repeatedly Symptom: checkpoint timeout (default 10min). Job continues but no recovery point. Causes: a) RocksDB checkpoint upload to S3 too slow (state is 500GB, bandwidth 100MB/s = 83min) b) Backpressure: upstream operator so slow that buffers fill, barriers can't flow Fix: a) enable incremental checkpoints (only upload changed SST files). b) fix slow operator, not checkpoint timeout. Monitor: Flink UI → checkpoints tab → checkpoint duration per subtask. SCENARIO 9: Schema evolution breaks downstream job Producer adds new REQUIRED field to Avro schema. Old Kafka messages have no value for it. Flink consumer reading old messages: fails to deserialize. Job crashes. Fix: ALWAYS add new fields with a DEFAULT value. Schema Registry: enforce BACKWARD compatibility. BACKWARD means new schema can read data written with old schema. Old messages still valid. SCENARIO 10: Zookeeper quorum lost — Kafka cluster goes offline 3-node ZK: ZK1 dies, ZK2 dies. ZK3 alone cannot form quorum (needs majority > N/2). Kafka: controller cannot talk to ZK → cluster freezes. No new producer/consumer connections. Fix: replace failed ZK nodes. Restore quorum. Kafka detects ZK reconnect → resumes. Prevention: 5-node ZK (tolerates 2 failures). Or: KRaft mode (no ZK dependency). Alert: zookeeper_outstanding_requests → page if > 10 for > 1 minute.
