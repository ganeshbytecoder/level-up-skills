Hardware Communication Protocols

The languages of embedded systems, IoT, robotics, and edge AI — decoding how devices talk to each other.

UART, I2C, SPI USB, CAN, Ethernet
"Think of hardware communication protocols like languages between electronic devices. Just like humans use English or Spanish, devices use UART, I2C, or SPI. Each protocol solves a different problem of speed, complexity, and scale."
Prologue

The Big Picture & Core Ideas

Every hardware communication system is fundamentally built on three core ideas:

  • Data: The actual information being sent (the bits).
  • Clock: Timing synchronization so the receiver knows when to read the bits.
  • Wires: The physical connection carrying the signals.

Historically, systems used Parallel Communication, where many bits were sent together over multiple wires (e.g., 8 wires = 8 bits at the same time). It was fast but expensive, requiring bulky cables and complex routing. Old computers used this heavily.

Modern protocols use Serial Communication, where bits are sent one after another over just one or two data wires (e.g., 10110101). It uses fewer wires, saves space, and thanks to modern speeds, is incredibly fast. ALL modern protocols in this book (UART, I2C, SPI, USB, Ethernet) are serial.

Lesson I

UART — The Simplest Protocol

Universal Asynchronous Receiver/Transmitter

UART is a very simple serial communication protocol between two devices. It has no shared clock (asynchronous). Used heavily in microcontrollers, debugging, GPS modules, Bluetooth modules, and embedded Linux boards (like Raspberry Pi/Arduino).

Mental Model: Imagine two people talking directly to each other. No manager. No network. No addressing. Just You ↔ Friend.

🔌How it works & Wiring
Uses just 2 data wires plus Ground:
• TX (Transmit) → connects to RX of the other device
• RX (Receive) → connects to TX of the other device
• GND (Ground) → Shared reference voltage

Why cross them? Because your mouth (TX) talks to another person's ears (RX).
FeatureDetails
TypeSerial, Asynchronous (No shared clock)
Wires2 (TX, RX) + GND
SpeedLow to Medium (Baud rates: 9600, 115200, 921600)
ComplexityVery Simple
Multi-deviceNo (Point-to-point only)

How Data Travels: If you send the letter 'A', the computer converts it to binary (01000001). UART sends these bits one-by-one.

Baud Rate: Since there is no shared clock wire, both devices MUST agree on the speed (Baud rate) in advance. If one speaks at 9600 bits/sec and the other listens at 115200 bits/sec, the data becomes complete garbage (HELLO becomes #@!$%).

Example: An Arduino reading from a GPS module. The GPS module continuously sends coordinates (Latitude, Longitude, Time) over its TX pin, and the Arduino reads them via its RX pin.

In Linux: UART interfaces often show up as device files like /dev/ttyUSB0. A USB-to-UART adapter is a very important tool for connecting your PC to UART devices.

import serial
# Python UART Example
ser = serial.Serial('/dev/ttyUSB0', 115200)

while True:
    data = ser.readline()
    print(data)
Lesson II

I2C — Many Devices, Two Wires

Inter-Integrated Circuit

Created by NXP Semiconductors, I2C is a protocol for connecting multiple chips on the same board using only two wires. It is the go-to protocol for sensors, RTC clocks, EEPROMs, small OLED displays, and battery controllers.

Mental Model: Imagine a classroom. The teacher (Master) says: "Student 0x68, answer me." Only that specific student (Slave) responds. All other students ignore the request, even though they all heard it in the same room.

🔌How it works & Wiring
Uses exactly 2 wires for everything:
• SDA (Serial Data) → Sends the actual bits
• SCL (Serial Clock) → Synchronizes the timing

Instead of needing 2 wires per device, you connect dozens of devices to the exact same SDA and SCL lines (the "bus"). This massively simplifies PCB routing.
Note: I2C requires pull-up resistors on both lines so the bus defaults to HIGH. Devices only pull the signal LOW. Without pull-ups, the bus floats and communication fails.
FeatureDetails
TypeSerial, Synchronous (Shared clock)
Wires2 (SDA, SCL)
SpeedStandard (100kbps), Fast (400kbps), High (3.4Mbps)
AddressingYes (Hardware hex addresses)
Multi-deviceYes (Many Masters, Many Slaves)

Device Addresses: Because they share wires, every device has a unique hardware address. A temperature sensor might be 0x48, an IMU might be 0x68, and an OLED display might be 0x27. The Master controls the communication and specifies the address it wants to talk to.

On Raspberry Pi: You can enable I2C using sudo raspi-config. You can scan the bus using i2cdetect -y 1. It will ping every address and show you which ones reply (e.g., it will output 27 48 68).

from smbus2 import SMBus
# Python I2C Example
bus = SMBus(1)
address = 0x68 # IMU sensor

# Read data from register 0x00
data = bus.read_byte_data(address, 0x00)
print(data)
Lesson III

SPI — High-Speed Communication

Serial Peripheral Interface

SPI is a synchronous serial protocol designed for speed. When I2C is too slow for things like color displays, SD cards, fast flash memory, or high-speed ADCs, you use SPI.

Mental Model: SPI is like a dedicated, private, high-speed multi-lane highway between devices.

🔌How it works & Wiring
Uses 4 or more wires:
• MOSI (Master Out Slave In) → Master sends to Slave
• MISO (Master In Slave Out) → Slave sends to Master
• SCLK (Serial Clock) → Timing
• CS / SS (Chip Select) → Master pulls this wire LOW to wake up a specific device.

To add a 2nd device, you share MOSI/MISO/SCLK, but you need a brand new, dedicated CS wire just for that device (e.g., CS1 for Display, CS2 for Flash).
FeatureDetails
TypeSerial, Synchronous
Wires4+ (Depends on number of devices)
SpeedHigh (Tens of Mbps)
Full DuplexYes (Can send and receive at the exact same time)
AddressingHardware based (CS pin), no software overhead

Why is it faster than I2C? Three reasons: 1) It's Full Duplex (two dedicated lanes for reading/writing simultaneously, like a phone call). 2) There is no addressing overhead in the software payload; addressing is handled physically via the CS wire. 3) It has a dedicated connection.

On Raspberry Pi: Devices appear as /dev/spidev0.0.

import spidev
# Python SPI Example
spi = spidev.SpiDev()
spi.open(0, 0) # Bus 0, Device 0 (CS0)

# Send 0x01 and 0x02, simultaneously read response
response = spi.xfer([0x01, 0x02])
print(response)
Lesson IV

USB — Computer Communication

Universal Serial Bus

USB is NOT a simple hardware protocol like UART or I2C. It is an extremely complex, full operating system level ecosystem.

Mental Model: A strict hierarchy where a Host PC manages everything. Devices (cameras, keyboards, mice, SSDs, phones) cannot talk to each other directly; they must wait for the Host to manage them.

USB has enumeration, drivers, endpoints, power management, and device classes. It's incredibly advanced.

VersionMax Speed
USB 2.0480 Mbps
USB 3.05 Gbps
USB 3.110 Gbps
USB 440+ Gbps

Why USB is powerful: A single cable provides data, power delivery, and automatic device detection (plug-and-play).

USB Classes: When a device connects, it announces its "Class" (e.g., HID for Keyboards, UVC for Webcams, Mass Storage for SSDs, Audio Class for Audio). The OS automatically enumerates it, loads the correct driver, and creates an interface. For example, a Linux machine will create /dev/video0 for a webcam, which apps can then use.

Lesson V

CAN Bus — The Automotive Network

Controller Area Network

Used heavily in cars, industrial automation systems, robotics, and ECUs. It is famous for being incredibly reliable in terrible physical environments (electrical noise, heat, vibrations).

Mental Model: Imagine a car-wide messaging network. Every ECU (Engine unit, ABS, Airbags, Dashboard) is connected and listens to the exact same wires.

The Key Idea: Messages don't have destination addresses; they have Priorities. Important messages win. If the Engine ECU says "Speed=82 km/h" and the Airbag ECU simultaneously says "DEPLOY!", the protocol physically guarantees that the higher priority message (Airbag) wins the bus arbitration without losing any bits, while the other ECUs back off and retry later.

Lesson VI

Ethernet — Networking

The backbone of IP communication

Ethernet is your local internet. It forms the base physical layer that allows IP, TCP, and UDP to function. It is used for networking, servers, PCs, industrial automation, and IP cameras. Common speeds are 100 Mbps, 1 Gbps, and 10 Gbps over long distances.

Ethernet Stack: Physical cable → Ethernet → IP → TCP/UDP → Application.

Why it's important for Edge AI: Modern AI camera systems rely heavily on Ethernet. Streaming video from an IP camera requires RTSP or ONVIF protocols. Real-time video communication requires WebRTC. All of these sit on top of the Ethernet stack via TCP/UDP.

💡Important Interview Question
Why use SPI over I2C?
SPI is much faster, supports full-duplex (read and write simultaneously), and has less protocol overhead, making it better for real-time/high-throughput devices like displays. However, I2C uses fewer wires (only 2 vs 4+) and handles multiple devices seamlessly via addressing, making it simpler for PCB routing and easier for multi-device support.

Real-World AI Camera System Architecture

Once you understand these protocols, edge systems become much easier to decode. In a single smart camera device:

  • I2C talks to the local Temperature Sensor to prevent overheating.
  • SPI connects to the internal Flash memory chip holding the bootloader.
  • MIPI CSI handles the massive raw data directly from the camera image sensor.
  • PCIe or USB writes the processed video to the internal or external SSD storage.
  • Ethernet connects the camera to the network.
  • RTSP/WebRTC (over Ethernet) streams the feed to the cloud.
  • UART is exposed on a header pin for engineers to access the debug console.

Your Practical Learning Path:

If you want to master embedded systems, hardware streaming, GStreamer, and edge AI, learn in this order:

  1. Beginner: UART (deeply), I2C (deeply), SPI (deeply). Buy a Raspberry Pi 5, a USB UART adapter, an I2C OLED, an SPI display, an MPU6050 sensor, and a logic analyzer.
  2. Intermediate: Linux device drivers, GPIO/interrupts, CAN Bus, and USB basics.
  3. Advanced: Ethernet, PCIe, MIPI CSI, ONVIF, RTSP/WebRTC, Camera interfaces, GStreamer pipelines, CUDA/TensorRT, and Edge AI systems.

The Final Truth: Protocols are simply rules for moving bits between devices. That’s all. The differences are just trade-offs in speed, wiring, complexity, reliability, and scale.
