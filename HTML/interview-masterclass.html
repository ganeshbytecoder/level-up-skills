Explain Consistent Hashing and how it resolves the rehashing problem when scaling a cluster.
HARD
↓

The Problem: Standard hashing (hash(key) % N) fails when N (number of servers) changes. If you add 1 server to a 4-server cluster, 80% of all keys will suddenly route to a different server, causing a massive cache miss spike (a thundering herd) that can crash the database.

Consistent Hashing: Maps both data keys and servers to a conceptual "Hash Ring" (e.g., a circle of 0 to 2^32-1). To find where a key goes, you hash the key, place it on the ring, and walk clockwise to the first server you encounter.

The Fix: When a new server is added, it splits the key-space of only one existing server (the one immediately clockwise to it). Only the keys mapped to that specific segment are re-routed. The rest of the cluster (N-1 servers) is completely unaffected.

Virtual Nodes (vnodes): To prevent unequal load distribution (where one server gets a huge slice of the ring), each physical server is mapped to multiple "virtual nodes" scattered randomly around the ring, ensuring perfectly uniform load distribution.

Compare API Gateway vs Service Mesh. When do you need one over the other?
MED
↓

API Gateway: Manages North-South traffic (Client to Data Center). It acts as the single entry point for external clients. Its primary duties are Authentication, Rate Limiting, IP Allowlisting, Request Routing, and Protocol Translation (REST to gRPC).

Service Mesh: Manages East-West traffic (Service to Service inside the Data Center). It is deployed via a "sidecar proxy" (like Envoy) alongside every microservice container. Its primary duties are mTLS encryption between services, advanced load balancing, retries, circuit breaking, and distributed tracing.

When to use: You almost always need an API Gateway if you have external clients. You only need a Service Mesh when you have hundreds of microservices and the operational overhead of managing mutual TLS, retries, and network policies in the application code becomes impossible.

Design a pagination strategy for a table with 100 Million rows. Why does OFFSET/LIMIT fail?
MED
↓

The Problem with OFFSET: A query like SELECT * FROM users ORDER BY created_at LIMIT 50 OFFSET 1000000 forces the database to read, sort, and discard 1,000,000 rows just to return the 50 rows you want. This becomes exponentially slower the deeper you paginate.

The Fix - Cursor Pagination (Keyset Pagination): Instead of counting rows, you use the last seen value as an anchor.

SELECT * FROM users WHERE created_at < '2024-01-01T12:00:00' ORDER BY created_at DESC LIMIT 50

Why it works: The database hits the B-Tree index at exactly that timestamp and sequentially reads the next 50 rows. Time complexity drops from O(N) to O(1) index lookup + 50 reads. It is blazing fast regardless of depth.

Compare Cache-Aside, Write-Through, and Write-Behind caching strategies.
MED
↓

Cache-Aside (Lazy Loading): App checks cache. If miss, app reads from DB, updates cache, returns data. Great for read-heavy workloads. Cache only holds requested data. Drawback: Cache misses penalize latency.

Write-Through: App writes data to cache, and the cache synchronously writes to the DB. Guarantee: Cache and DB are always perfectly in sync. Drawback: High write latency (waits for two network hops).

Write-Behind (Write-Back): App writes to cache, returns success instantly. Cache asynchronously flushes data to DB later. Unbeatable write throughput. Drawback: If the cache crashes before flushing, data is permanently lost.

Design an idempotent Payment API. How do you guarantee a user is not charged twice if their client retries a timeout?
HARD
↓

The Core Pattern: We must use an Idempotency Key (typically a UUID) generated by the client, not the server.

The Flow:

  • Client generates Idempotency-Key: abc-123 and sends the POST request.
  • Server attempts to insert the Idempotency Key into an idempotency_keys table with a UNIQUE constraint.
  • If insert succeeds: This is a new request. Process the payment, update the idempotency record with the final response/status, and return 200 OK.
  • If insert fails (key exists): This is a retry. The server looks up the existing record. If the payment is still processing, return a 409 Conflict. If it completed, return the cached HTTP response of the original transaction.

Explain XID wraparound in PostgreSQL and why it is a critical failure mode. How do you mitigate it?
HARD
↓

The Mechanism: PostgreSQL uses Multi-Version Concurrency Control (MVCC). Every row version is tagged with a 32-bit transaction ID (XID). Because 32 bits max out at ~4 billion, the XID space acts as a circular buffer. Postgres considers any row with an XID in the "past" 2 billion as visible, and the "future" 2 billion as invisible.

The Failure: If a database processes 2 billion transactions without cleaning up old XIDs, the XIDs "wrap around." Suddenly, old, valid rows appear to be in the "future" and become completely invisible. To prevent data loss, Postgres shuts down and refuses connections if it approaches wraparound.

The Mitigation: The autovacuum daemon periodically scans tables and marks old rows as "frozen" (a special XID always considered in the past). You must monitor the age(datfrozenxid) metric and ensure long-running transactions aren't blocking autovacuum.

Compare B-Tree vs LSM Tree internals. Why do modern NoSQL databases prefer LSM Trees?
HARD
↓

B-Tree (Postgres/MySQL): Designed for in-place updates on disk. Fast for reads. However, random writes cause "page splits" and fragmentation. Every write requires moving the disk head, making it severely bottlenecked by random I/O.

LSM Tree (Cassandra/RocksDB): Optimized for write-heavy workloads. Writes are buffered in memory (Memtable) and then flushed to disk as immutable, sequentially written files (SSTables). Sequential disk I/O is 100x faster than random I/O, allowing massive write throughput.

Trade-off: LSM trees have incredible write speed but suffer from "Read Amplification" (must check multiple SSTables for a read) and require continuous background "Compaction" to merge files and clean up tombstones.

Explain Clustered vs Secondary Indexes in MySQL (InnoDB). What is a covering index?
HARD
↓

Clustered Index: In InnoDB, the Primary Key IS the table. The actual row data is stored in the leaf nodes of the Primary Key B+ tree, physically ordered on disk.

Secondary Index: A secondary index does NOT store the row data. It stores the indexed column(s) and a "pointer" to the row. In InnoDB, that pointer is the Primary Key value.

The Issue (Bookmark Lookup): If you query SELECT * FROM users WHERE email='x' using a secondary index on email, MySQL searches the email index to find the Primary Key, then does a SECOND search in the Clustered Index to get the rest of the row.

Covering Index: An index that contains all the columns needed for a query (e.g., CREATE INDEX idx_email_name ON users(email, name) for SELECT name FROM users WHERE email='x'). The DB never needs to touch the clustered index, making the query vastly faster.

Explain the impact of Tombstones in Cassandra. Why does frequent DELETE + INSERT lead to ReadTimeouts?
HARD
↓

The Architecture: Cassandra uses an LSM Tree. Data is written to immutable SSTables. Therefore, a DELETE does not remove data; it writes a special marker called a Tombstone with a deletion timestamp.

The Impact: During a read, Cassandra reads the live data and the tombstones, filters them in memory, and returns the result. If you frequently update/delete rows, you generate millions of tombstones. Reading a partition forces Cassandra to scan all millions of tombstones into JVM memory, causing GC pauses and ReadTimeoutException.

The Fix: Tombstones are permanently removed during Compaction after gc_grace_seconds. Avoid data models requiring frequent deletes; use append-only time-series designs instead.

How do you choose a MongoDB shard key? What happens if you choose a monotonic key like a timestamp?
MED
↓

The Rule: A shard key must have high cardinality, low frequency, and ideally not be strictly monotonic.

The Monotonic Disaster: If you shard by created_at timestamp or an Auto-Incrementing ID, all new inserts will always map to the "latest" chunk. This means 100% of your write traffic hits exactly ONE shard, while the other shards sit idle. This is called a write hotspot.

The Fix: Use a hashed shard key (e.g., { "user_id": "hashed" }) to distribute writes perfectly evenly across all shards, or a compound key to group related data together while still maintaining distribution.

How do you handle severe consumer lag in Kafka during an unexpected massive traffic spike?
MED
↓

Immediate Mitigations (The "Bleeding" Phase):

  • Scale out Consumers: If you have unused partitions, spin up more consumer instances. (You cannot have more active consumers than partitions in a consumer group).
  • Batch Processing: Modify consumer code to process messages in batches, significantly reducing DB roundtrips.

Structural Fixes (The "Healing" Phase):

  • Dead Letter Queues (DLQ) / Retry Topics: If lag is caused by slow external API calls, immediately move failing messages to a DLQ/Retry topic so the main consumer can continue processing healthy messages.
  • Backpressure: If the downstream DB is the bottleneck, scaling consumers will just DDOS your own DB. Rate-limit the consumers and let Kafka act as the buffer it was designed to be.

Compare Choreography vs Orchestration in the Saga pattern for Distributed Transactions.
HARD
↓

The Problem: A user places an order. You must deduct inventory, charge card, and create shipping label across 3 microservices. 2PC (Two Phase Commit) blocks too long. A Saga breaks it into local transactions.

Choreography (Event-Driven): No central controller. Service A finishes and publishes an event. Service B listens, does its work, publishes an event. If Service C fails, it publishes a failure event, and A & B listen to it to execute "compensating transactions" (rollbacks).

  • Pros: No single point of failure, highly decoupled.
  • Cons: "Spaghetti architecture" — hard to track the state of a transaction across 15 topics.

Orchestration (Command-Driven): A central "Orchestrator" service explicitly commands A, then B, then C via direct calls or targeted commands. It maintains a state machine (e.g., AWS Step Functions or temporal.io). If C fails, the Orchestrator commands B and A to rollback.

  • Pros: Easy to monitor, debug, and trace transaction state.
  • Cons: Orchestrator becomes a god-service and point of failure.

What is the Outbox Pattern? Why is publishing an event right after a database transaction dangerous?
HARD
↓

The Danger (Dual Write Problem):

BEGIN TX;
insert_order_to_db();
publish_to_kafka();
COMMIT TX;

If Kafka is down, the DB rolls back. Fine. But if the DB commit fails after Kafka publishes, the system is permanently inconsistent (Kafka has an event for an order that doesn't exist).

The Outbox Pattern:

  • Create a table called outbox_events in the SAME database.
  • In a single ACID transaction, write the order to the orders table AND write the event payload to the outbox_events table. It either both succeeds or both fails.
  • A separate asynchronous process (e.g., Debezium CDC reading the binlog, or a cron job) continuously polls the outbox_events table and safely pushes them to Kafka with at-least-once guarantees.

Explain Exactly-Once vs At-Least-Once delivery in messaging. How is "Exactly-Once" practically achieved?
MED
↓

At-Least-Once: The message broker guarantees delivery, but retries (network timeouts) may cause the consumer to receive the same message multiple times.

Exactly-Once: Mathematically impossible across a network due to the Two Generals' Problem. However, it is functionally simulated through Idempotency.

The Implementation: You design the consumer to be Idempotent. The consumer receives the message, extracts a unique ID, and processes it. If it receives the exact same message again, it identifies the duplicate ID and safely discards it or returns the cached result. To the end user, it appears as "Exactly-Once" processing.

What is a "Thundering Herd" problem? How does caching exacerbate it, and how do you prevent it?
MED
↓

The Problem: A "Thundering Herd" occurs when a highly requested cache key suddenly expires (or is evicted). Instantly, thousands of concurrent requests miss the cache simultaneously. They all bypass the cache and hit the underlying database at the exact same millisecond to compute the same value, causing the database to spike to 100% CPU and crash.

Prevention Strategies:

  • Mutex/Locks: On cache miss, the first thread acquires a distributed lock (e.g., Redis SETNX). Other threads requesting the same key wait/poll until the first thread populates the cache.
  • Probabilistic Early Expiration (XFetch): Instead of letting a key expire, a worker randomly decides to recompute the cache before it expires. The closer the key is to expiration, the higher the probability.
  • Background Refresh: The cache never expires for reads. A separate background worker constantly updates hot keys directly from the DB.

Explain the three states of the Circuit Breaker pattern. Why is it essential for microservices?
MED
↓

If Service A calls Service B, and Service B is experiencing massive latency, A's threads will all block waiting for B. Eventually, A runs out of threads and crashes too. This is a cascading failure.

Circuit Breaker States:

  • Closed: Normal operation. Requests flow freely. If failures/timeouts cross a threshold (e.g., 50% fail over 10s), the circuit trips to Open.
  • Open: All requests fail instantly (fast-fail) returning an error or fallback response. This gives Service B time to recover instead of hammering it while it's down.
  • Half-Open: After a timeout, the breaker lets a few test requests through. If they succeed, it closes. If they fail, it trips back to Open.

What is a Retry Storm? How do Exponential Backoff and Jitter prevent it?
MED
↓

Retry Storm: Service A goes down for 5 seconds. Thousands of clients instantly retry at the exact same 1-second interval. When Service A recovers, it is immediately hit by a synchronized wave of thousands of retries, crashing it again.

Exponential Backoff: Increases the delay between retries exponentially (e.g., 1s, 2s, 4s, 8s). This reduces the total load on the recovering system.

Jitter (Randomness): Even with backoff, if 1000 clients failed at time T=0, they will all retry at exactly T=1, T=3, T=7. Jitter adds random noise (e.g., delay = base_delay * random(0.5, 1.5)) so the retries smear across a time window, breaking the synchronization and smoothing the load.

Explain the Bulkhead pattern and how it isolates failures in a monolith.
MED
↓

Inspired by ship design where flooding in one compartment is contained by bulkheads, preventing the whole ship from sinking.

In software, if a Monolith shares a single connection pool or thread pool for all API endpoints, a slow response on the /generate_pdf endpoint will consume all available threads, causing the critical /login endpoint to fail due to thread starvation.

The Fix: Allocate strictly separated resource pools (bulkheads). /generate_pdf gets max 10 threads. /login gets 50 threads. If PDF generation hangs, it maxes out its 10 threads and starts rejecting requests, but the remaining 50 threads keep the rest of the application perfectly healthy.

Compare Blue/Green Deployments vs Canary Releases. When should you use which?
MED
↓

Blue/Green: You maintain two identical production environments. Blue is currently live. You deploy the new version to Green, test it privately. When ready, you flip the load balancer router to point 100% of traffic to Green. Rollback is instant (flip router back to Blue). Requires 2x the hardware/compute.

Canary Release: You deploy the new version to a small subset of servers (e.g., 5%). The load balancer routes 5% of real user traffic to the "canary". You monitor error rates and metrics. If healthy, you gradually ramp traffic to 100%. If errors spike, you roll back the 5%.

When to use: Blue/Green is great for monolithic apps or schema-less DBs where instant cutover is needed. Canary is preferred for microservices at scale (FAANG), minimizing blast radius of a bad release without needing 2x infrastructure.

Explain Layer 4 (Network) vs Layer 7 (Application) Load Balancing.
MED
↓

Layer 4 (Transport Level): Operates at the TCP/UDP level. It looks only at IP addresses and Ports. It forwards packets blindly without inspecting the content. It is incredibly fast, consumes almost zero CPU, and can handle millions of connections. (e.g., AWS Network Load Balancer).

Layer 7 (Application Level): Operates at the HTTP/HTTPS level. It decrypts the TLS connection, reads the HTTP headers, URL path, and cookies. It can route traffic based on the URL (e.g., /api/v1/ goes to service A, /images/ goes to service B). It is much more CPU intensive but allows smart routing, rate limiting, and WAF rules. (e.g., AWS Application Load Balancer, Nginx).

What is a CDN (Content Delivery Network) and how does it handle dynamic vs static content?
MED
↓

A CDN (like Cloudflare or Fastly) is a globally distributed network of proxy servers ("Edge Locations") positioned close to end-users to reduce latency.

Static Content: Images, CSS, JS are cached at the Edge. User requests hit the Edge, and the Edge serves the file directly without touching your origin server. Drops latency from 150ms to 15ms and offloads 90% of traffic.

Dynamic Content: CDNs cannot cache highly personalized dynamic API data (like a bank balance). However, they accelerate dynamic traffic via Connection Pooling and BGP Anycast Routing, maintaining persistent, optimized TCP/TLS connections from the Edge back to the Origin, bypassing public internet congestion.

Explain the difference between OLTP and OLAP databases. Why can't Postgres do both well at scale?
MED
↓

OLTP (Online Transaction Processing): Designed for highly concurrent, fast, small read/write operations (e.g., e-commerce checkouts). Row-oriented storage. Example: PostgreSQL, MySQL.

OLAP (Online Analytical Processing): Designed for complex queries analyzing massive datasets (e.g., "Sum of all sales grouped by region for the last 5 years"). Column-oriented storage. Example: Snowflake, ClickHouse, BigQuery.

Why Columnar matters: If you run SELECT SUM(revenue) FROM sales on a Row-oriented DB, it must read entire rows (customer_id, date, status, etc.) from disk into memory just to extract the revenue. A Column-oriented DB stores all revenue values contiguously on disk. It reads ONLY the revenue column in a single I/O sweep and compresses it massively (run-length encoding), making it 100x to 1000x faster for analytics.

Compare Apache Kafka vs RabbitMQ. When do you choose a Message Queue vs an Event Streaming platform?
HARD
↓

RabbitMQ (Smart Broker, Dumb Consumer): A traditional message queue. Messages are pushed to consumers. Once processed and ACKed, the message is deleted from the queue. Perfect for transient task routing, complex routing rules, and point-to-point worker queues (e.g., sending emails).

Kafka (Dumb Broker, Smart Consumer): An event streaming platform. It acts as an immutable, distributed append-only log. Messages are NOT deleted when read; they are retained on disk for days or years. Consumers track their own "offset" (where they are in the log).

Why Kafka? Because data is retained, multiple independent systems (Analytics, Billing, Search Indexer) can read the exact same events at their own pace. You can also "rewind" the log to reprocess data after deploying a bug fix, which is impossible in RabbitMQ once data is ACKed.

Estimate the timeline Fan-Out throughput for Twitter/X.
HARD
↓

  • 10,000 Tweets per second (TPS) globally.
  • Average user has 200 followers.
