Prologue

The Last Lesson

Ara was seventy-three years old when she finally sat down to write.

She had spent forty years building the city — not the buildings, not the guilds, but the shape of how they all fit together. The rules beneath the rules. The principles that determined not just what to build, but how to think about what to build.

She had watched a hundred engineers build systems that worked technically — the code ran, the tests passed — but fell apart within a year because the shape was wrong. The wrong things were too tangled together. The wrong boundaries had been drawn. The domain was misunderstood. The system was designed for the machine, not for the humans who would maintain it for decades.

She called Priya. "One more lesson. Not about code. About shape."

They sat together in Ara's study for six days.

Lesson I

The Twelve Laws of the Living Service

The 12-Factor App — the minimum any deployable service must achieve

Ara began with the question Priya had asked so many times: "Why does this system that works in staging fail in production?"

"A group of engineers who had built and operated more services than any before them sat down and asked that exact question. They discovered the answer was almost always one of twelve things. Twelve factors — twelve properties — that any service must have to be deployable, operable, and maintainable."

Factor I

Codebase — One Codebase, One Repository

The Delivery Guild had three different codebases for three routes. When a bug appeared in the shared routing logic, it had to be fixed in three places. One was always missed.
One codebase tracked in version control; many deploys from it. Multiple codebases = distributed system, not one app. Each service has its own repo. Shared code extracted to libraries.
Factor II

Dependencies — Explicitly Declare, Never Assume

The Payment Guild's code assumed a particular script was installed on every machine. It was — until a new server was provisioned. Within an hour: down.
Explicitly declare all dependencies via a manifest (pom.xml, package.json). Never rely on system-wide packages. Run in a clean environment to verify all dependencies are declared.
Factor III

Config — In the Environment, Not in the Code

Database credentials were hardcoded in the Records Guild's code. The security auditor found the production password on line 47 of the main service class.
Everything that varies between environments (dev/staging/prod) goes in environment variables — NOT in code, NOT in committed config files. Database URLs, API keys, feature flags: all environment-injected. Never hardcode. Never commit secrets.
Factor IV

Backing Services — Treat as Attached Resources

The Notification Guild's SMS provider address was hardcoded in twelve places. When the provider changed, it took two weeks to update everywhere.
Databases, queues, SMTP, and third-party APIs are attached resources. Interchangeable: swap a local DB for a cloud one via config only, not code changes. Reference by URL/config.
Factor V

Build, Release, Run — Three Strictly Separate Stages

An engineer pushed a "quick fix" directly to the running production server. Three hours later, nobody could reproduce the issue or roll back.
Build: code + dependencies → artifact (JAR, container image). Code is frozen. Release: artifact + config. Every release is immutable and rollback-able. Run: execute the release. No code changes possible at runtime.
Factor VI

Processes — Stateless and Share-Nothing

The Scheduling Guild stored user sessions in server memory. When it scaled to five servers, users reported being randomly "logged out" — their session was on server 1, the next request hit server 5.
Services are stateless. No share-nothing: all persistent state lives in backing services (DB, Redis). Any instance handles any request. This enables horizontal scaling and failover without data loss.
Factor VII

Port Binding — Export Services via Port

The Medical Guild's service could only run embedded in the city's application server. Testing required running the full server — four minutes to start, every test run.
The service is self-contained and binds to a port itself. Not embedded in a container (Tomcat, IIS) — it IS the server. Enables running locally with a simple command. Clean, portable, testable independently.
Factor VIII

Concurrency — Scale Out via the Process Model

The Order Guild handled concurrency by making one monolithic process with more threads. It ran out of memory at 800 concurrent orders. Scale-out would have run more processes, not bigger ones.
Scale by running more processes, not bigger ones. Different workloads can run different process types (web, worker, scheduler). Horizontal scaling is the natural answer to load.
Factor IX

Disposability — Fast Start, Graceful Stop

During a deployment, old Audit Guild processes were killed immediately. Two hundred in-progress audit records were lost — the process hadn't finished writing them.
Processes start in seconds and stop gracefully. On SIGTERM: finish current request, close connections, stop accepting new work. Fast startup enables autoscaling. Graceful shutdown prevents data loss.
Factor X

Dev/Prod Parity — Keep Environments as Similar as Possible

Development used SQLite. Production used PostgreSQL. A query that worked in development failed in production — discovered by a citizen, not a test.
Minimize differences between development, staging, and production. Same database type. Same OS. Same versions. Docker makes this tractable: if it runs in the container on your laptop, it runs the same in production. "Works on my machine" is an architectural failure.
Factor XI

Logs — Treat Them as Event Streams

The Verification Guild wrote logs to files on each server. When debugging, engineers SSH'd to each server and tailed files manually. When servers autoscaled down, the log files were destroyed with them.
A service never manages its own log files. Write log events to stdout/stderr as a time-ordered stream. The environment collects, aggregates, and indexes them (ELK, CloudWatch, Datadog). Service is decoupled from log storage entirely.
Factor XII

Admin Processes — One-Off Tasks as First-Class Citizens

The database migration script lived as a comment in a Slack message. Each migration required someone to remember to run it, SSH to a server, and execute it from memory.
Database migrations, backups, one-off scripts: first-class processes. Run in the same environment, using the same code, tracked in the same repository. Never "log in and run a command." Codify it. Version it. Run it as a process.
Lesson II

Knowing Your Domain First

Domain-Driven Design — before you design the system, understand the world it models

On the second day, Ara said something unexpected: "Before you design a system, you must understand the world it models. Most engineers skip this step. They start with the database schema. They model tables, not understanding. Then they spend the next three years fighting a system that encodes misunderstanding at its foundation."

💬 Ubiquitous Language

A shared, precise vocabulary developed by engineers and domain experts together. The exact words the domain uses — not technical jargon, not business euphemism. If the domain calls it "merchant profile," the code class is MerchantProfile, not UserAccount.

⭕ Bounded Context

A boundary within which a particular model applies precisely. "Account" means different things to Payment, Registration, and Medical contexts — and that's correct. Each is right within its boundary. Translation happens at the edges.

🏷️ Entity

An object defined by its identity (a unique ID) rather than its attributes. Even if all attributes change, the same ID = the same Entity. Ravi remains the same merchant regardless of name, address, or category changes.

💰 Value Object

Defined entirely by its attributes. Two Value Objects with the same attributes are fully interchangeable. Money(5, "gold") equals any other Money(5, "gold"). Immutable — when a value changes, replace the object entirely.

🌳 Aggregate & Root

A cluster of domain objects treated as a single unit. The Aggregate Root is the only entry point from outside. All modifications go through it — it enforces all consistency invariants of the cluster.

📢 Domain Events

Something meaningful that happened — named in past tense from the domain's perspective: TradeAgreementSigned, MerchantRegistered. Not "update DB row." How Bounded Contexts communicate without coupling.

💾 Repository Pattern

An abstraction hiding all data persistence from the domain. Domain speaks domain language; Repository handles loading/saving to the database. No SQL in domain code. No database concepts in business logic.

🧱 Anti-Corruption Layer

A translation layer between two Bounded Contexts with different models. Prevents one context's concepts from bleeding into and corrupting the other. Essential when integrating with legacy systems or external APIs.

💡Bounded Context in Practice
// Payment Context
Account { id, balance, currency, transactions[] }

// Registration Context
Account { id, username, passwordHash, lastLogin }

// They share only: merchantId (the bridge between contexts)
// Two completely different classes named "Account" — and both are correct.
Lesson III

The Shape of the Service Itself

Hexagonal Architecture (Ports & Adapters)

Ara drew a hexagon on the board. "The domain is at the center. Pure. It knows nothing about databases, HTTP, Kafka, or command lines. It speaks only the domain's language."

She drew arrows pointing inward. "Outside the world wants to interact with the domain. HTTP requests, Kafka events, scheduled jobs. These enter through Ports — defined interfaces."

She drew arrows pointing outward. "The domain needs to interact with the outside world — save data, send notifications, emit events. These are outbound Ports. The implementations — the actual HTTP handler, the actual database class — these are Adapters. They plug in and can be swapped without touching the domain."

// DOMAIN (center — knows nothing about the outside world) class TradeService { merchantRepo: MerchantRepository // Outbound Port (interface) eventPublisher: DomainEventPublisher // Outbound Port (interface) executeTrade(request: ExecuteTradeRequest): TradeResult { const merchant = merchantRepo.findById(request.merchantId) const trade = merchant.createTrade(request.terms) const events = trade.execute() merchantRepo.save(merchant) events.forEach(e => eventPublisher.publish(e)) return TradeResult.success(trade.id) } } // INBOUND ADAPTERS — connect outside world TO domain class TradeRestController { // HTTP adapter @POST("/trades") handleRequest(httpReq) { return tradeService.executeTrade(mapToRequest(httpReq)) } } class TradeKafkaConsumer { // Kafka adapter @KafkaListener("trade-commands") handleMessage(event: TradeCommandEvent) { tradeService.executeTrade(mapToRequest(event)) } } // OUTBOUND ADAPTERS — connect domain TO outside world class PostgresMerchantRepo implements MerchantRepository { // SQL here — domain never touches it } class KafkaEventPublisher implements DomainEventPublisher { // Kafka here — domain never touches it }
💡Key Benefit
The domain is testable in complete isolation — inject in-memory implementations of all Ports. No database required to test business logic. The domain can also serve HTTP and Kafka from the same core — just write new Adapters without touching the domain.
Lesson IV

The Layered Architecture

Clean Architecture — the law of dependency direction

┌──────────────────────────────────────────────────────────────┐ │ FRAMEWORKS & DRIVERS (outermost layer) │ │ Web framework · Database drivers · UI · External APIs │ │ ┌────────────────────────────────────────────────────────┐ │ │ │ INTERFACE ADAPTERS │ │ │ │ Controllers · Presenters · Gateways · Repositories │ │ │ │ ┌────────────────────────────────────────────────┐ │ │ │ │ │ APPLICATION USE CASES │ │ │ │ │ │ What the system can DO │ │ │ │ │ │ ┌──────────────────────────────────────────┐ │ │ │ │ │ │ │ DOMAIN (innermost — most stable) │ │ │ │ │ │ │ │ Entities · Value Objects · Rules │ │ │ │ │ │ │ │ No dependencies on anything outer │ │ │ │ │ │ │ └──────────────────────────────────────────┘ │ │ │ │ │ └────────────────────────────────────────────────┘ │ │ │ └────────────────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────────────┘ THE DEPENDENCY RULE: Source code dependencies can ONLY point inward. Nothing in an inner circle can know anything about an outer circle.
💡Why It Matters
Swap databases (PostgreSQL → MongoDB) by rewriting only the outermost ring's adapter. Domain and Use Cases are untouched. In practice:

• Domain Entity: no framework imports, no @Entity, no @Table. Pure records.
• Use Case: imports Domain only. No framework imports.
• Controller: imports Use Cases only. No domain imports directly.
• DB Adapter: imports Use Case interfaces. Knows about JPA/SQL.
Lesson V

Migrating Without Breaking Everything

The Strangler Fig Pattern — the only safe way to replace a living system

"The city's original palace — the monolith — did not disappear overnight. It took seven years to replace it with the guild system. During those seven years, both existed simultaneously. Merchants used both. Some transactions went to the palace, some to the guilds."

💡Strangler Fig Pattern
Incrementally replace a monolith by building new functionality alongside it — routing traffic to the new system piece by piece — until the old one can be removed.

Steps:
  1. Identify a seam — one capability you can extract without breaking others
  2. Build it separately as a new service
  3. Route at the boundary via a Facade/proxy
  4. Verify stability, then advance to the next capability
  5. When all capabilities migrated, delete the monolith

Key rule: Never "big bang rewrite." Every failed software project that tried to rewrite a working system from scratch has a lesson for you. 
class TradingFacade {
  monolith: OldTradingSystem
  newPaymentService: PaymentService    // Already migrated
  newMerchantService: MerchantService  // Already migrated

  handleRequest(req: Request): Response {
    if (req.path.startsWith("/payments"))
      return newPaymentService.handle(req)    // ✅ Migrated
    if (req.path.startsWith("/merchants"))
      return newMerchantService.handle(req)   // ✅ Migrated
    return monolith.handle(req)              // ⏳ Not yet migrated
  }
}
Lesson VI

One Frontend, Many Backends

Backend for Frontend (BFF) — shaped data for each client's exact needs

The merchant dashboard on the web needed detailed trade analytics — forty fields, complex graphs, historical data. The merchant's mobile device needed only current balance and the last five trades. The Delivery Guild's terminal needed only routing information. Three clients. Completely different needs. One general-purpose Trading API — serving all three equally poorly.

💡Backend for Frontend (BFF)
A dedicated backend service for each type of frontend client. Each BFF aggregates calls to downstream services and shapes data exactly for its client's needs. The BFF is owned by the frontend team — they control their own data contract without waiting for a shared API.
// Web Dashboard BFF — 40-field rich response
class WebDashboardBFF {
  async getdashboard(merchantId) {
    const [trades, analytics, agreements] = await Promise.all([
      tradeService.getDetailedHistory(merchantId, 90_DAYS),
      analyticsService.getFullMetrics(merchantId),
      contractService.getAll(merchantId)
    ]);
    return { trades, analytics, agreements, ... }; // 40 fields
  }
}

// Mobile BFF — 8-field lean response
class MobileBFF {
  async getdashboard(merchantId) {
    const [balance, trades] = await Promise.all([
      accountService.getBalance(merchantId),
      tradeService.getRecent(merchantId, 5)  // Only last 5
    ]);
    return { balance, trades }; // 8 fields, optimized for mobile
  }
}
Lesson VII

The Sidecar and the Ambassador

Infrastructure Patterns — move cross-cutting concerns out of every service

Every service needed logging, metrics, mutual TLS, and rate limiting. The naive solution: add this code to every service. Forty services, forty copies of the same logging logic, the same metrics client, the same TLS setup. The elegant solution: move it out of the service entirely.

💡Sidecar Pattern
A helper container deployed alongside each service container. The sidecar handles cross-cutting concerns (logging, metrics, proxying, TLS) so the service doesn't have to.

Service calls sidecar via localhost — it speaks plain HTTP. The sidecar handles all infrastructure complexity: mTLS, retry, metrics, rate limiting.

Real-world: Istio/Envoy, Linkerd, Dapr.
💡Ambassador Pattern
A sidecar that proxies outbound calls. The service calls "localhost/payments" — the Ambassador knows the actual address, handles retries, circuit breaking, and authentication. The service code never needs to be updated when infrastructure changes.
💡Service Mesh
A collection of sidecars across all services that, together, form a programmable infrastructure layer — enabling observability, traffic management, and security without touching a single line of service code.
Lesson VIII

Data Architecture in Microservices

Database Per Service and Polyglot Persistence

💡Database Per Service
Each microservice owns its own database. No other service can read from or write to it directly — only through the service's API or emitted events. Enables independent scaling, deployment, and database type selection per service. Trade-off: cross-service queries require API calls or event-driven data aggregation.
💡Polyglot Persistence
Choose the right database for each service's access pattern:
Payment Service
PostgreSQL · ACID critical
Product Catalog
MongoDB · Flexible schema
Session Store
Redis · Fast key-value, TTL
Search Service
Elasticsearch · Full-text
Analytics
Cassandra · High write throughput
Event Store
Kafka · Immutable event log
Lesson IX

The Caching Kingdom

Cache-Aside, Write-Through, Write-Behind, Read-Through + Eviction Strategies

💡Cache-Aside (Lazy Loading) — Most Common
Check cache → if miss, load from DB → populate cache → return. 
getMerchant(id):
  merchant = cache.get(id)
  if merchant is null:
    merchant = db.findById(id)
    cache.set(id, merchant, TTL=5min)
  return merchant
Pros: Cache only what's actually requested.
Cons: Cache miss causes 2 round trips. Cold start = all misses.
💡Write-Through — Always Fresh
Write to cache AND database simultaneously on every write. Cache is always up to date. Cost: write latency increases (two writes per write operation).
💡Write-Behind (Write-Back) — Ultra-Fast Writes
Write to cache immediately; flush to DB asynchronously later. Ultra-fast writes. Risk: if cache dies before async write completes — data loss. Use only when write durability can be relaxed.
💡Cache Eviction Strategies
LRU (Least Recently Used): Evict the item not accessed for the longest time — good for temporal access patterns.
LFU (Least Frequently Used): Evict the item accessed least often — good for stable hot data.
TTL (Time To Live): Evict items after fixed time — good for data that must refresh periodically.
FIFO: Evict oldest items first — simple but often wrong.

Choose based on access pattern: trading rate data → TTL. Merchant profiles → LRU.
Lesson X

The API Design Laws

REST, GraphQL, gRPC, WebSocket, Webhook — each correct in its place, wrong everywhere else

💡REST
Resources in URLs, standard HTTP methods, stateless, wide client support. Use when: Public APIs, browser clients, simple CRUD, interoperability critical.
Design rules: nouns in URLs not verbs; HTTP methods convey intent; status codes are meaningful; idempotent by design; versioning via /v1/.
💡gRPC
Binary protocol over HTTP/2. Protobuf schemas. Auto-generated clients. Use when: Service-to-service where performance matters. Strongly typed contracts. Streaming needed. 5–10x smaller payload than REST.
💡GraphQL
Client specifies exactly what data it needs. Use when: Aggregating data from multiple services, mobile clients with varying data needs, over-fetching or under-fetching is a real problem. Watch out for N+1 query problems — use DataLoader.
💡WebSocket
Persistent full-duplex connection. Use when: Truly real-time: live price feeds, collaborative editing, live chat. Complex to manage at scale (connection state per client). Use SSE if communication is server→client only.
💡Webhook
Client registers a URL; server calls it when an event occurs. Use when: Notifying external systems of events without polling. Payment provider callbacks, CI/CD notifications. vs polling: polling asks repeatedly; webhook is notified instantly.
Epilogue

The Shape Beneath Everything

On the sixth day, Ara and Priya sat in silence for a time. Then Ara said:

"The biggest mistake I made in my first decade was believing that architecture was about technology. Kafka or RabbitMQ. PostgreSQL or MongoDB. REST or gRPC."

"It's not. Architecture is about boundaries. Where does one thing end and another begin? Who is responsible for what? What can change independently of what else?"

Before designing any system, answer these questions:

What is this system's domain? What are its core concepts?
Where are the boundaries? What should be separate?
What depends on what — and in which direction?
What is the right database for this access pattern?
How will this be deployed? How will it be tested?
What changes frequently? What must be stable?
What is the simplest architecture that works for the current scale?

Priya walked home through the city. She passed the Payment Guild — Service Domain, Hexagonal, PostgreSQL, Database-Per-Service. She passed the River — Event-driven backbone, Consumer Groups, Offsets. She passed the Watchtower — Observability, Centralized Logging. She passed the API Gateway — Facade, Ambassador, Rate Limiting.

Everything she had learned in three books was here, in the city around her. It had always been here. She had just lacked the language to see it.

She sat down at her desk and opened a new parchment. She began to design.
