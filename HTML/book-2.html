Part I — Creational Patterns

How objects come into being. Five patterns for controlling object creation.

Pattern 1 · Creational

Singleton

The Trading Platform needed an Exchange Rate Registry — one source of truth for every currency conversion rate. Every service needed access to it. But if each service created its own Registry, seventeen copies would exist, each potentially out of sync.

Corvin's Entry — The Royal Mint: "In the kingdom of Valdrus, any merchant could establish a Mint and print coins. The result: twelve different 'gold coins' of different actual weights. The King decreed: one Mint. One ledger. One authority. Not because gold was scarce — but because trust required a single source of truth."
💡Singleton Pattern
Ensures a class has only one instance and provides a global access point to it.

When to use: Configuration objects, connection pools, logging services, caches — when one instance is both necessary and sufficient.
Warning: Don't use as a way to avoid passing dependencies. Use only when true single-instance is an architectural requirement. 
class ExchangeRateRegistry {
  private static instance = null;

  private constructor() { this.loadRatesFromDB(); }

  static getInstance(): ExchangeRateRegistry {
    if (!this.instance) this.instance = new ExchangeRateRegistry();
    return this.instance; // Always the same object
  }
}

// Every caller
ExchangeRateRegistry.getInstance().getRate("GOLD", "SILVER")
Pattern 2 · Creational

Factory Method

The platform needed to create payment processors for 42 merchant types. Priya's first instinct: a massive if block. She wrote three lines before recognizing the smell — code that requires changes in seventeen places every time a new merchant type appears.

💡Factory Method Pattern
Defines an interface for creating an object, but lets subclasses decide which class to instantiate. The creator works with the result through the interface, never the concrete class. Key benefit: Adding a new type requires one new class + one factory entry — touching zero existing code.
interface PaymentProcessor {
  process(amount: Money, currency: Currency): Receipt;
}

class GrainPaymentProcessor implements PaymentProcessor { ... }
class SpicePaymentProcessor implements PaymentProcessor { ... }
class ForecastFuturesProcessor implements PaymentProcessor { ... }

class PaymentProcessorFactory {
  static create(merchantType: string): PaymentProcessor {
    switch (merchantType) {
      case "grain":     return new GrainPaymentProcessor();
      case "spice":     return new SpicePaymentProcessor();
      case "futures":   return new ForecastFuturesProcessor();
      // Add new types HERE ONLY — nothing else changes
    }
  }
}
Pattern 3 · Creational

Abstract Factory

Seventeen kingdoms each had their own family of tools — contract generator, payment verifier, dispute resolver — all compatible with each other. Mixing families (Northern contract + Eastern verifier) was catastrophic. Each family needed to come from the same source.

💡Abstract Factory Pattern
Provides an interface for creating families of related, compatible objects without specifying their concrete classes. The factory guarantees that all objects it creates work together. vs Factory Method: Factory Method creates one product. Abstract Factory creates a coordinated suite.
interface TradingToolsFactory {
  createContractGenerator(): ContractGenerator;
  createPaymentVerifier():   PaymentVerifier;
  createDisputeResolver():   DisputeResolver;
}

class NorthernTradingFactory implements TradingToolsFactory {
  createContractGenerator() { return new ParchmentContractGenerator(); }
  createPaymentVerifier()   { return new WaxSealVerifier(); }
  createDisputeResolver()   { return new NorthernGuildArbitrator(); }
  // All three are guaranteed to be compatible
}

// Client: only knows the factory interface
const tools = TradingToolsFactoryProvider.get("Northern");
const contract = tools.createContractGenerator().generate(trade);
Pattern 4 · Creational

Builder

A Trade Agreement had 23 configuration parameters — some optional, some mutually exclusive, some requiring others first. A 23-argument constructor was a disaster Priya recognized immediately.

💡Builder Pattern
Separates complex object construction from its representation. Step-by-step creation with optional parameters. build() validates required fields and produces an immutable object. Real-world: StringBuilder, HttpRequest.newBuilder(), Lombok's @Builder, SQL query builders.
const agreement = new TradeAgreementBuilder()
  .withParties(merchant1, merchant2)      // Required
  .withPaymentSchedule(PaymentSchedule.MONTHLY)
  .withPenaltyClause(PenaltyRate.TEN_PCT)
  .withJurisdiction(Kingdom.NORTHERN)
  .withExpiryDate(Duration.ONE_YEAR)
  // .withOptionalClause(...)             // Optional — omit if not needed
  .build(); // Validates, returns immutable TradeAgreement
Pattern 5 · Creational

Prototype

The platform had 40 Standard Agreement Templates used hundreds of times per day. Each time: fetch from DB, deserialize, configure. The DB queries alone were adding 200ms to every trade initiation. The solution: clone the template instead of rebuilding from scratch.

💡Prototype Pattern
Creates new objects by cloning an existing prototype. Avoids expensive re-initialization. Warning: Deep copy vs shallow copy — be explicit. A shallow clone shares references to nested objects.
// At startup: load all 40 templates ONCE
const templates = new Map();
(await db.loadTemplates()).forEach(t => templates.set(t.id, t));

// At runtime: clone in microseconds — no DB call
function getNewAgreement(templateId: string): TradeAgreement {
  const template = templates.get(templateId);
  return deepClone(template); // ~0.1ms vs 200ms
}

const agreement = getNewAgreement("GRAIN_SILVER_N_E");
agreement.setParties(merchant1, merchant2); // Customize the clone

Part II — Structural Patterns

How objects fit together. Seven patterns for composing objects into larger structures.

Pattern 6 · Structural

Adapter

The Kingdom of Aldenmere had a payment protocol from 600 years ago — different format, different field names, different data types. They had no interest in updating it. The Trading Platform needed to communicate with it without touching Aldenmere's code or its own standards.

💡Adapter Pattern
Converts the interface of a class into another interface that clients expect. Makes incompatible classes work together without modifying either. Real-world: Java's InputStreamReader, Spring HandlerAdapter, third-party SDK wrappers.
// Trading Platform expects:
interface ModernPaymentGateway {
  processPayment(transactionId: string, amount: Money): Receipt;
}

// Aldenmere's ancient system has:
class AncientAldenmereSystem {
  executeTran(tranNo: number, goldUnits: number, silverTenths: number): AncientReceipt;
}

// The Adapter — bridges the two:
class AldenmereAdapter implements ModernPaymentGateway {
  private ancient = new AncientAldenmereSystem();

  processPayment(txId: string, amount: Money): Receipt {
    const tranNo = parseInt(txId);
    const gold   = amount.wholeGoldUnits();
    const silver = amount.tenthsOfSilver();
    const r = this.ancient.executeTran(tranNo, gold, silver);
    return this.convertToModernReceipt(r); // Translate the response too
  }
}
Pattern 7 · Structural

Bridge

Trade data needed to display in 3 formats (report scroll, River message, visual map) from 2 data sources (real-time, historical). Without Bridge: 6 classes. Add a new format + source: 6 more. Next year: combinatorial explosion.

💡Bridge Pattern
Decouples abstraction from implementation so the two can vary independently. Adding a new renderer: 1 class. Adding a new data source: 1 class. Never 2×3=6 combination classes.
// Implementation hierarchy (renderers)
interface TradeRenderer {
  render(data: TradeData): Output;
}
class ScrollRenderer   implements TradeRenderer { ... }
class RiverRenderer    implements TradeRenderer { ... }
class VisualMapRenderer implements TradeRenderer { ... }

// Abstraction hierarchy (data sources) — holds a renderer reference
abstract class TradeDataSource {
  constructor(protected renderer: TradeRenderer) {}
  abstract getData(): TradeData;
  display(): Output { return this.renderer.render(this.getData()); }
}
class RealtimeSource  extends TradeDataSource { ... }
class HistoricalSource extends TradeDataSource { ... }

// Any combination — no new classes ever
new RealtimeSource(new ScrollRenderer()).display();
new HistoricalSource(new VisualMapRenderer()).display();
Pattern 8 · Structural

Composite

Merchants formed guilds; guilds formed federations; federations formed kingdom alliances. Applying a trade regulation meant applying it to one merchant, a guild, or a federation. The calling code should be identical for all three.

💡Composite Pattern
Composes objects into tree structures. Clients treat individual objects and compositions uniformly. Real-world: File systems (files + folders), UI components, XML/JSON parsing.
interface TradingEntity {
  applyRegulation(reg: Regulation): void;
  getTotalVolume(): Money;
}

class Merchant implements TradingEntity {
  applyRegulation(reg) { this.regulations.push(reg); }
  getTotalVolume() { return this.trades.sum(); }
}

class Guild implements TradingEntity {
  members: TradingEntity[] = []; // Can hold Merchants OR other Guilds

  applyRegulation(reg) {
    this.members.forEach(m => m.applyRegulation(reg)); // Recursive
  }
  getTotalVolume() {
    return this.members.map(m => m.getTotalVolume()).reduce(Money.add);
  }
}

// One call applies to thousands of merchants at any depth:
northernKingdomFederation.applyRegulation(newTariff);
Pattern 9 · Structural

Decorator

Agreements needed Encryption, Audit Logging, Compression, Digital Signing — in any combination. Subclassing: EncryptedAuditedCompressedSignedAgreement — infinite combinations. The Decorator wraps the object instead.

💡Decorator Pattern
Attaches additional responsibilities to an object dynamically. Same interface as the original. Can combine any capabilities without class explosion. Real-world: Java I/O streams, HTTP middleware chains, Spring AOP, Express.js middleware.
interface Agreement { execute(): Result; }

class BaseAgreement implements Agreement {
  execute() { return this.processTerms(); }
}

class EncryptionDecorator implements Agreement {
  constructor(private wrapped: Agreement) {}
  execute() {
    this.encrypt();
    const r = this.wrapped.execute(); // Delegate
    this.decrypt(r);
    return r;
  }
}

class AuditDecorator implements Agreement {
  constructor(private wrapped: Agreement) {}
  execute() {
    this.log("start");
    const r = this.wrapped.execute();
    this.log("end", r);
    return r;
  }
}

// Compose any combination:
const agreement = new AuditDecorator(
                    new EncryptionDecorator(
                      new CompressionDecorator(
                        new BaseAgreement())));
Pattern 10 · Structural

Facade

Guild onboarding involved 27 steps across 6 different services. New guild engineers called each step individually, in the wrong order. One simple onboardMerchant() call — hiding all 27 steps inside the Facade.

💡Facade Pattern
A simplified interface to a complex subsystem. The subsystem doesn't change — the Facade hides its complexity. Real-world: SLF4J (logging facade), JDBC, REST APIs as facades over business logic.
class MerchantOnboardingFacade {
  // Hides 6 services and 27 steps:
  async onboardMerchant(merchant: Merchant): Promise<OnboardingResult> {
    const identity  = await this.identityService.verifyInFiveSteps(merchant);
    const contract  = await this.contractService.generateInThreeSteps(identity);
    const payment   = await this.paymentService.configureInSevenSteps(merchant);
    const river     = await this.riverService.subscribeInFourSteps(merchant);
    await this.complianceService.checkInSixSteps(merchant);
    await this.notificationService.sendWelcome(merchant);
    return OnboardingResult.success();
  }
}

// Guild engineer's entire experience:
await facade.onboardMerchant(myGuild); // Done.
Pattern 11 · Structural

Flyweight

50 million trade history records. Each contained a TradeType object, a Currency object, a Kingdom object — duplicated millions of times. Loading as distinct objects: 40GB. The server had 8GB. Flyweight: share the common state, store only unique state per record.

💡Flyweight Pattern
Shares common (intrinsic) state among many objects. Only unique (extrinsic) state is stored per instance. Real-world: Java String interning, integer caching (-128 to 127), character glyphs in text rendering.
// Intrinsic state — created ONCE, shared by millions
class TradeTypeData { // e.g., "GRAIN", "FUTURES"
  description: string;
  taxRate: Percentage;
  regulatoryCode: string;
  // No unique-per-trade data
}

class TradeTypeFactory {
  private static cache = new Map<string, TradeTypeData>();
  static get(typeName: string): TradeTypeData {
    if (!this.cache.has(typeName))
      this.cache.set(typeName, new TradeTypeData(typeName));
    return this.cache.get(typeName)!; // Same object every time
  }
}

// Extrinsic state — stored per trade record
class TradeRecord {
  tradeType: TradeTypeData; // Shared flyweight — NOT a copy
  amount: Money;            // Unique to this trade
  date: DateTime;           // Unique to this trade
}
// 50M records share ~20 TradeTypeData objects
Pattern 12 · Structural

Proxy

The Royal Treasury required 3-second initialization (connecting to secure vaults). Most services needed it rarely — loading eagerly wasted 3s at startup. Additionally: only authorized parties should access it. The Proxy handles both: lazy loading + access control.

💡Proxy Pattern
Same interface as the real object, with additional control. Three flavors:
• Virtual Proxy: Lazy initialization — don't load the real object until needed
• Protection Proxy: Access control — check authorization before delegating
• Remote Proxy: Local facade for a remote resource
Real-world: Spring @Lazy, Hibernate lazy-loaded relations, NGINX reverse proxy. 
class SecureTreasuryProxy implements Treasury {
  private realTreasury: RealTreasury | null = null; // Not loaded yet

  getRecord(merchantId: string, caller: ServiceIdentity): TreasuryRecord {
    // Protection: check authorization first
    if (!this.authService.isAuthorized(caller, "TREASURY_READ"))
      throw new UnauthorizedException();

    // Virtual: lazy-load the expensive real treasury on first use
    if (!this.realTreasury)
      this.realTreasury = new RealTreasury(); // 3-second init — deferred

    return this.realTreasury.getRecord(merchantId);
  }
}

Part III — Behavioral Patterns

How objects talk to each other. Eleven patterns for interaction and responsibility.

Pattern 13 · Behavioral

Chain of Responsibility

Every trade needed: format validation, regulatory compliance check, fraud analysis, kingdom-specific rules, and credit standing verification. Each was different code. Some trades needed all five; some needed two. The decision of which handler handled which trade should not be hardcoded.

💡Chain of Responsibility
A chain of handlers. Each handler either processes the request or passes it to the next. The sender doesn't know which handler will ultimately respond. Real-world: HTTP middleware (Express, Spring Filters), logging handlers, exception handlers.
abstract class TradeValidator {
  protected next: TradeValidator | null = null;

  setNext(v: TradeValidator): TradeValidator {
    this.next = v; return v; // Enables fluent chaining
  }

  validate(trade: Trade): ValidationResult {
    const result = this.check(trade);
    if (!result.passed || !this.next) return result;
    return this.next.validate(trade); // Pass to next
  }

  protected abstract check(trade: Trade): ValidationResult;
}

// Build the chain — order matters
const format = new FormatValidator();
format.setNext(new RegulatoryValidator())
      .setNext(new FraudValidator())
      .setNext(new KingdomRulesValidator())
      .setNext(new CreditValidator());

format.validate(trade); // Flows through chain automatically
Pattern 14 · Behavioral

Command

The Royal Trading Council needed to execute trades, queue them for later, undo incorrectly submitted trades, log all actions for audit, and replay sequences for testing. All of this required the trade action to be an object — something that could be stored, passed, executed, or reversed.

💡Command Pattern
Encapsulates a request as an object with execute() and undo(). Enables queuing, logging, undo/redo, and macro operations. Real-world: Database transactions, Git commits, task queues (Celery), UI undo/redo.
interface TradeCommand {
  execute(): void;
  undo(): void;
}

class ExecuteTradeCommand implements TradeCommand {
  constructor(private trade: Trade) {}
  execute() { tradingEngine.process(this.trade); }
  undo()    { tradingEngine.reverse(this.trade); }
}

class CommandHistory {
  private stack: TradeCommand[] = [];

  execute(cmd: TradeCommand): void {
    cmd.execute();
    this.stack.push(cmd); // Log it
  }

  undoLast(): void {
    const cmd = this.stack.pop();
    if (cmd) cmd.undo();
  }
}

// Queue for later:
commandQueue.add(new ExecuteTradeCommand(grainsForSilverTrade));
// Undo last 3:
history.undoLast(); history.undoLast(); history.undoLast();
Pattern 15 · Behavioral

Iterator

Merchant lists were stored in hash tables, sorted trees, and linked lists — chosen for different access patterns. Code that iterated over merchants shouldn't need to know which structure was used. That was an implementation detail, not the caller's concern.

💡Iterator Pattern
Standard traversal interface for any collection without exposing its implementation. Real-world: Java's Iterator, Python's __iter__, database cursors, JS generators.
interface Iterator<T> {
  hasNext(): boolean;
  next(): T;
}

// The caller never changes — regardless of underlying structure:
function processMerchants(iter: Iterator<Merchant>) {
  while (iter.hasNext()) {
    process(iter.next());
  }
}

// Works with any collection:
processMerchants(new MerchantTreeIterator(sortedTree));
processMerchants(new MerchantHashIterator(hashTable));
processMerchants(new MerchantListIterator(linkedList));
Pattern 16 · Behavioral

Mediator

Seven active services each calling multiple others: 42 possible connections, each needing maintenance. When the Fraud Service's API changed, 6 services needed updating. The Mediator replaced all 42 connections with 7 — one per service to the central coordinator.

💡Mediator Pattern
Central coordinator through which all objects communicate — preventing direct connections. N×(N-1)/2 connections become N. Real-world: Chat rooms, event buses, air traffic control, trading floor auctioneers.
class TradingMediator {
  notify(sender: Service, event: string, data: TradeData): void {
    switch (event) {
      case "TRADE_SUBMITTED":
        this.creditService.check(data);    // Mediator coordinates
        this.fraudService.analyze(data);
        break;
      case "FRAUD_DETECTED":
        this.settlementService.block(data);
        this.notificationService.alert(data);
        break;
      case "CREDIT_APPROVED":
        this.settlementService.proceed(data);
    }
  }
}

// Services communicate ONLY with the Mediator:
mediator.notify(this, "TRADE_SUBMITTED", tradeData);
// Change FraudService: only update Mediator — nothing else changes
Pattern 17 · Behavioral

Memento

Draft Trade Agreements needed save/resume across sessions, rollback to previous versions, and state comparison — all without exposing the Agreement's internals to the storage mechanism.

💡Memento Pattern
Captures and externalizes an object's state for later restoration without violating encapsulation. The Caretaker stores mementos but cannot read them. Real-world: Browser history, git stash, game save states, database savepoints.
class TradeAgreement {
  private terms: Terms;
  private parties: Party[];
  private conditions: Condition[];

  save(): Memento {
    return new Memento({ terms: this.terms, parties: [...this.parties],
                         conditions: [...this.conditions] }); // Deep copy
  }

  restore(m: Memento): void {
    const state = m.getState();
    this.terms = state.terms;
    this.parties = state.parties;
    this.conditions = state.conditions;
  }
}

class DraftHistory { // Caretaker — stores but never reads Mementos
  private history: Memento[] = [];
  save(a: TradeAgreement)  { this.history.push(a.save()); }
  undo(a: TradeAgreement)  { if (this.history.length) a.restore(this.history.pop()!); }
}
Pattern 18 · Behavioral

Observer

When a trade's status changed, 12 subsystems needed to react. Original design: Settlement directly called all 12. Every new subscriber required modifying Settlement. Adding Analytics: Settlement's code changed — which had nothing to do with settlement.

💡Observer Pattern
One-to-many dependency — when the subject changes, all registered observers are notified automatically. Real-world: Java EventListeners, JavaScript addEventListener, React state/effects, Kafka consumers, RxJava.
interface TradeObserver {
  onStatusChanged(trade: Trade, status: TradeStatus): void;
}

class TradeSettlement {
  private observers: TradeObserver[] = [];

  addObserver(o: TradeObserver)    { this.observers.push(o); }
  removeObserver(o: TradeObserver) { this.observers.splice(this.observers.indexOf(o), 1); }

  settle(trade: Trade): void {
    this.processTrade(trade);
    trade.status = TradeStatus.SETTLED;
    // Notify all — Settlement doesn't know or care who they are:
    this.observers.forEach(o => o.onStatusChanged(trade, trade.status));
  }
}

// Add new observer without touching Settlement:
settlement.addObserver(new NotificationService());
settlement.addObserver(new AnalyticsService());
settlement.addObserver(new NewComplianceService()); // Zero Settlement changes
Pattern 19 · Behavioral

State

Trade Agreements passed through: DRAFT → SUBMITTED → UNDER_REVIEW → APPROVED → ACTIVE → EXPIRED. Each state allowed different operations. Original: every method had a massive if (state == DRAFT) ... else if (state == APPROVED)... matrix — grew exponentially with each new state.

💡State Pattern
Each state is an object. The subject delegates behavior to its current state object. Adding a state: one new class, not dozens of scattered conditions. Real-world: Vending machines, traffic lights, order status, TCP connection lifecycle.
interface AgreementState {
  edit(a: Agreement): void;
  submit(a: Agreement): void;
  approve(a: Agreement): void;
}

class DraftState implements AgreementState {
  edit(a)     { a.updateTerms(); }              // Allowed
  submit(a)   { a.setState(new SubmittedState()); }
  approve(a)  { throw new Error("Cannot approve a draft"); }
}

class SubmittedState implements AgreementState {
  edit(a)     { throw new Error("Cannot edit once submitted"); }
  submit(a)   { throw new Error("Already submitted"); }
  approve(a)  { a.setState(new ApprovedState()); }
}

class Agreement {
  private state: AgreementState = new DraftState();
  setState(s: AgreementState) { this.state = s; }
  edit()    { this.state.edit(this); }    // All behavior delegated to state
  approve() { this.state.approve(this); }
}
Pattern 20 · Behavioral

Strategy

Seventeen kingdoms calculated tariffs differently: flat rate, percentage, tiered brackets, seasonal multipliers. The calculation logic belonged outside the core Trade class — it was policy, not business fact. Changing a kingdom's tariff method should touch zero Trade code.

💡Strategy Pattern
A family of interchangeable algorithms — each encapsulated as a class. The client chooses which algorithm to use at runtime. Real-world: Sorting algorithms, payment processor selection, compression strategies, route optimization.
interface TariffStrategy {
  calculate(amount: Money, kingdom: Kingdom): Money;
}

class FlatRateTariff    implements TariffStrategy { calculate(a, k) { return Money.of(50); } }
class PercentageTariff  implements TariffStrategy { calculate(a, k) { return a.multiply(0.05); } }
class TieredTariff      implements TariffStrategy { calculate(a, k) { return this.applyBrackets(a); } }
class SeasonalTariff    implements TariffStrategy { calculate(a, k) { return this.applyMultiplier(a); } }

class TradeTransaction {
  private strategy: TariffStrategy;

  constructor(kingdom: Kingdom) {
    this.strategy = TariffStrategyRegistry.getFor(kingdom);
  }

  calculateTariff(amount: Money): Money {
    return this.strategy.calculate(amount, this.kingdom);
  }
}
// Northern Kingdom changes method: update TariffStrategyRegistry ONLY
Pattern 21 · Behavioral

Template Method

All report generation followed the same flow: fetch → validate → format → apply header/footer → output. The "format" step differed completely for each report type. The overall algorithm is fixed. The specific step varies.

💡Template Method Pattern
Base class defines the algorithm skeleton (final method). Subclasses fill in specific abstract steps. The overall flow can never be changed. Real-world: JUnit setUp/tearDown, Spring JdbcTemplate, HTTP request handlers.
abstract class ReportGenerator {
  // Template method — FINAL, defines the invariant structure
  final generateReport(tradeId: string): Report {
    const data   = this.fetchData(tradeId);    // Same for all
    const valid  = this.validate(data);         // Same for all
    const body   = this.format(data);           // OVERRIDDEN by subclasses
    const report = this.applyHeaderFooter(body); // Same for all
    return this.output(report);                  // Same for all
  }

  protected abstract format(data: TradeData): FormattedContent;

  // Other methods may also be overridden (hooks) or stay fixed:
  protected fetchData(id: string): TradeData { return db.fetch(id); }
}

class MonthlyReportGenerator extends ReportGenerator {
  protected format(data) { return groupByMonth(data).toTable(); }
}
class AuditReportGenerator extends ReportGenerator {
  protected format(data) { return detailEveryAction(data).withTimestamps(); }
}
Pattern 22 · Behavioral

Visitor

The platform had 23 contract types. Tax calculation, audit reporting, and risk assessment each worked differently per type. Adding each operation required modifying every contract class. 23 types × 3 operations = 69 ad-hoc additions scattered through the codebase.

💡Visitor Pattern
Add new operations to elements of an object structure without modifying them. The Visitor implements the operation per type; element classes just call accept(visitor). When to use: Object types are stable; many new operations are expected.
interface ContractVisitor {
  visitSimple(c: SimpleContract): void;
  visitDerivative(c: DerivativeContract): void;
  visitFutures(c: FuturesContract): void;
}

interface Contract {
  accept(visitor: ContractVisitor): void;
}

class SimpleContract implements Contract {
  accept(v: ContractVisitor) { v.visitSimple(this); }
}

// New operation = one new Visitor class — NO contract changes:
class TaxCalculatorVisitor implements ContractVisitor {
  visitSimple(c)     { this.calculateFlatTax(c); }
  visitDerivative(c) { this.calculateComplexTax(c); }
  visitFutures(c)    { this.calculateFuturesTax(c); }
}

class RiskAssessmentVisitor implements ContractVisitor { ... }

// Apply to all contracts:
contracts.forEach(c => c.accept(new TaxCalculatorVisitor()));
Pattern 23 · Behavioral

Interpreter

Merchants defined custom trade rules in a simple language: IF trade.amount > 10000 AND trade.kingdom == "Northern" THEN apply_tariff(HIGH). The platform needed to parse these rules and execute them — a domain-specific language interpreted at runtime.

💡Interpreter Pattern
Defines a grammar for a language and an interpreter that evaluates it. Each grammar rule is a class with an interpret() method. Real-world: SQL parsing, regular expression engines, template engines, rule DSLs.
interface Expression {
  interpret(ctx: TradeContext): boolean;
}

class AmountGreaterThan implements Expression {
  constructor(private threshold: Money) {}
  interpret(ctx) { return ctx.trade.amount.isGreaterThan(this.threshold); }
}

class KingdomEquals implements Expression {
  constructor(private kingdom: string) {}
  interpret(ctx) { return ctx.trade.kingdom === this.kingdom; }
}

class AndExpression implements Expression {
  constructor(private left: Expression, private right: Expression) {}
  interpret(ctx) { return this.left.interpret(ctx) && this.right.interpret(ctx); }
}

// Rule: "amount > 10000 AND kingdom == Northern"
const rule = new AndExpression(
  new AmountGreaterThan(Money.of(10000)),
  new KingdomEquals("Northern")
);

rule.interpret(tradeContext); // true or false

🎯 The Pattern Selector

Your 90-second decision tree. When facing a design problem, start here.

You're CREATING objects: → Need exactly one instance? → Singleton → Don't know which class to instantiate? → Factory Method → Need a compatible family of objects? → Abstract Factory → Complex object with many optional steps? → Builder → Creating same expensive object repeatedly?→ Prototype You're COMPOSING objects: → Two incompatible interfaces must work together? → Adapter → Abstraction and implementation should vary independently?→ Bridge → Part-whole hierarchies (trees)? → Composite → Add behavior without subclassing? → Decorator → Simplify a complex subsystem for common use? → Facade → Millions of similar objects consuming too much memory? → Flyweight → Control access or delay expensive loading? → Proxy You're defining BEHAVIOR: → Multiple handlers, one processes the request? → Chain of Responsibility → Need to queue, undo, or log operations? → Command → Traverse collections without knowing structure? → Iterator → Too many direct connections between N objects? → Mediator → Save and restore object state? → Memento → One state change must notify many dependents? → Observer → Behavior changes entirely based on state? → State → Interchangeable algorithms for the same task? → Strategy → Fixed algorithm structure, varying specific steps? → Template Method → New operations on a stable type hierarchy? → Visitor → Evaluate a language or rule grammar? → Interpreter
