The 6-Step Universal Approach

Use this structure every single time. Interviewers want to see structured thinking — not just answers. This is the framework senior engineers at FAANG use internally.

1
2–3 minutes
Clarify Requirements
Functional: What does the system do? Who are the users? What are key user journeys? What's explicitly out of scope?
Non-Functional: Latency target (p99 < 200ms?). Availability (99.9% = 8.7hr/yr downtime, 99.99% = 52min/yr). Read-heavy or write-heavy? CAP trade-off needed?
2
2 minutes
Estimate Scale — Do the Math Out Loud
DAU → rps → storage/day → bandwidth.
Formula: 1M DAU × 10 actions/day ÷ 86,400 = 115 rps. 100M DAU = 1,150 rps.
Storage: 500B/message × 50M msgs/day = 25 GB/day = 9 TB/year.
Bandwidth: 1MB avg video × 100K concurrent viewers = 100 GB/s — needs CDN.
3
5 minutes
High-Level Design — Box Diagram First
Draw: Client → CDN → API Gateway → Auth → Core Services → DB + Cache.
Don't over-engineer. State your choices confidently: "I'll use PostgreSQL for ACID guarantees here". Get the shape right; details come in step 4.
4
15 minutes — the meat
Deep Dive — APIs, Schema, Critical Flow
APIs: Define 2–3 critical endpoints (method, path, request/response).
DB Schema: Key tables with columns, indexes, relationships.
Critical Flow: Walk through the most complex user journey step-by-step.
Bottlenecks: "This table gets 10K writes/sec — here's how I handle that."
5
5 minutes
Trade-offs — Defend Your Choices
SQL vs NoSQL, strong vs eventual consistency, push vs pull, sync vs async, sharding key choice, caching strategy.
Format: "I chose X over Y because at this scale, [reason]. The downside is [trade-off], which I'd handle by [mitigation]."
6
3 minutes
Failure, Scaling & Observability
SPOFs and circuit breakers, sharding strategy, CDN, rate limiting, blue-green deployments, disaster recovery (RTO/RPO).
Mention: metrics you'd monitor, alerts you'd set, how you'd debug a production incident in this system.

CRITICAL FLOW: News Feed — Fan-out Strategy
PROBLEM: 100M users. Each posts 2×/day. Each follows 500 people. Feed load: whose posts? In what order? In under 200ms? APPROACH 1 ── Fan-out on Write (Push) User posts → Write to Posts DB → Fetch all 500 followers (async Kafka job) → Write post_id to each follower's feed table (Redis ZSET) Feed load: O(1) — just ZRANGE user:feed:{userId} 0 49 ✓ Ultra-fast reads ✗ Celebrity problem: Cristiano (500M followers) = 500M Redis writes per post. Kafka job takes hours. NOT FEASIBLE. APPROACH 2 ── Fan-out on Read (Pull) User posts → Only write to Posts DB Feed load: Fetch all followees → JOIN their posts → Sort by time ✓ Works for celebrities ✗ Slow at scale (O(following × posts)) APPROACH 3 ── HYBRID (What Instagram & Twitter actually use) Regular users ( < 1,000 followers): → Fan-out on Write (precomputed feed in Redis) Celebrity users ( > 1,000 followers): → Fan-out on Read (merged at query time) Feed load sequence: 1. ZRANGE user:feed:{userId} 0 49 ← precomputed Redis feed (regular users) 2. Identify followed celebrities ← cached list, TTL 5min 3. ZRANGE celebrity:{id}:posts 0 9 for each celebrity ← separate call 4. Merge + deduplicate by post_id + rank by score 5. Return top 50 posts in < 200ms FEED RANKING SCORE (not raw chronological): score = (w1 × recency) + (w2 × engagement_rate) + (w3 × affinity) + (w4 × content_type_pref) // Recency: exponential decay. Affinity: how often you interact with this person. VIRAL POST PROBLEM (10M views in 5 minutes): Normal: post_id → cache fetch Viral: 10M rps hit same Redis key ← Cache stampede Fix 1: Lazy locking — only 1 thread regenerates, others wait (Redisson) Fix 2: Local L1 cache in each app server (10s TTL) to absorb burst Fix 3: Pre-warm hot content cache before expected spike (known events) VIDEO PIPELINE (Netflix / YouTube): Upload → S3 raw bucket → Kafka event → Transcoding farm (FFmpeg workers) Transcode: [4K,1080p,720p,480p,360p,240p] × [H.264,H.265,VP9,AV1] = 24 variants Chunk each variant into 2-10s segments (MPEG-DASH or HLS) Store chunks in S3: /video/{id}/{quality}/{segment}.mp4 Netflix Open Connect: cache popular chunks on ISP-level CDN servers ABR player: monitors bandwidth every 2s, switches quality segment-by-segment
🌐
Social Graph DB
followers(user_id, follower_id) in PostgreSQL up to ~100M rows. At billion-scale: adjacency list in Cassandra (user_id → list of follower_ids). Graph DB (Neo4j) for friend-of-friend and recommendation queries. Redis ZSET for "people you may know".
📈
Trending Topics
Redis ZINCRBY trending:{window} 1 {hashtag} per event. Background job every 15min: ZREVRANGE top 20. At Twitter scale: Count-Min Sketch (probabilistic, ~1% error, 1000× less memory). Sliding window via two Redis ZSETs: current + previous window.
🎥
Adaptive Bitrate (ABR)
Player measures throughput every 2s. If >25Mbps → request 4K chunk. Falls to 1080p at 8Mbps, 720p at 4Mbps. Netflix uses VMAF quality metric — compares chunks perceptually, not just by bitrate. Proactively prefetches next 3 segments.
🤖
TikTok For You Algorithm
Signals: completion rate (3x weight), shares (2x), comments, likes (1x). Reinforcement learning: each user is a state, each served video is an action. Reward = engagement. Cold start: new users get top 7 trending per category, then personalize based on first 20 interactions.
🛡️
Content Moderation
Upload → AI pipeline: NSFW detector + violence classifier + copyright fingerprinting (Content ID). Confidence > 95%: auto-remove. 70–95%: human review queue. <70%: publish. YouTube checks 500 hours of uploaded video per minute using ML models trained on 100M+ labeled samples.
💾
Image Storage at Scale
Original → S3. Resize on demand: 4 standard sizes (thumbnail, medium, large, original). CDN caches all variants. Content-addressed storage: SHA256(bytes) = filename → automatic deduplication. Instagram stores billions of photos in Cassandra with custom keys; hot photos in Memcached.

CRITICAL FLOW: WhatsApp — Exactly-Once Message Delivery
MESSAGE SEND — User A → User B: 1. A's app generates: messageId = UUID() ← Before sending. Critical for idempotency. 2. A's WebSocket → Chat Server A 3. Chat Server A: a) Write to Message DB (Cassandra) Key: (conversation_id, timestamp) — allows ordered reads per conversation b) Check Redis: GET user:B:server ← Which WebSocket server is B on? → B Online: Route to Chat Server B → Push to B's WebSocket B's app sends ACK → Server marks "delivered" ← double tick ✓✓ → B Offline: Store in undelivered_msgs table (Cassandra) Send push notification via FCM (Android) / APNs (iOS) When B reconnects: fetch all msgs since last_sync_cursor 4. B reads message → App sends READ receipt → Chat Server → to A ← blue ticks ✓✓ EXACTLY-ONCE GUARANTEE: Client retries if no ACK within 5s (network failure). Same messageId sent. Server: INSERT INTO messages WHERE messageId = ? ON CONFLICT IGNORE → Idempotent: second insert with same messageId: no-op. No duplicate stored. PRESENCE SERVICE (Online/Offline): Heartbeat: Client pings every 10s → Redis: SET user:{id}:online 1 EX 30 Status = online if key exists. "Last seen" = stored in DB when TTL expires (key gone). Scale: 1B users × 1 Redis key × ~20 bytes = 20GB. Fits in Redis cluster. GROUP CHAT (1000 members): Naive: Push to 1000 WebSocket connections synchronously ← Too slow. Blocks. Solution: Kafka topic per group_id. Each online member's chat server subscribes. Chat Server B is subscribed to groups B is in. Receives event, pushes to B's WebSocket. Offline members: pulled in batch (last N messages) on reconnect. E2E ENCRYPTION (Signal Protocol): Key agreement: X3DH (Extended Triple Diffie-Hellman) — keys derived on first message Each message encrypted with ephemeral key (ratcheting). Server stores only ciphertext. Even WhatsApp cannot read your messages. Keys exist only on devices.
🔌
WebSocket at Scale
Each WebSocket = 1 persistent TCP connection. 1 server holds ~65K connections. At 500M concurrent users: need 8M+ servers. Solution: stateful routing — user_id → server_id stored in Redis. When routing: look up server, forward via internal HTTP or gRPC.
📱
Multi-Device Sync
devices(user_id, device_id, push_token, last_sync_offset). Each device independently tracks its Kafka offset. On new device login: fetch all messages from offset 0 (or last 30 days). Kafka retention: 7 days default, configurable. All devices of one user are independent consumers.
📨
Notification Fan-out
Event → Kafka topic:notifications. Fan-out workers consume, look up user preferences (push enabled? email? SMS?). Route to FCM/APNs/Twilio. Priority queue: urgent (payment) = immediate, marketing = batched. Exponential backoff on delivery failure. DLQ for undeliverable.

🗺️
GeoHash Indexing
World divided into hierarchical grid. Cell = base32 string: "u4pruyd". Nearby cells share long prefix. Precision: 7 chars = 153m × 153m cell. Redis GEO uses sorted set with score = 52-bit GeoHash integer. Range query: decode prefix, find neighboring cells, union results.
📍
GPS Drift Handling
Kalman filter smooths noisy GPS. If driver shows 200m jump in 1s (impossible): discard, use dead reckoning (last_pos + speed + heading × Δt). If offline >30s: freeze map position, show "Locating..." UI. Resume tracking when ping received. Never show stale coordinates as current.
🧭
Google Maps Routing
Road graph: 30M+ segments. Nodes = intersections, edges = road segments (with weights = travel time). A* algorithm: heuristic = straight-line distance. Hierarchical: first route on highways, then zoom in. Real-time: Flink updates edge weights every 30s from millions of GPS probes. ETA = predicted travel time (ML model).

CRITICAL FLOW: BookMyShow — 100K Users, 200 Seats, 9:00 AM
SCENARIO: Avengers 5. 200 seats. At exactly 9:00:00 AM, 100,000 users hit "Book Now". Your system MUST NOT double-book a single seat. STEP 1: PRE-SALE WARM-UP (before 9 AM): Pre-load seat map into Redis: HSET show:{showId}:seats A1 "AVAILABLE" A2 "AVAILABLE" ... Pre-populate cache with show metadata (movie, theatre, timing) Prepare rate limiting: SET show:{showId}:concurrent 0 STEP 2: USER SELECTS SEATS — LOCKING STRATEGY: Off-peak (low contention) → Optimistic Locking: SELECT id, version FROM seats WHERE id IN (A1,A2) AND status='AVAILABLE' UPDATE seats SET status='LOCKED', version=version+1, locked_until=now()+5min WHERE id IN (A1,A2) AND version=5 ← CAS: fails if version changed by someone else IF rows_affected == 0: seat taken → return 409 Conflict 9 AM rush (high contention) → Redis Distributed Lock (most scalable): MULTI GET seat:A1:lock ← Nil means available SET seat:A1:lock {userId} NX EX 300 ← NX=only if not exists, EX=5min TTL SET seat:A2:lock {userId} NX EX 300 EXEC If any SET returned nil: someone else locked it → fail → All succeed: seats temporarily held for this user STEP 3: SAGA PAYMENT FLOW: T1: Lock seats (done above) T2: POST /payments → Payment Service → Stripe T3: Confirm booking → Generate ticket → Kafka: "booking.confirmed" T4: Notification Service → Email + SMS confirmation COMPENSATION (if T2 fails: payment declined): Publish "booking.failed" event Seat Service: DEL seat:A1:lock → seats return to AVAILABLE Notification Service: send failure email STEP 4: HANDLE 100K CONCURRENT USERS: Rate limit per show: Redis INCR show:{showId}:concurrent > 1000 → HTTP 429, "Please wait" Virtual queue: WebSocket feedback "Position 847 in queue" Seat lock expiry: Redis TTL handles it automatically. No cleanup job needed.
🔐
Optimistic vs Pessimistic Lock
Optimistic: Assume no conflict. Read → update → fail if version changed. No blocking. Good for low contention (Tuesday matinee). Pessimistic: Acquire lock before read. Guaranteed exclusivity. Use for high contention (Avengers opening night). DB-level: SELECT FOR UPDATE.

Airbnb Availability
calendar(property_id, date, status, booking_id). Check: SELECT COUNT(*) WHERE property_id=X AND date BETWEEN checkin AND checkout AND status='AVAILABLE'. Book: UPDATE calendar SET status='BOOKED', booking_id=? WHERE status='AVAILABLE' AND version=?. Atomic with optimistic lock.
🔑
Idempotency Key
Client generates UUID before first request. Sends same key on all retries. Server: UNIQUE constraint on idempotency_key column. INSERT ON CONFLICT(key) DO NOTHING — return saved result. If retried: Stripe returns same charge result. Prevents any duplicate transaction.

CRITICAL FLOW: Dropbox — 10GB Upload, Interrupted, Resumed
UPLOAD FLOW: Step 1: CLIENT CHUNKING Split file into 4MB chunks. Hash each: chunkHash = SHA256(chunk_bytes) 10GB file = 2,500 chunks. fileHash = SHA256(all bytes concatenated) Step 2: DEDUPLICATION CHECK POST /upload/init Body: { fileHash: "a3f9...", chunkHashes: ["b22d...","c71e...", ...] } Server checks each chunkHash against chunk_store table: Response: { "b22d...": "EXISTS", "c71e...": "NEEDED", ... } → "EXISTS" chunks: skip upload entirely (same file uploaded by another user!) → Only "NEEDED" chunks are uploaded. Can save 60-80% of bandwidth. Step 3: PARALLEL UPLOAD Upload "NEEDED" chunks in parallel (5 concurrent connections) Each chunk: PUT /chunks/{chunkHash} → Storage Service → S3 (key = chunkHash) Step 4: NETWORK DROPPED AT CHUNK 1,500 (6GB) Client tracks acked_chunk_hashes set. On resume (2 days later): GET /upload/{fileHash}/status Server returns: { committed: ["b22d...","c71e...", ...1498 hashes] } Client uploads only the missing 1,000 chunks. Perfect resume from interruption. Step 5: FINALIZE POST /upload/complete Body: { fileHash, orderedChunkHashes: [...] } Server creates: file → [chunk_hash_1, chunk_hash_2, ..., chunk_hash_2500] No data duplication. Chunks shared across all users who have the same content. SYNC DELTA (only changed chunks travel the network): File modified on Device A → compute new chunk hashes → diff with stored hashes Upload only changed chunks (e.g., 3 chunks changed out of 2500 = 12MB instead of 10GB) Sync Service → Kafka: "file.updated" event for all user's other devices Device B pulls manifest, downloads only changed chunks, reassembles file S3 DURABILITY — 11 NINES (99.999999999%): Erasure coding (not plain 3× replication): Object split into k=6 data blocks + m=4 parity blocks (Reed-Solomon) Any 6 of 10 blocks can reconstruct the full object 10 blocks stored across 3+ Availability Zones Annual data loss probability: 0.00000000001% — 1 object per trillion per year
📦
Block vs Object vs File
Block (EBS): Raw device access, lowest level. For DBs needing fast random I/O. Object (S3): Key-value, unlimited scale, no file hierarchy, eventual consistency. Best for unstructured data (media, backups). File (EFS/NFS): POSIX hierarchy, concurrent access from multiple instances.
☁️
CDN Cache Invalidation
TTL: Cache-Control: max-age=86400. Content expires, CDN re-fetches. Purge API: Explicit DELETE by URL. Instant. Use for security or major updates. Versioned URLs: /img.v3.jpg — change URL = new cache entry. Zero invalidation needed. Best practice for static assets.
🔀
Conflict Resolution
Last-Write-Wins: Simplest, loses data. Vector clocks: Track causality, detect concurrent edits. OT (Google Docs): Transform operations mathematically so both concurrent edits merge correctly. CRDT: Convergent structures that auto-merge without coordination. Best for offline-first apps.

CRITICAL ARCHITECTURE: Inverted Index + Autocomplete
INVERTED INDEX — <200ms search across billions of docs: Corpus: Doc 1: "Kafka handles distributed message streaming efficiently" Doc 2: "Kafka is a distributed commit log" Doc 3: "Redis is fast in-memory storage" Build inverted index at indexing time: "kafka" → [ {doc:1, tf:1, pos:[0]}, {doc:2, tf:1, pos:[0]} ] "distributed" → [ {doc:1, tf:1}, {doc:2, tf:1} ] "redis" → [ {doc:3, tf:1} ] "fast" → [ {doc:3, tf:1} ] Query: "kafka distributed" 1. Fetch posting lists for "kafka" and "distributed" 2. Intersect: docs that contain BOTH = [doc1, doc2] 3. Score via BM25 (better than TF-IDF for varying doc lengths): BM25(t,d) = IDF(t) × (tf × (k+1)) / (tf + k × (1 - b + b × docLen/avgDocLen)) 4. Rank by score, return top 10 HORIZONTAL SCALING (Elasticsearch): Index → shards (e.g., 6 primary shards based on doc_id hash) Each shard = independent Lucene index with its own inverted index Query fan-out: coordinator broadcasts query to all 6 shards Each shard returns its local top-K results Coordinator merges + re-ranks global top-K → return to client Replicas: each shard has 1+ replicas for HA and read scaling TYPEAHEAD — response in <50ms: Option A: TRIE (in-memory) children["g"]["o"]["o"]["g"]["l"]["e"] → search_node{frequency:10M, top_suggestions:["google","google maps",...]} Problem: at Google scale = 100GB+ of trie. Can't fit in single machine. Option B: REDIS SORTED SET (production — what most companies use) For each prefix "go": ZADD prefix:go 10000 "google" 8000 "good" 6000 "gopro" Query: ZREVRANGE prefix:go 0 9 → top 10 in O(log N) Background job: update scores hourly from query logs Storage: ~50 bytes per prefix×suggestion. For 1M top queries = manageable. WEB CRAWLER: URL Frontier (priority queue + politeness) → Fetcher (HTTP client) → Parser (DOM, link extractor) → URL Normalizer → Bloom Filter (seen_urls ~100B bits for 10B URLs, 1% FP rate) → if not seen: URL Frontier (with priority = PageRank score) Politeness: per-domain rate limiting (1 req/sec), respect robots.txt, crawl-delay header Anti-trap: max depth=20, cycle detection, SimHash for near-duplicate content

FOR UPDATE SKIP LOCKED
PostgreSQL's secret weapon for job queues. SELECT ... FOR UPDATE locks promising rows. SKIP LOCKED skips already-locked rows instead of waiting. 10 workers query simultaneously: each picks a different row. No contention, no blocking. This alone handles most job queue scaling needs up to 50K jobs/min.

Priority Queues

JWT Anatomy
Three parts: HEADER.PAYLOAD.SIGNATURE. Header: alg (RS256/HS256), typ=JWT. Payload: iss, sub (userId), exp, iat, custom claims (roles, tenantId). Signature: HMAC-SHA256 or RSA. Verify: decode header → check alg → verify signature with key. Never trust payload without verified signature.

Token Refresh Flow
Access token: short-lived (15min). Refresh token: long-lived (7d), stored in DB, single-use. On expiry: POST /auth/refresh { refreshToken } → server validates in DB, issues new access token + rotating refresh token. Invalidate old refresh token immediately. Logout: delete refresh token from DB.

Cross-Cutting Concepts

These show up in every system design. Mention them proactively — interviewers expect you to bring these up without being asked.

Redis Cache
Hot data: user sessions, feed items, rate limits, presence, feature flags, leaderboards. Cache-aside pattern. TTL per use case. Warning: not a primary store. Can lose data on restart without AOF/RDB persistence.
Kafka
Async streaming: service decoupling, analytics pipelines, event replay, fan-out (notifications). High throughput (millions/sec). Durable. Use when: order matters, need replay, high fan-out, or eventual consistency is acceptable.
PostgreSQL
ACID critical: payments, bookings, user accounts, inventory. Foreign keys, transactions, complex joins. Single-leader replication. Read replicas for scaling reads. Shard with Citus for write scale.
Cassandra
High write throughput + time-series: messages, events, IoT sensor data, activity logs. Design by access pattern (no joins). Consistent hashing + replication factor 3. Tunable consistency (QUORUM for balance).
Elasticsearch
Full-text search + aggregations: log analytics, user search, complex filters. Always a secondary index — sync from primary DB via CDC (Debezium → Kafka → Elastic). Never your source of truth.
CDN
Static + media content: images, video chunks, JS/CSS bundles. TTL via Cache-Control headers. Reduce origin load by 90%+. Versioned URLs for instant invalidation without purge calls.
SAGA Pattern
Distributed transactions: any multi-service operation (booking, order, payment). Choreography (events, decentralized) for simple flows. Orchestration (central coordinator) for complex flows needing visibility. Always define compensation steps.
Outbox Pattern
Atomic DB write + event publish: prevents dual-write inconsistency. Write DB record + event to outbox table in one transaction. Outbox relay publishes to Kafka asynchronously. Required for: any state change that must also publish an event reliably.
Circuit Breaker
Prevent cascade failures: every inter-service call needs a circuit breaker. Closed → Open after N failures. Half-Open on timeout probe. Resilience4j in Java. Measure: failure rate in rolling time window.
Idempotency Key
Safe retries: client generates UUID before first request. Sends same key on all retries. Server: UNIQUE constraint → skip if already processed, return stored result. Required for: all payment, booking, and mutation endpoints.
Consistent Hashing
Distribute across nodes: minimal reshuffling when nodes join/leave. Virtual nodes for even distribution. Used by: Redis Cluster, Cassandra, DynamoDB. Key → hash position → first node clockwise on ring.
CQRS
Separate read/write models: write to PostgreSQL, sync to Elasticsearch for search, sync to Redis for fast reads. Sync via Kafka. Use when read/write scaling requirements diverge significantly, or when read shape ≠ write shape.
Bloom Filter
Probabilistic membership: "Is this URL already crawled?", "Has this user seen this notification?". False positives OK. No false negatives. 10 bits/element → 1% FP rate. 100B items = 125GB — still feasible.
Observability
Always mention all 3 pillars: Metrics (Prometheus/Grafana: latency p95/p99, error rate, saturation), Logs (structured JSON → ELK), Traces (OpenTelemetry → Jaeger: correlation_id through all services). SLO/SLA definitions.
