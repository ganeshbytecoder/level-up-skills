Prologue

The Archivist's Invitation

Maren met Kira at the entrance to the Hall the morning after Kira's first major production incident — a deployment that had taken the city's payment system offline for three hours during the Harvest Festival peak.

"Come," Maren said simply. "I want to show you something."

The Hall of Failures was not a place of punishment nor shame. It was a museum. Each gallery contained the careful, loving preservation of a disaster — not to remember who failed, but to remember what patterns had caused the failure. Not names, not dates, not blame. Only: what happened, why it happened, and what a different shape would have prevented it.

"In forty years of studying production failures," Maren said as they entered, "I have found that almost every one falls into one of twelve categories. The same twelve. Over and over. Different systems. Different companies. Different years. The same twelve shapes."

Gallery One

The Human Error Gallery

Failures caused by human action without sufficient safeguards

The largest gallery in the hall. Every failure here had a human being at the triggering moment. But that was never the root cause — human error is expected. The root cause was always the same: a system that had no safeguard between a human intention and a catastrophic outcome.

💀 The Unreversible Command
An engineer runs a database command directly in production that cannot be undone. No staging test. No dry-run. No second approval. Result: 72 hours of data loss.

Prevention: No direct production DB access. All changes via migration scripts versioned in git. Dry-run mode required. Multi-person approval for destructive operations.
💀 The Accidental Flag
A feature flag is accidentally enabled in production instead of staging — activating unfinished code for all users. Result: billing system sends incorrect invoices to 2M merchants.

Prevention: Environment-specific flag namespaces. UI clearly distinguishes production from staging flags. Audit log on every flag change.
💀 The Forgotten Rollback
A deployment is made. Five minutes later, error rates spike. The engineer who knows how to roll back is on a flight. No documented rollback procedure. No runbook. Two hours to diagnose what took five seconds to introduce.

Prevention: Runbook for every deployment. Rollback procedure tested quarterly. On-call engineer briefed on every deployment before it happens.
Gallery Two

The Missing Limits Gallery

Failures caused by infinite requests, unbounded data, and missing constraints

💀 The Runaway Queue
A job queue has no maximum depth limit. One malfunctioning consumer stops processing. The queue grows to 40 million items. When the consumer restarts, it tries to process all 40M simultaneously — taking down the database that stores the queue state.

Prevention: Always set max queue depth. Dead Letter Queue for items exceeding retry limit. Consumer lag alerting — alert before queue is full, not after.
💀 The Unbounded Query
An API endpoint returns all records matching a query — with no LIMIT clause. In development this returns 50 rows. In production it returns 4 million. The JSON serialization allocates 8GB of heap. The service OOMs and dies.

Prevention: All list endpoints require explicit pagination. Server-enforced maximum page size. Never trust the client to set a reasonable limit.
💀 The Missing Rate Limit
One client — a rogue analytics job — sends 50,000 requests per second to an API with no rate limiting. The database cannot respond to any other clients for 40 minutes.

Prevention: Rate limit per client ID on every public and internal API. Token bucket with 429 responses. Circuit break the offending client automatically when sustained high rate is detected.
💀 The Infinite Retry Loop
Service A calls Service B. B returns a 503. A retries immediately. B retries its own downstream. Every retry generates more retries. Within 90 seconds, total request rate is 40,000/second across the cluster. All services go down.

Prevention: Exponential backoff + jitter on all retries. Maximum retry count (usually 3). Jitter prevents synchronized retry storms. Circuit breakers stop cascading retries.
Gallery Three

The Dependency Failure Gallery

Failures caused by trusting that everything external will always work

💀 The Blocking Call Without Timeout
Service A calls Service B with no timeout configured. B starts responding slowly — not down, just slow. A's threads accumulate waiting for B's response. In 3 minutes, all of A's threads are blocked. A stops responding to all requests — including unrelated ones that don't call B at all.

Prevention: Every network call must have a timeout. Default timeouts in the SDK are often wrong — set them explicitly. Alert when response time approaches timeout.
💀 The Missing Circuit Breaker
The Fraud Detection Service goes down. Every payment request times out waiting for fraud detection (10 seconds). Users see spinning loaders. The timeout thread pool fills. The Load Balancer marks the Payment Service as unhealthy. The Payment Service has been taken down by a non-critical dependency.

Prevention: Circuit Breaker on every inter-service call. Define fallback behavior: can payment proceed without fraud check? (With risk flag?) Can fraud check be async?
💀 The Third-Party API Gone Dark
The system depends on an external geocoding API. One Tuesday morning, the provider has an outage. Every delivery address is unresolvable. Thousands of orders fail silently — showing success to the merchant but failing internally.

Prevention: Cache previous successful responses with generous TTL. Graceful degradation: accept the order without geocoding, geocode asynchronously. Never let a non-essential third party take down core functionality.
Gallery Four

The Data Inconsistency Gallery

Failures caused by state that doesn't agree with itself

💀 The Dual Write
The service writes to the database, then publishes to Kafka. The DB write succeeds. Before the Kafka publish, the service crashes. The event is never published. The order exists in the DB but downstream services never know — inventory is never decremented, shipping is never triggered.

Prevention: Outbox Pattern: write the event to the same DB transaction as the data. A relay service publishes from the outbox table. Atomic and guaranteed.
💀 The Missing Idempotency
A payment request is made. The response is lost in transit — the client doesn't know if it succeeded. The client retries. The payment is processed twice. The merchant is charged double. The refund process takes 5 business days.

Prevention: Every mutating operation accepts an idempotency key. Check key before processing. Return previous result if key was already processed. Stripe and every payments API implement this.
💀 The Race Condition
A merchant's account balance is $100. Two payments of $80 are initiated simultaneously. Both read the balance: $100. Both check: sufficient. Both deduct $80. Final balance: -$60. Overdraft of $60 — in a system with no overdraft protection.

Prevention: Optimistic locking (version field on records) or pessimistic locking (SELECT FOR UPDATE). Database-level constraints. Atomic operations (compare-and-swap).
Gallery Five

The Configuration Gallery

Failures caused by the wrong setting in the wrong place

💀 The Wrong Environment Variable
A deployment to production accidentally uses staging's database URL. For four hours, production traffic is processed against the staging database. Staging data is corrupted. Production has four hours of unrecorded transactions.

Prevention: Config validation on startup — fail fast if critical config is wrong. Environment-specific checks. Alert on unexpected database connection counts in production.
💀 The Connection Pool Too Small
The service has a connection pool of 5. Load increases from 20 rps to 200 rps. Threads queue for available connections. The 95th percentile latency climbs from 20ms to 8 seconds. The service appears "up" to health checks but is functionally down for users.

Prevention: Connection pool size based on load testing. Alert on connection pool exhaustion. Measure pool wait time as a metric.
💀 The Memory Leak Nobody Noticed
A new version introduces a small memory leak — 1MB per 1000 requests. At 10 rps, unnoticeable. After 6 hours, the service has leaked 2GB and is OOM-killed. Since the deployment was 6 hours ago, no one connects the crash to the deployment.

Prevention: Monitor JVM/heap metrics continuously. Alert on steady upward memory trend. Canary deployment with extended monitoring before full rollout.
The Gallery of Real Failures

Twenty Real Outages — Every Pattern Illustrated

Amazon, GitHub, Facebook, Stripe, Cloudflare, and more — real incidents with real lessons

🏢 Amazon S3 · 2017 · 4 hours
A Typo That Took Down Half the Internet
An engineer debugging the S3 billing system accidentally typed a parameter that removed too many servers from the capacity management system. S3 entered a state it had never been in — taking down almost every major website and app that relied on it.
→ Lesson: Validate parameters against a safe range before executing. Manual operations in production need a second approval. Recovery procedures must be tested before they're needed. Never assume safe config ranges.
💻 GitHub · 2018 · 24 hours
The Split-Brain Database That Wouldn't Heal
A maintenance operation created network partitioning between GitHub's primary database and replica. When the network recovered, the replica had become a primary. Two primaries now existed. Merging the diverged data without data loss took 24 hours.
→ Lesson: Split-brain scenarios must have a documented resolution procedure tested in drills. Automated failover without careful validation can cause more damage than the original failure.
📘 Facebook · 2021 · 6 hours
BGP Routing That Locked Engineers Out
A routine BGP update accidentally removed all of Facebook's route advertisements. Facebook entirely disappeared from the internet. The BGP update also took down the internal systems Facebook engineers used to connect remotely — locking them out of the tools needed to fix the issue. Engineers physically drove to the datacenter to fix it.
→ Lesson: Emergency access to fix systems must not depend on the systems being fixed. Out-of-band access procedures. Change validation with rollback capability before BGP changes.
💳 Stripe · 2019 · 2 hours
The SSL Certificate That Expired During Peak Traffic
An SSL certificate expired during peak payment processing hours. Stripe's infrastructure had automated renewal, but a configuration change two weeks earlier had quietly disabled it. Thousands of payment integrations began returning SSL errors.
→ Lesson: Certificate expiry monitoring with 30/14/7/1 day alerts. Test renewal automation regularly. Config changes should audit the effect on automated operations.
☁️ Cloudflare · 2019 · 27 minutes
The Regex That Consumed All CPU
A new WAF rule introduced a poorly written regex that backtracked catastrophically — consuming 100% CPU across all Cloudflare's processing nodes globally within seconds of deployment. Traffic dropped to near zero worldwide.
→ Lesson: Test regex patterns with performance profiling. Deploy WAF rules gradually with canary rollout. Immediate rollback capability for all rule changes.
🌐 AWS us-east-1 · 2011 · Multiple days
The EBS Cascade That Exposed Single-Region Danger
An automated network change caused EBS volumes to enter a "re-mirroring storm" — every EBS volume began re-mirroring simultaneously, consuming all available capacity. Services that had been architected as single-region with no failover stayed down for days.
→ Lesson: Critical systems must be multi-region. Single availability zone = single point of failure. Health check design must differentiate "service itself is broken" from "cloud provider has an issue."
🎯 Reddit · 2016 · 1.5 hours
The Autoscaler That Deleted All the Servers
An autoscaling policy was misconfigured to scale down aggressively during low-traffic periods. It correctly identified that traffic was low at 3 AM and terminated all servers. When traffic returned, it attempted to scale up — but the AMI used for new instances had a misconfigured database connection string. Startup failed on every new instance.
→ Lesson: Set minimum instance counts that prevent autoscaler from going to zero. Test scale-up procedures from zero. New AMI validation before it enters the autoscaling group.
🗄️ GitLab · 2017 · 18 hours of data loss
The Database Deletion Nobody Could Undo
A DBA, trying to remove data from the wrong server, ran a command that deleted the primary production database. The database backup had been failing silently for months. The replication was lagging and couldn't recover the lost data. 6 hours of data and 5000 projects were permanently lost.
→ Lesson: Backup success must be monitored — not just that the backup ran, but that it completed and is restorable. Test restores monthly. Separate human error from database deletion (delete → staging first, then production).
📦 npm · 2016 · Multiple hours
The left-pad Unpubish That Broke the Internet
A developer unpublished a tiny 11-line npm package (left-pad) after a trademark dispute. Thousands of packages that depended on it — including React and Babel — failed to build worldwide, breaking CI/CD pipelines globally.
→ Lesson: Vendor lock-in risk, even in open source. Pin exact versions in lock files. Mirror critical dependencies internally. Understand the transitive dependency tree — left-pad was a 4th-level transitive dependency for most affected packages.
📱 Instagram · 2012 · Ongoing degradation
The N+1 Query That Scaled Quadratically
An endpoint that loaded a user's feed executed one query per post to load author details. At 100 followers with 5 posts each: 501 queries per page load. At 1M users: millions of queries per second, nearly all redundant. Database could not keep up with O(n) queries per user.
→ Lesson: Profile all database queries in production traffic. N+1 queries are invisible in unit tests. ORM queries must be audited. Batch loading (DataLoader, JPA @BatchFetch) is mandatory at scale.
🐦 Twitter · 2009 · "Fail Whale" Era
The Database That Couldn't Scale with Social Graph Reads
Twitter's monolithic MySQL database could not serve the read volume generated by celebrity users with millions of followers. Every tweet from a celebrity triggered millions of timeline updates in a fan-out write model. The database became a bottleneck that no amount of hardware could solve.
→ Lesson: Fan-out write vs fan-out read is an architectural decision with massive scale implications. Hybrid approaches (cache for celebrities, fan-out-on-write for small followings) must be designed early, not retrofitted.
⚡ Heroku · 2017 · 2 hours
The Scheduler That Ran 10,000 Times in 5 Minutes
A bug in Heroku's scheduler caused a scheduled job to run in an infinite loop instead of once. In 5 minutes, it sent 10,000 emails to 500,000 users. Support requests flooded in from users confused about why they were receiving duplicate system emails.
→ Lesson: Scheduled jobs must be idempotent. Execution count alerting: if a job that runs daily has run 5 times today — alert immediately. Emails need deduplication at the mailer level.
🔐 LastPass · 2022 · Ongoing
The Developer Machine That Started an Enterprise Breach
A threat actor compromised a developer's machine, gained credentials, and accessed LastPass's development environment. Over months, they exfiltrated encrypted password vault backups and encryption keys from different systems.
→ Lesson: Blast radius of employee compromise must be bounded. Short-lived credentials. Least privilege for every access. Separation of dev environments from production data — always.
🎮 PlayStation Network · 2011 · 23 days
The Breach That No One Was Ready to Recover From
An intrusion into PSN systems led to 77 million account records being compromised. Sony chose to take the network down rather than continue operating with a breach in progress. Recovery was complicated by unclear incident response procedures and legal requirements across multiple jurisdictions.
→ Lesson: Incident response procedures must be documented and drilled before they're needed. Legal and PR involvement in security incident response planning. Recovery time is determined by preparation, not by the incident itself.
🚘 Uber · 2015 · Multiple hours
The Deployment That Broke Surge Pricing at New Year
A deployment of changes to Uber's pricing service was made on New Year's Eve — the highest-traffic day of the year. The deployment introduced a bug in surge pricing calculation. Thousands of rides were priced incorrectly. Revenue impact and customer refund costs were significant.
→ Lesson: Deployment freeze windows for peak traffic events. Canary releases that include business metric validation (not just technical health). No deployments without verified rollback paths.
📊 Google · 2014 · 5 minutes
The BGP Leak That Made Google Routes Disappear
A BGP configuration change by a third-party provider accidentally caused Google's IP prefixes to be advertised via routes with incorrect AS paths. Global traffic destined for Google was rerouted or dropped.
→ Lesson: BGP route changes must go through validation pipelines with staged rollout. Route filtering to prevent accepting obviously invalid BGP advertisements. Even Google needs defense against third-party configuration errors.
🏦 TSB Bank · 2018 · 5 weeks
The Platform Migration That Locked Customers Out for Weeks
TSB migrated 5.2 million customer accounts from an old system to a new banking platform. The migration had not been tested at full scale. The new system could not handle production load. Customers were locked out of their accounts for up to 5 weeks.
→ Lesson: Production-scale load testing before migration. Phased migration (not big-bang). Rollback plan that actually works at scale. Never migrate a banking system during peak period without a tested, validated fallback path.
☁️ Azure · 2019 · 9 hours
The Certificate Rotation That Broke Authentication
An automated certificate rotation changed a certificate before all services had been updated to trust the new certificate. Authentication services began rejecting valid tokens. The rollout order had not accounted for dependency between certificate issuance and certificate trust.
→ Lesson: Certificate rotation must be backwards-compatible during the transition window. Both old and new certificates must be trusted simultaneously. Test rotation procedures in staging before production.
🎵 Spotify · 2022 · Multiple hours
The Cache Stampede After a Sudden Outage
A brief datacenter issue caused Spotify's caches to flush simultaneously. When service restored, every client reconnected at once. Every cache was cold. Every request went to the database. The database — not designed to serve 100% of traffic without cache — fell over under the load.
→ Lesson: Cache warmup procedures before returning traffic after an outage. Traffic shaping during recovery (not all clients at once). Database load testing at cache-miss traffic levels.
🔄 AWS DynamoDB · 2019 · 8 hours
The Metadata Partitioning Bug
A bug in DynamoDB's internal metadata service caused it to be overloaded. Requests to DynamoDB were timing out or failing, affecting thousands of AWS customers and downstream services across the internet.
→ Lesson: Cloud provider failures are your business's failures. Multi-region and multi-provider strategies for critical paths. Graceful degradation when the database itself is unavailable.
Act VI — Maren's Last Notes

The Advanced Gallery

Event Sourcing, O(n²) Complexity, Large Transactions, OAuth2 Security, and the patterns that appear in senior-level interviews

⚡ Event Sourcing Gone Wrong
A team adopts Event Sourcing (storing events rather than current state). After one year, replaying 10 million events to rebuild a merchant's current state takes 45 seconds. Every API call that needs current state triggers a full replay.

Pattern: Event Sourcing is powerful but requires Snapshot support. Periodically save the current state as a snapshot. Replay only from the last snapshot, not from the beginning of time.

When to use Event Sourcing: When the history of changes is as important as the current state. Audit trails, financial records, debugging complex state. Not as a default choice.
⚡ The O(n²) Hidden in Production
A guild matching algorithm worked perfectly in testing with 100 merchants. In production with 10,000 merchants, it ran for 8 hours. The algorithm was O(n²) — comparing every merchant to every other merchant.

n=100: 10,000 operations. n=10,000: 100,000,000 operations.

Prevention: Analyze algorithm complexity during code review. Benchmark with production-scale data. Load testing is not just for throughput — also for correctness at scale.
⚡ The Large Transaction Timeout
An end-of-month settlement process runs a single giant database transaction updating 500,000 merchant balances. It takes 45 minutes. At minute 44, the database's transaction timeout kills it. The rollback of 44 minutes of work takes another 20 minutes.

Prevention: Break large operations into batches of 1,000-10,000. Each batch is its own transaction. Track progress. Resume from last successful batch on failure. Never run multi-hour transactions in production.
⚡ The Read Replica Serving Stale Data on Critical Reads
The system routes all reads to read replicas to reduce primary load. The payment verification check reads from a replica. The replica lags 2 seconds behind the primary. A payment is made. The refund check happens within 2 seconds and reads the pre-payment balance from the replica — marking the refund as unsuccessful.

Prevention: Consistency requirements per read. Critical reads (post-write verification) must go to the primary. Analytical reads can go to replicas. Know the replication lag.
⚡ JWT Signature Not Verified
The authentication service issues JWTs. An internal service decodes the JWT payload to extract the user ID — but never verifies the signature. An attacker with a test account crafts a JWT with an admin user ID. The internal service trusts it entirely.

Prevention: Always verify the JWT signature. Always check exp (expiry), iss (issuer), and aud (audience). Never decode without verifying. A JWT is only as secure as the signature verification step.
The FAANG Pre-Flight Checklist

Six Questions Before Any Production Feature

⚡ Run This Before Every Deploy

1
Is every mutating operation idempotent?
Networks fail. Clients retry. Double processing = double charging, double shipping, double deleting. Every write operation must handle being called twice.
2
Are retries bounded with backoff and jitter?
Unbounded immediate retries amplify failures rather than recovering from them. Every retry must wait longer and add randomness.
3
Does every external call have a timeout and circuit breaker?
A slow downstream should not block all your threads indefinitely. A dead downstream should not take down your service.
4
Is schema evolution backward compatible?
Old consumers will still be running when the new schema is deployed. Required fields must never be added without defaults. Field types must never change.
5
Are metrics, logs, and traces in place?
You cannot debug what you cannot observe. Shipping a feature without observability is shipping a feature you cannot fix in production.
6
What is the blast radius and rollback plan?
Every deployment introduces risk. Know who is affected if this goes wrong. Know exactly how to undo it in under 5 minutes.
