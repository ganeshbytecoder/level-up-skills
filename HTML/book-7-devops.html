HOW DOCKER WORKS INTERNALLY + PRODUCTION OPTIMIZATION
ISOLATION PRIMITIVES (Linux kernel features, NOT a VM): Namespaces — process-level isolation: pid: container sees only its own processes (PID 1 = your app) net: container gets its own network stack (eth0, routing table) mnt: container gets its own filesystem view (from image layers) uts: container gets its own hostname ipc: separate shared memory / semaphores user: map container root (uid 0) to unprivileged host uid (e.g., 65534) cgroups — resource limiting: cpu: --cpus=0.5 → container can use max 50% of one core memory: --memory=512m → OOM kill if exceeded io: limit disk read/write throughput No cgroup limits = container can steal all host CPU. Always set limits in prod. DOCKERFILE — BAD vs GOOD (800MB vs 90MB): BAD (common mistakes): FROM node:20 ← pulls full Debian, 1.1GB base COPY . . ← copies node_modules too (huge) RUN npm install ← runs every build. Cache busted by any file change. RUN npm run build EXPOSE 3000 CMD ["node", "src/index.js"] GOOD (multi-stage + optimized layers): # Stage 1: Build FROM node:20-alpine AS builder ← alpine = 7MB base vs 1.1GB debian WORKDIR /app COPY package*.json ./ ← COPY deps FIRST — cache layer until package.json changes RUN npm ci --only=production ← ci = exact lockfile. --only=prod = skip devDeps COPY . . RUN npm run build # Stage 2: Runtime (only production artifacts) FROM node:20-alpine AS runtime RUN addgroup -S app && adduser -S app -G app ← Never run as root WORKDIR /app COPY --from=builder /app/dist ./dist ← Only compiled output, ~5MB COPY --from=builder /app/node_modules ./node_modules USER app ← Switch to non-root EXPOSE 3000 HEALTHCHECK CMD wget -qO- http://localhost:3000/health || exit 1 CMD ["node", "dist/index.js"] Result: 800MB → ~90MB. Attack surface reduced 90%. CONTAINER NETWORKING MODES: bridge (default): veth pair → docker0 bridge → host network Containers communicate via 172.17.0.0/16 subnet Port publish: -p 8080:3000 → iptables DNAT rule host: container shares host network stack directly No NAT. Highest performance. Loses isolation. Use: metrics collection, eBPF overlay (Swarm/K8s): VXLAN tunnel between hosts Container on Host A → VXLAN encapsulation → UDP 4789 → Host B → container none: no networking. Use: batch jobs, security-sensitive workloads .dockerignore (critical for build performance): node_modules/ ← ALWAYS ignore. Never copy into build context. .git/ *.log dist/ coverage/ .env ← NEVER bake secrets into images. Use runtime env vars or secrets.
📦
Image Layers & Cache
Each RUN/COPY/ADD = new layer. Each layer is content-addressed (SHA256). Cache: if instruction + all preceding layers unchanged → cache hit (0ms). Key: order instructions from least to most frequently changing. COPY package.json before COPY src/ — avoids reinstalling deps on src change.
🔐
Docker Security
Never run as root. Use USER directive. Scan images: docker scout or Trivy/Snyk. Read-only filesystem: --read-only flag. Drop capabilities: --cap-drop=ALL --cap-add=NET_BIND_SERVICE. No privileged containers in prod. Secrets: inject via env vars or mounted secret volumes — never COPY or ARG.
🏥
Healthchecks
HEALTHCHECK --interval=30s --timeout=10s --retries=3 CMD curl -f http://localhost:3000/health. States: starting → healthy → unhealthy. K8s uses liveness/readiness probes separately. Docker restart policy: --restart=unless-stopped. Unhealthy container → K8s kills + restarts it automatically.
💾
Volumes vs Bind Mounts
Volume: Docker-managed, /var/lib/docker/volumes/. Best for DB data, persistent state. Survives container removal. Bind mount: maps host path directly. Best for dev (live code reload). tmpfs: RAM-only, ephemeral. Best for sensitive data that must not hit disk (keys, session tokens).

K8s ARCHITECTURE + WHAT HAPPENS WHEN YOU RUN kubectl apply
CONTROL PLANE (brain — runs on master nodes): kube-apiserver — All state changes pass through here. Stateless. Horizontally scalable. etcd — Distributed key-value store. Source of truth for ALL cluster state. Raft consensus. 3 or 5 replicas for HA. NEVER let etcd go down. kube-scheduler — Watches for unscheduled pods. Runs scheduling algorithm to pick node. kube-controller-manager — Runs control loops: Deployment controller: actual replicas? = desired replicas? If not → create/delete pods. Node controller: detect node failure → evict pods after 5min (configurable). Endpoint controller: update Service endpoints when pods start/stop. DATA PLANE (workers — run your workloads): kubelet — Agent on every node. Watches apiserver for pod assignments. Starts containers via CRI (containerd). Reports pod status back. kube-proxy — Programs iptables/IPVS rules for Service ClusterIP routing. Every request to ClusterIP → kernel routes to one of the healthy pods. Container Runtime — containerd (standard). Pulls images, starts/stops containers. WHAT HAPPENS WHEN YOU kubectl apply -f deployment.yaml: 1. kubectl authenticates to kube-apiserver (your kubeconfig cert) 2. apiserver validates manifest → stores in etcd 3. Deployment controller notices: desired=3 replicas, actual=0 → creates 3 Pod specs in etcd 4. Scheduler notices 3 unscheduled pods → runs scoring algorithm: - Filters: enough CPU/memory? taints tolerated? affinity rules met? - Scores: spread across zones? close to data? packed efficiently? → Assigns pod to best node. Writes node name to pod spec in etcd. 5. Node's kubelet notices pod assigned to it → calls containerd to pull image + start container 6. Container starts → kubelet runs readiness probe every 10s Readiness passes → kubelet updates pod status=Ready 7. Endpoint controller: pod Ready → add pod IP to Service endpoints 8. kube-proxy on all nodes: update iptables → traffic now reaches new pod Total time: typically 15-45 seconds per pod. Parallel for multiple pods. ZERO-DOWNTIME DEPLOYMENT (Rolling Update): strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 ← Allow 1 extra pod during update (33% over capacity) maxUnavailable: 0 ← Never reduce below desired. Zero downtime guaranteed. Lifecycle: Old: [v1, v1, v1] Step 1: Start v2 pod → wait for readiness probe to pass Step 2: v2 Ready → add to Service endpoints → remove one v1 from endpoints Step 3: Terminate v1 pod (SIGTERM → 30s grace period → SIGKILL if still running) Step 4: Repeat until all v1 pods replaced. If v2 readiness probe never passes → rollout stalls. Run: kubectl rollout undo deployment/app HORIZONTAL POD AUTOSCALER (HPA): apiVersion: autoscaling/v2 spec: metrics: - type: Resource resource: {name: cpu, target: {type: Utilization, averageUtilization: 70}} - type: External external: {metric: {name: kafka_consumer_lag}, target: {type: Value, value: "1000"}} behavior: scaleUp: {stabilizationWindowSeconds: 30, policies: [{type:Pods, value:4, periodSeconds:60}]} scaleDown: {stabilizationWindowSeconds: 300, policies: [{type:Percent, value:50, periodSeconds:60}]} Scale-up: fast (30s stabilization). Scale-down: slow (300s / 5min stabilization). Reason: false scale-down during traffic spike would cause immediate OOMKills. CRITICAL: PodDisruptionBudget (during node drain/upgrades): minAvailable: 2 ← At least 2 pods must be running during any voluntary disruption. Without PDB: kubectl drain might kill all 3 pods simultaneously → complete outage. With PDB: drain respects budget → always keeps 2 running → graceful cluster upgrade.
🏷️
Taints & Tolerations
Taint a node: kubectl taint nodes gpu-node gpu=true:NoSchedule. No pod can land there unless it tolerates the taint. Toleration in pod spec: tolerations: [{key: gpu, value: "true"}]. Use case: reserve GPU nodes for ML workloads only. Also: system pods on master nodes tolerate node-role.kubernetes.io/master.
📍
Pod Anti-Affinity
Spread replicas across zones so a zone failure doesn't kill all pods: podAntiAffinity.preferredDuringScheduling: topologyKey: topology.kubernetes.io/zone. Hard anti-affinity (required): guarantees spread. Soft (preferred): best-effort. At FAANG scale: always use multi-zone anti-affinity for critical services.
🔐
K8s RBAC
ServiceAccount: identity for pods. Role: permissions in one namespace. ClusterRole: cluster-wide. RoleBinding: bind role to ServiceAccount. Least privilege: app pod gets only get/list pods, not *. Audit: kubectl auth can-i --list --as=system:serviceaccount:default:myapp.
📦
ConfigMaps & Secrets
ConfigMap: non-sensitive config (feature flags, endpoints). Secret: base64-encoded (NOT encrypted by default). Encrypt at rest: enable EncryptionConfiguration in apiserver with AES-CBC or KMS provider. External secrets: use External Secrets Operator → pull from AWS Secrets Manager/HashiCorp Vault. Mount as files or env vars.
⚡
Resource Requests & Limits
requests: scheduler uses this to place pod. limits: hard ceiling — OOMKilled if exceeded. Best practice: set requests=limits for critical services (Guaranteed QoS class). Node overcommit: if pod requests < limits, node can be oversubscribed. CPU throttled at limit, memory OOMKilled.
🗄️
StatefulSets
Ordered pod names: db-0, db-1, db-2. Stable network identity: db-0.db-svc.default.svc.cluster.local. PVC per pod: each gets its own 100GB volume. Scale down: terminates highest index first (db-2, then db-1). Use for: databases (Postgres, Cassandra, Kafka). Never use Deployment for stateful workloads.

PRODUCTION NGINX CONFIG — The Pattern That Handles 100K RPS
WORKER TUNING (top of nginx.conf): worker_processes auto; # = number of CPU cores worker_connections 10000; # per worker. Total = workers × 10000 use epoll; # Linux event model. Non-blocking I/O. Key to 100K rps. multi_accept on; # Accept all pending connections per event loop UPSTREAM (backend pool + health checks): upstream app_servers { least_conn; # Route to server with fewest active connections keepalive 32; # Pool of 32 persistent connections per upstream server server app1:3000 weight=3; # 3× more traffic than weight=1 servers server app2:3000 weight=3; server app3:3000 weight=1 backup; # Only used if primary servers are DOWN server app4:3000 max_fails=3 fail_timeout=30s; # Remove after 3 failures in 30s } SERVER BLOCK (HTTPS + security headers): server { listen 443 ssl http2; server_name api.yourapp.com; ssl_certificate /etc/letsencrypt/live/yourapp/fullchain.pem; ssl_certificate_key /etc/letsencrypt/live/yourapp/privkey.pem; ssl_protocols TLSv1.2 TLSv1.3; # Disable TLS 1.0/1.1 (insecure) ssl_ciphers ECDHE-RSA-AES256-GCM-SHA512:DHE-RSA-AES256-GCM-SHA512; ssl_session_cache shared:SSL:10m; # Session resumption — skip TLS handshake on reconnect # Security headers add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always; add_header X-Frame-Options DENY; add_header X-Content-Type-Options nosniff; add_header Content-Security-Policy "default-src 'self'" always; # Rate limiting limit_req_zone $binary_remote_addr zone=api:10m rate=100r/m; limit_req zone=api burst=20 nodelay; limit_req_status 429; location /api/ { proxy_pass http://app_servers; proxy_http_version 1.1; # Required for keepalive proxy_set_header Connection ""; # Required for keepalive proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_connect_timeout 5s; proxy_send_timeout 60s; proxy_read_timeout 60s; } location /static/ { root /var/www; expires 30d; add_header Cache-Control "public, immutable"; # CDN-friendly gzip_static on; } } L4 vs L7 LOAD BALANCER: L4 (Transport Layer — TCP/UDP): Sees: IP addresses + ports only. Cannot inspect HTTP headers. Routing: pure IP/port hash. Blindly forwards TCP streams. Pros: Extremely fast (minimal processing). Low latency. Any protocol. Cons: No path-based routing. No header-based decisions. Tools: AWS NLB, HAProxy (TCP mode), MetalLB (K8s bare metal) Use for: TCP databases, MQTT, gRPC (fastest path), raw speed L7 (Application Layer — HTTP/HTTPS): Sees: full HTTP request (URL path, headers, cookies, body). Routing: /api/* → api servers, /static/* → CDN, based on JWT claims, etc. Pros: Smart routing, SSL termination, request rewriting, auth. Cons: Higher CPU (parse HTTP). Cannot handle raw TCP. Tools: Nginx, AWS ALB, Envoy, Traefik, Kong Use for: microservices routing, canary deployments, rate limiting per user

API GATEWAY REQUEST LIFECYCLE — Every Plugin, Every Step
REQUEST FLOW (Kong / Envoy gateway pattern): Client → API Gateway Phase 1: SECURITY a) TLS termination (decrypt HTTPS → internal HTTP) b) IP allowlist/denylist (block known bad actors) c) DDoS protection: rate limit by IP at connection level (L4) d) WAF (Web Application Firewall): check for SQLi, XSS, OWASP Top 10 patterns Phase 2: AUTHENTICATION e) Extract Authorization: Bearer {jwt} header f) Validate JWT signature (asymmetric RS256, public key from JWKS endpoint) g) Check expiry (exp claim), issuer (iss claim) h) If invalid → 401 Unauthorized immediately. Upstream never called. Phase 3: AUTHORIZATION i) Check scopes in JWT (e.g., scope: "read:orders write:orders") j) Check rate limit: INCR user:{userId}:rps in Redis → 429 if over limit k) Check API quota: daily/monthly call limits per API key/plan Phase 4: REQUEST TRANSFORMATION l) Add upstream headers: X-User-Id, X-User-Roles, X-Request-Id (correlation) m) Strip auth header before forwarding (microservice doesn't need JWT) n) Protocol translation: REST/HTTP → gRPC (if backend speaks gRPC only) o) Request validation: schema validation before hitting backend Phase 5: ROUTING p) Match path: /api/v2/orders/* → order-service:8080 q) Load balance across instances (least-conn / round-robin) r) Circuit breaker: if order-service returning >50% errors → open circuit Phase 6: RESPONSE s) Add CORS headers (Access-Control-Allow-Origin) t) Add response headers (X-Request-Id for tracing) u) Cache GET responses if enabled (Cache-Control honored) v) Compress response (gzip/brotli) w) Emit access log → Fluentd → Elasticsearch API VERSIONING STRATEGIES: URL path: /api/v1/users, /api/v2/users ← Most common. Easy to route. Header: API-Version: 2024-01-01 ← Stripe's approach. Cleaner URLs. Query param: /api/users?version=2 ← Breaks caching. Avoid in prod. Gateway: Route /api/v1/* → service:v1, /api/v2/* → service:v2 Run both simultaneously during migration. Sunset v1 after 6 months.

ENVOY SIDECAR + ISTIO CONTROL PLANE — Complete Architecture
WHY SERVICE MESH? (The problem it solves): Without mesh: each microservice must implement: - Retry logic (gRPC / HTTP) - Circuit breakers (Resilience4j) - mTLS (custom cert management per service) - Tracing (inject trace headers) - Metrics (expose /metrics endpoint) 500 services = 500× duplicated, inconsistent implementations. With mesh: ALL of the above handled by the sidecar. Zero application code changes. ENVOY SIDECAR ARCHITECTURE: Every pod gets an Envoy sidecar injected automatically (via MutatingWebhook): Pod: [Your Container] + [Envoy Proxy (port 15001)] iptables rules: ALL inbound/outbound traffic intercepted → Envoy App → Envoy (localhost:15001) → [policy, retry, mTLS] → network Network → Envoy → [auth, telemetry] → App (localhost:8080) App has NO idea mTLS is happening. Envoy is transparent. ISTIO CONTROL PLANE: (istiod — the "brain") Pilot: Converts Istio config (VirtualService, DestinationRule) → Envoy xDS config Citadel: Certificate Authority. Issues mTLS certs to every Envoy sidecar (SPIFFE/SVID format) Galley: Config validation and distribution xDS APIs (dynamic configuration — no restart needed): LDS (Listener Discovery): what ports Envoy should listen on RDS (Route Discovery): URL path → upstream cluster mapping CDS (Cluster Discovery): upstream service instances EDS (Endpoint Discovery): individual IP:Port of each pod mTLS — ZERO TRUST BETWEEN SERVICES: Without mTLS: order-service can call payment-service with no authentication. Any compromised pod could call payment-service and initiate transactions. With Istio mTLS (PeerAuthentication: STRICT): 1. istiod issues X.509 cert to every Envoy: "This is order-service" 2. order-service Envoy → presents cert → payment-service Envoy 3. payment-service Envoy: validates cert against istiod CA → identity verified 4. If cert invalid → connection REJECTED before app code is touched 5. AuthorizationPolicy: allow order-service → payment-service:POST/charge ONLY Commands: kubectl apply -f peer-auth-strict.yaml # Enforce mTLS globally istioctl analyze # Check for misconfigurations istioctl proxy-status # Are all Envoys synced from control plane? CANARY DEPLOYMENT — 5% traffic to new version, no code changes: VirtualService: http: - match: [{headers: {x-canary: {exact: "true"}}}] # Internal testing header route: [{destination: {host: app, subset: v2}}] - route: # Everyone else - destination: {host: app, subset: v1} weight: 95 - destination: {host: app, subset: v2} weight: 5 DestinationRule: subsets: - name: v1 labels: {version: v1} - name: v2 labels: {version: v2} Monitor: watch error rate in Grafana for v2 subset. If error rate > 1%: set v2 weight=0. Automatic rollback. If healthy after 30min: set v2 weight=100. Old v1 pods drained. CIRCUIT BREAKER IN ISTIO (DestinationRule): outlierDetection: consecutiveGatewayErrors: 5 # 5 consecutive 5xx = eject pod interval: 10s baseEjectionTime: 30s # Pod ejected for 30s then re-checked maxEjectionPercent: 50 # Never eject >50% of pods (prevent full outage) connectionPool: http: http1MaxPendingRequests: 100 # Circuit opens if >100 requests queued http2MaxRequests: 1000
🔄
Retry Policies
VirtualService retries: retries: {attempts: 3, perTryTimeout: 2s, retryOn: "gateway-error,connect-failure,retriable-4xx"}. Critical: only retry idempotent requests. Never retry POST /charge. Use idempotency keys if you must retry non-idempotent operations. Total timeout = perTryTimeout × attempts = 6s max.
🔬
Fault Injection (Chaos)
Istio can inject faults for resilience testing: fault: {delay: {percentage: 10%, fixedDelay: 5s}} — adds 5s delay to 10% of requests. fault: {abort: {percentage: 5%, httpStatus: 503}} — forces 5% failure rate. Use in staging to verify circuit breakers and retries work correctly before production.
📊
Istio Telemetry
Every Envoy emits automatically: Metrics (rps, latency p50/p95/p99, error rate per route) to Prometheus. Traces (Zipkin/Jaeger headers propagated across services). Access logs (full request/response metadata). No instrumentation code needed. Grafana: pre-built Istio dashboards.

DNS → TCP → TLS → HTTP — What Happens in the First 200ms of a Request
DNS RESOLUTION (T=0ms to T=50ms): Browser: "I need api.yourapp.com" 1. Check browser DNS cache (TTL not expired?) → HIT: skip to step 6 2. Check OS /etc/hosts file → not found 3. OS → Recursive resolver (ISP's 8.8.8.8 or local) 4. Recursive resolver → Root nameserver (13 root servers worldwide, anycast) Root: "For .com, ask Verisign's TLD servers: 192.5.6.30" 5. Recursive → TLD (.com) server → "yourapp.com? Ask ns1.yourregistrar.com" 6. Recursive → Authoritative NS (Route53/Cloudflare) → Return: "api.yourapp.com A 104.21.5.23" 7. OS returns IP to browser. Cached for TTL (60-3600s). TCP 3-WAY HANDSHAKE (T=50ms): Client → SYN (sequence=100) Server → SYN+ACK (seq=200, ack=101) Client → ACK (ack=201) Cost: 1 RTT before any data can flow. For 50ms RTT = 50ms wasted. HTTP/3 with QUIC: 0-RTT connection on repeat visits. Bypasses this entirely. TLS 1.3 HANDSHAKE (T=100ms): 1-RTT (vs TLS 1.2's 2-RTT): Client → ClientHello {supported ciphers, key_share (ephemeral public key)} Server → ServerHello {chosen cipher, server key_share, Certificate, Finished} Client → {Finished} → APPLICATION DATA 0-RTT Session Resumption: client sends request with first packet (PSK from previous session) Risk: replay attacks. Safe for: GET requests. Avoid for: POST /charge. HTTP EVOLUTION: HTTP/1.1: 1 request per TCP connection (or keep-alive with 1 concurrent request) Head-of-line blocking: request 2 waits for response 1. Workaround: 6 parallel TCP connections per browser. Inefficient. HTTP/2 (T=2015): Multiplexing: multiple concurrent requests on 1 TCP connection (streams) Binary framing: not text-based. Efficient parsing. Header compression: HPACK — duplicate headers compressed. Saves 80% header bytes. Server Push: server can send CSS/JS before browser requests it. Still TCP: TCP head-of-line blocking at packet level (one lost packet stalls all streams) HTTP/3 / QUIC (T=2022): Built on UDP (not TCP). No head-of-line blocking. Connection ID: connection migrates across IP changes (WiFi → LTE → WiFi). 0-RTT: data on first packet for returning users. Built-in TLS 1.3: encrypted by default, always. Adoption: YouTube, Gmail, Instagram all running HTTP/3. CDN CACHE LIFECYCLE: User → CDN edge PoP (nearest, via anycast DNS) Cache MISS: edge → origin server → cache response → serve to user Cache-Control: max-age=86400, s-maxage=3600 ← s-maxage for CDN, max-age for browser Cache HIT: serve from edge memory. 0 origin calls. Cache PURGE: API call to CDN → invalidate specific URLs → next request = fresh fetch Cache keys: by default = URL. Customize: Vary: Accept-Language, Accept-Encoding Versioned URLs: /static/app.v3.js — new version = new URL = instant cache bypass

OWASP TOP 10 + ZERO TRUST ARCHITECTURE — What FAANG Expects
OWASP TOP 10 — KNOW THESE DEEPLY: A1. INJECTION (SQLi, NoSQLi, Command Injection): Attack: POST /login { username: "admin'--", password: "x" } → bypasses auth Fix: ALWAYS use parameterized queries / prepared statements. NEVER string concatenation. Java: PreparedStatement. Python: cursor.execute("SELECT * FROM users WHERE id=%s", [id]) A2. BROKEN AUTHENTICATION: Attack: Weak sessions (sequential IDs), no brute-force protection, weak password hashing Fix: cryptographically random session IDs (256-bit), bcrypt/argon2 for passwords (NOT SHA256), rate limit login: 5 attempts per 15min per IP, MFA for privileged accounts A3. SENSITIVE DATA EXPOSURE: Attack: API returns full credit card number, SSN in response Fix: Never log/return sensitive data. Encrypt at rest (AES-256) + in transit (TLS 1.2+). PCI-DSS: store only last 4 digits. Tokenization: store vault token, not card data. A5. SECURITY MISCONFIGURATION: Attack: Default credentials (admin/admin), debug endpoints exposed (/actuator/env) Fix: CIS Benchmarks for OS/K8s. Disable unnecessary endpoints. Regular audits. K8s: disable anonymous auth in apiserver. Restrict /metrics to internal network. A7. XSS (Cross-Site Scripting): Attack: POST comment "<script>document.cookie</script>" → runs in other users' browsers Fix: Output encoding (htmlspecialchars()). Content-Security-Policy header. React/Vue auto-escape by default — but dangerouslySetInnerHTML bypasses it. A10. SSRF (Server-Side Request Forgery): Attack: POST /fetch { url: "http://169.254.169.254/latest/meta-data/" } → calls AWS EC2 metadata → exposes IAM credentials. Critical at cloud-native scale. Fix: Whitelist allowed domains. Block 169.254.x.x / 10.x.x.x / 172.16-31.x.x ranges. ZERO TRUST ARCHITECTURE (BeyondCorp model — Google, Cloudflare): Old model: "Inside the firewall? Trusted." Attack: Attacker breaches one internal service → free to move laterally. Zero Trust: "Never trust, always verify — regardless of network location." 5 Principles: 1. Identity-based access: every request authenticated + authorized (not IP-based) 2. Least privilege: each service can only call what it explicitly needs (Istio AuthorizationPolicy) 3. Microsegmentation: network policies restrict pod-to-pod traffic (Calico NetworkPolicy) 4. Continuous validation: re-auth frequently (short-lived tokens), anomaly detection 5. Encrypt everything: mTLS between all services (Istio), no plaintext internal traffic DDOS MITIGATION — LAYERED DEFENSE: Layer 3/4 (Volumetric — 1Tbps+): BGP blackholing: ISP drops all traffic to victim IP at network level Scrubbing centres: Cloudflare/Akamai absorb traffic, forward only clean traffic Rate limit by IP at network edge: drop >10K pps per IP at router Layer 7 (Application — Slowloris, HTTP flood): Challenge page: JavaScript/CAPTCHA challenge for suspicious requesters WAF rules: block requests matching attack patterns Rate limit: Nginx/Envoy 100 rps per IP per endpoint Geo-blocking: temporary block if attack originates from specific countries SECRETS MANAGEMENT (HashiCorp Vault pattern): NEVER: hardcode secrets in code, env files, Docker image, K8s ConfigMap ALWAYS: inject at runtime from a secrets store Vault dynamic secrets: App authenticates to Vault using K8s ServiceAccount token Vault: "Here's a temporary DB password, expires in 1 hour" App: uses credentials for 1hr, Vault auto-rotates Attack surface: no long-lived passwords to steal. Breach = 1hr window. K8s: Bitnami Sealed Secrets (encrypted in git), External Secrets Operator (pull from Vault/AWS)

PRODUCTION CI/CD PIPELINE — Every Stage, Every Gate
COMPLETE PIPELINE (git push → production in ~15 minutes): Stage 1: COMMIT (T=0s) git push → triggers CI (GitHub Actions / Jenkins / Tekton) Pre-commit hooks: lint (ESLint), format (Prettier), type check (tsc --noEmit) Stage 2: BUILD (T=0-3min) Install dependencies (npm ci — exact lockfile, cached) Run SAST (Semgrep, SonarQube) — static code analysis for vulnerabilities Run unit tests (Jest/JUnit) — must be fast (<3min). No DB calls. Build artifact (npm run build / mvn package) Build Docker image (multi-stage, cache layers) Stage 3: TEST (T=3-8min) Push image to registry (ECR/GCR) with tag: git-sha (e.g., sha-a3f4b2c) Run integration tests against test DB in Docker Compose Run DAST (OWASP ZAP) — dynamic security scan against staging endpoint Run SCA (Snyk/OWASP Dependency-Check) — scan dependencies for known CVEs Scan Docker image (Trivy) — OS + package vulnerabilities Stage 4: STAGING DEPLOY (T=8-10min) Update staging K8s manifest: image tag sha-a3f4b2c ArgoCD detects diff → syncs → rolling update in staging Run smoke tests (critical user journeys: login, checkout, API health) Run performance regression test (k6: ensure p99 latency within SLA) Stage 5: PRODUCTION DEPLOY (T=10-15min) Manual approval gate (or auto on main branch with SLO checks) Update prod K8s manifest (GitOps — PR to infra repo) ArgoCD syncs: rolling update (maxSurge=1, maxUnavailable=0) Monitor: watch error rate in Grafana for 10 minutes Automated rollback: if error_rate > 0.5% → kubectl rollout undo DEPLOYMENT STRATEGIES: Blue-Green: Two identical environments (Blue=current, Green=new) Deploy new version to Green. Run full test suite against Green. Switch DNS/load balancer: Green is now live. Blue is standby. Rollback: switch back to Blue (instant, <30s via DNS TTL or LB config) Cost: 2× infrastructure during switch. Best for: major version changes. Canary: Route 5% → new version. Monitor metrics. Gradually increase to 100%. Metric gates: error_rate < 0.1%, p99 latency < 500ms → proceed to 50% Auto-rollback: Flagger/Argo Rollouts monitors and automatically reduces weight on regression. Best for: risk mitigation on user-facing changes. Feature Flags: Deploy code but enable via flag. Decouple deployment from release. Dark launch: code in production, flag off. No users see it. Gradual: enable for 1% → 10% → 100% of users. Each behind same deploy. Kill switch: instant disable without a code deploy. 100ms propagation. DATABASE MIGRATIONS — ZERO DOWNTIME (Expand/Contract pattern): Problem: ALTER TABLE ADD COLUMN NOT NULL locks table for minutes in production. Step 1 EXPAND: Add column with DEFAULT and NULLABLE (no lock): ALTER TABLE orders ADD COLUMN status_v2 VARCHAR DEFAULT 'pending' NULL; Deploy app v2: writes to BOTH status + status_v2. Reads from status. Step 2 BACKFILL: Background job fills status_v2 for existing rows (no lock needed): UPDATE orders SET status_v2 = status WHERE status_v2 IS NULL LIMIT 1000; # Batched Step 3 SWITCH: App v3 reads from status_v2. Drop old column later: ALTER TABLE orders DROP COLUMN status; # Safe now — nothing reads it 3 separate deploys. Zero downtime. Zero locks.
🔄
GitOps (ArgoCD)
Git repo = single source of truth for cluster state. ArgoCD constantly compares desired state (git) vs actual state (K8s cluster). Drift detected → auto-sync or alert. Never kubectl apply manually in prod. All changes via PR → review → merge → ArgoCD deploys. Full audit trail via git history.
⏮️
Rollback Strategy
Instant: kubectl rollout undo deployment/app — switches to previous ReplicaSet (already pulled image, ~30s). GitOps: revert git commit → ArgoCD auto-deploys previous version. DB rollback: hardest part — use backward-compatible migrations only (never DROP in same deploy as code change).
🌿
Trunk-Based Dev
Everyone commits to main (trunk) at least daily. Short-lived feature branches (max 1 day). Feature flags gate incomplete work. No long-lived branches → no merge conflicts → continuous deployment. Google / Meta engineering model. Requires: strong automated testing, feature flags, and monitoring.

Architecture: Apps expose: GET /metrics → Prometheus text format Prometheus scrapes every 15s → stores in time-series TSDB Grafana queries Prometheus → dashboards + alerts The 4 Golden Signals (Google SRE — monitor these for EVERY service): Latency: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) Traffic: rate(http_requests_total[5m]) Errors: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) Saturation: container_memory_usage_bytes / container_spec_memory_limit_bytes Alerting rules: ALERT HighErrorRate IF rate(http_errors_total[5m]) / rate(http_requests_total[5m]) > 0.01 # >1% errors FOR 2m # Must be true for 2 minutes (avoids flapping alerts) LABELS {severity="page"} ANNOTATIONS {summary="Error rate above 1% for {{$labels.service}}"} → Alertmanager → PagerDuty (critical) / Slack (warning) PILLAR 2: LOGS (Fluentd → Elasticsearch) — "What happened?" Fluent Bit (lightweight, C-based): runs as DaemonSet on every K8s node Input: tail /var/log/containers/*.log (all container stdout/stderr) Parse: JSON structured logs (key=value pairs, not plaintext messages) Filter: add pod_name, namespace, node_name, deployment from K8s metadata Buffer: in-memory (fast) + disk (durability). Don't lose logs on restart. Output: → Fluentd aggregator (via forward protocol, compressed TCP) Fluentd aggregator: Receives from all Fluent Bit agents Routing: ERROR logs → Elasticsearch + PagerDuty webhook All logs → Elasticsearch for search (7-day retention) All logs → S3 Glacier (90-day cheap archive, compliance) Elasticsearch: indexed by timestamp + service + log_level Kibana dashboard: search across all service logs with correlation_id CRITICAL: Log format must be structured JSON: {"timestamp":"2024-04-19T12:00:00Z","level":"ERROR","service":"payment", "trace_id":"abc123","span_id":"def456","msg":"Stripe timeout","duration_ms":30012} NOT: "ERROR: Stripe timed out after 30s" ← unstructured, unsearchable PILLAR 3: DISTRIBUTED TRACING (OpenTelemetry + Jaeger) — "WHY did it happen?" Problem: user request fails after 3 seconds. Which of the 12 microservices caused it? Trace context: TraceID (128-bit, globally unique) propagated across every service call Flow: API Gateway: generate TraceID=abc123, SpanID=span1 Order Service: receives TraceID=abc123, creates child span2 calls Inventory Service: same TraceID, child span3 calls Payment Service: same TraceID, child span4 OpenTelemetry SDK: instruments every inter-service call automatically Jaeger: stores and visualizes trace trees: Trace abc123: ├── API Gateway (12ms) └── Order Service (2988ms) ← SLOW ├── Inventory Service (15ms) └── Payment Service (2970ms) ← ROOT CAUSE └── Stripe timeout after 30s (HTTP 504) With TraceID: 1 Jaeger search → pinpoint root cause across 12 services. Without tracing: 12 service log searches, manually correlating timestamps. SLO / SLA / SLI FRAMEWORK: SLI (indicator): actual measurement = 99.87% of requests <200ms in last 30 days SLO (objective): target = 99.9% of requests <200ms (error budget = 0.1% = 43min/month) SLA (agreement): contractual = 99.5% uptime (with financial penalties) Error budget: Current month: used 0.03% of 0.1% budget = 70% remaining = safe to deploy
🎯
Metrics vs Logs vs Traces
Metrics: aggregated numbers over time. "Error rate is 2%." Fast, cheap, no context. Logs: discrete events with full context. "User 123 got error X at 12:04:03." Expensive at scale. Traces: causal chain across services. "Request took 3s because Stripe timed out." Use all 3 together — metrics to detect, traces to diagnose, logs for forensics.
📊
RED vs USE Method
RED (for services): Rate, Errors, Duration. What interviewers expect for API metrics. USE (for resources): Utilization, Saturation, Errors. For CPU, memory, disk, network. Google SRE: 4 Golden Signals (Latency, Traffic, Errors, Saturation) = combination of both.
🚨
Alerting Best Practices
Alert on symptoms, not causes. Bad: "CPU > 90%" (cause). Good: "error rate > 1%" (symptom). Add FOR: 2m to prevent false positives from transient spikes. Multi-window: fast burn (1h window, 5% error budget consumed) = page. Slow burn (6h window, 10% consumed) = ticket. Runbook URL in every alert.

10 CRITICAL SCENARIOS — Diagnosis + Root Cause + Prevention
SCENARIO 1: OOMKilled — pod keeps restarting kubectl describe pod app-7d9f → "OOMKilled: container exceeded memory limit" Root cause: memory.limits.requests too low, OR memory leak in app Fix A: increase memory.limits (temporary). Fix B: fix memory leak (permanent) Prevention: monitor container_memory_usage_bytes vs limit. Alert at 80%. Java: set -Xmx to 75% of container limit. Without -Xmx, JVM doesn't know it's in a container. SCENARIO 2: CrashLoopBackoff — pod never starts kubectl logs app-7d9f --previous # Logs from crashed container Common causes: bad env var (DB connection string), missing secret, port conflict, required service unreachable, bad health check threshold (liveness probe too aggressive) Liveness probe killing healthy pod: if app takes 60s to start but liveness checks after 10s → kills+restarts forever Fix: initialDelaySeconds: 60 on liveness probe. Or: use startupProbe instead. SCENARIO 3: THUNDERING HERD — cache miss storm Redis cluster reboots. All cache keys gone. 50K requests/sec all hit DB simultaneously. DB can handle 5K req/sec → queue builds → timeouts → 502s everywhere. Prevention: probabilistic early expiration (recalculate cache before TTL expires, randomly) Prevention: mutex locking (only 1 thread recalculates, others wait with fallback TTL) Prevention: never expire all keys simultaneously (add random jitter to TTL: base + rand(0, 300)) SCENARIO 4: CASCADING FAILURE — one slow service takes down everything Inventory service becomes slow (DB query unindexed → 30s response time) Order service calls inventory → waits 30s → thread pool exhausted Order service now slow → API gateway timeout → 502 for all users Root cause: no timeout + no circuit breaker between order and inventory Fix: timeout: 2s on inventory call. Circuit breaker: open after 5 failures. Fix: bulkhead: separate thread pool for each downstream dependency. Fix: fail-fast: return cached/stale inventory data rather than waiting. SCENARIO 5: K8s NODE NOT READY — pods evicted unexpectedly Node: disk pressure OR memory pressure → kubelet evicts pods (no warning) Low-priority pods evicted first. If no priority classes set → any pod evicted. Prevention: PriorityClass for critical services. DiskPressure: clean up images (crictl rmi --prune) Node goes NotReady → pod controller reschedules on other nodes (5 min default. Configure: node.kubernetes.io/not-ready:NoExecute tolerationSeconds: 60) SCENARIO 6: etcd DATA CORRUPTION — cluster brain dead etcd split-brain or disk failure. apiserver can't read cluster state. ALL kubectl commands fail. You can't deploy, scale, or even describe pods. Running pods continue (kubelet caches pod specs locally). No changes possible. Recovery: restore from etcd snapshot (taken every 30 min via CronJob ideally) Prevention: etcd on fast SSD (low latency critical for Raft consensus), etcd defrag regularly, monitor etcd_disk_wal_fsync_duration_seconds SCENARIO 7: IMAGE PULL ERROR — deployment stuck pending kubectl describe pod → "ErrImagePull: ImagePullBackOff" Causes: image tag doesn't exist (typo), private registry auth expired (imagePullSecret) Fix: kubectl create secret docker-registry regcred --docker-server=... Best practice: never use :latest tag in K8s. Pin to immutable SHA256 digest. image: myapp@sha256:a3f4b2c... ← immutable, reproducible, no surprises SCENARIO 8: NETWORK POLICY MISCONFIGURATION — service can't talk to DB NetworkPolicy is deny-all → new pod can't reach PostgreSQL All K8s pod-to-pod traffic allowed by default. Once ANY NetworkPolicy applied: default deny. Debug: kubectl exec -it app-pod -- nc -zv postgres-svc 5432 Fix: add explicit ingress rule allowing app pods to reach postgres on port 5432 Tools: Cilium Network Policy Editor, netshoot debug container SCENARIO 9: FLUENTD BACKPRESSURE — logs dropped silently Application logging 100K lines/sec. Fluentd processes 50K/sec. Buffer fills. Default: drop oldest logs when buffer full. You lose production logs during incident. Prevention: Fluent Bit memory_limit 512M + disk buffer fallback Tune: flush_interval 1s → 5s (reduce I/O pressure). chunk_limit_size 1M. Monitor: fluentd_output_status_buffer_queue_length → alert at 80% capacity SCENARIO 10: CERT EXPIRY — HTTPS breaks at 3 AM TLS cert expired. Users see "Your connection is not private." Revenue drops to 0. Prevention: cert-manager in K8s (auto-renews Let's Encrypt certs 30 days before expiry) Monitor: ssl_certificate_expiry_seconds → alert when <30 days remaining Wildcard certs: AWS Certificate Manager auto-renews and rotates ALB certs transparently.

⚡ The Command Cheatsheet

Commands every senior engineer must run fluently during incidents and interviews.

kubectl debug
kubectl exec -it pod -- /bin/sh — shell into running container. kubectl debug pod -it --image=nicolaka/netshoot — debug with network tools. kubectl logs pod --previous --tail=100 — last logs before crash.

docker layer

istioctl

nginx -t / reload
