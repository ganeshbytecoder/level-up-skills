Prologue

Dev's Last Night on Call

Dev had been on call for the River Network for six years. He was leaving at the end of the month — a new position, a new city. This was his last rotation.

The vaccination campaign had been announced three weeks ago. Physician counsels across all seventeen kingdoms had agreed: a new epidemic was moving from the east. The vaccination program needed to reach 500,000 citizens within 90 days. The River was the only system capable of coordinating it — appointment booking, clinic notification, confirmation delivery, inventory management, rescheduling.

Dev had set a reminder: Check the River before you sleep.

At 11:47 p.m., he looked at his monitoring dashboard. The consumer lag counter was at 0. Every topic was being consumed. Every service was responding. He began to close his laptop.

At 11:52 p.m., the lag counter for vaccination-appointments jumped to 847,000.

His phone showed a message from Priya, who was monitoring the citizen-facing confirmation service: "Citizens not receiving confirmations. First appointment hour starts in 6 hours."

Dev put his laptop back on his desk. He would not close it again until 6:18 a.m.

What he discovered that night was a complete taxonomy of everything that can go wrong in an event-driven system. He wrote it all down. This is that document.

All systems restored. First appointment wave begins on time. Dev sends his final on-call handover — the most thorough document the team had ever received.

Act I
Delivery & Messaging Problems
1. At-Least-Once Delivery — The Duplicate Event
Kafka guarantees at-least-once delivery. The same vaccination appointment event was processed twice — patient booked into two slots simultaneously. Consumer crashed after processing but before committing the offset — Kafka redelivered.
✅ Fix: Every consumer must be idempotent. Use an idempotency store (Redis or DB). Before processing: check if messageId was already processed. If yes: skip and ack. If no: process and record.
2. Out-of-Order Message Processing
Appointment CANCELLED event arrived before the CONFIRMED event for the same appointment. The clinic's state machine had no confirmed appointment to cancel — it raised an error and committed the wrong state.
✅ Fix: Use partition keys to guarantee ordering for related events: all events for appointmentId → same partition. Handle out-of-order gracefully: store events, process ordered sequence.
3. At-Most-Once Delivery — The Silent Drop
A producer was configured to not wait for acknowledgment (acks=0). During a brief broker hiccup, 340 appointment events were silently dropped — never stored, never delivered.
✅ Fix: For critical data, use acks=all (producer waits for all replicas to confirm). Use Kafka's idempotent producer to prevent duplicates while ensuring delivery.
4. Message Reordering Across Partitions
Events for the same clinic were spread across multiple partitions through a bug in the partition key logic. Events for clinic #77 arrived on partitions 2 and 5 — processed by different consumers in different orders.
✅ Fix: Consistent partition key selection — all events for the same logical group MUST use the same key. Test partition key distribution before production.
5. Event Loss Due to Log Compaction Misconfiguration
A topic was configured with log compaction but with a retention period shorter than the consumer's lag. By the time the slow consumer reached an event — the log compaction had deleted it. The event was gone forever.
✅ Fix: Retention period ≥ maximum expected consumer lag + safety margin. Alert when consumer lag approaches the retention boundary. Never assume an event will be there when the consumer gets to it "eventually."
6. Duplicate Event from Retry Without Idempotency Key
The booking service timed out and retried 3 times. The first attempt had actually succeeded. The second and third retries created duplicate bookings. No idempotency key was used on the producer side.
✅ Fix: Generate idempotencyKey on the client side before first attempt. Include it in every retry. Use Kafka's idempotent producer or check at the consumer with a deduplication store.

Act II
Consumer Problems
7. Consumer Lag — The Accumulating Debt
The Confirmation Service was processing 100 events/second. The producer was publishing 150 events/second. Consumer lag grew by 50 events per second. After 3 hours: 540,000 events behind. Confirmations were 90 minutes late — arriving after patient appointments had already passed.
✅ Fix: Alert on consumer lag, not just on consumer errors. Add more consumer instances (up to partition count). Profile slow processing: where is the bottleneck? Database? Downstream API? Serialize/deserialize?
8. Poison Pill Message — The Consumer Crash Loop
One malformed appointment event (null clinicId) caused the consumer's deserialization to throw an exception. The consumer crashed. Kafka redelivered the message on restart. The consumer crashed again. Loop. No other messages were processed.
✅ Fix: Dead Letter Queue (DLQ). After N failed attempts: route to DLQ instead of crashing. Consumer continues with next message. Operations team investigates DLQ. Log and alert on every DLQ write.
9. Slow Consumer Blocking the Entire Partition
One consumer was calling an external geocoding API synchronously per event. The geocoding API was slow (800ms per call). The consumer fell behind, accumulating lag, while three other consumers in the same group processed their partitions at full speed — unable to help.
✅ Fix: Increase partition count to distribute load. Batch API calls: accumulate N events, call API once. Use async processing: publish event to internal queue, process asynchronously. Never synchronously call slow external services inside a Kafka consumer loop.
10. Consumer Rebalancing Storm
A consumer crashed and restarted repeatedly (due to the geocoding API issue). Each crash triggered a consumer group rebalance. During rebalance, NO messages from the group were processed. Lasted 2 minutes per crash.
✅ Fix: Fix the underlying crash cause (that's the primary fix). Increase session.timeout.ms and heartbeat.interval.ms to prevent spurious rebalances. Static group membership (group.instance.id) to reduce rebalance frequency.
11. Committing Offset Before Processing Is Complete
The consumer committed the offset immediately on receiving the event, then processed it asynchronously. The processing failed. The event's offset was already committed — Kafka moved on. The failed appointment was never retried. 200 citizens received no confirmation.
✅ Fix: Commit offset AFTER processing is verified complete. Use manual offset commit. Enable auto.offset.reset=earliest for recovery. Never commit offsets before business logic is durably persisted.
12. Heartbeat Timeout Causing False Consumer Failure
A consumer was doing expensive processing (database batch insert) that took longer than session.timeout.ms. Kafka believed the consumer was dead and triggered a rebalance. The consumer then came back and re-processed the same messages.
✅ Fix: Set session.timeout.ms and max.poll.interval.ms based on the actual maximum processing time of one poll batch. Reduce the batch size (max.poll.records) if processing is slow.
13. Multiple Consumer Groups Interfering
Two teams deployed consumers for the same topic under the same consumer group ID — by accident. Kafka split the partitions between both services. Each service received only half the events. Both processed incorrect, incomplete state.
✅ Fix: Use unique, descriptive consumer group IDs. Convention: {service-name}-{topic-name}-consumer. Audit active consumer groups before deploying new consumers.

Act III
Schema & Contract Problems
14. Breaking Schema Change — Adding Required Field
The booking team added a required field (vaccinationType) to the appointment event schema. Old consumers parsing old messages failed — the field was missing. Partitions with old messages were now unprocessable.
✅ Fix: Schema evolution rules: NEVER add required fields. New fields must have defaults (optional + nullable). Old consumers ignore unknown fields. New consumers handle missing optional fields with defaults. Use Schema Registry (Confluent) to enforce compatibility rules.
15. Changing Field Type — The Silent Data Corruption
The appointmentId was changed from Integer to UUID (String) without a schema version bump. Consumers parsed the event, treating the UUID string as an integer. Data was silently corrupted — appointments were being associated with the wrong records.
✅ Fix: Never change the type of an existing field. Add a new field with the new type alongside the old one. Deprecate the old field with a migration period. Schema Registry prevents this if configured to enforce full compatibility.
16. Missing Schema Version — No Contract Enforcement
Events had no schemaVersion field. When the schema changed, there was no way to tell which version an event was. Old and new events were indistinguishable in the topic. Consumers had to handle all possibilities with complex conditional logic.
✅ Fix: Include schemaVersion in every event. Use Schema Registry with schema versioning. Consumers use the version to select the correct deserializer.
17. Producer and Consumer Out of Sync Schema
A new producer deployed with v2 schema before any consumers were updated to handle v2. 4 hours of v2 events were in the topic that no consumer could process. When consumers were updated, they processed v2 events correctly — but also tried to reprocess the already-processed v1 events because offsets were reset.
✅ Fix: Consumer-first deployment when adding fields to an existing schema. Update consumers to handle new fields (with defaults) before updating producers to send them. Schema Registry compatibility checks enforced in CI/CD.
18. Implicit Contract — Undocumented Event Assumptions
One team's consumer assumed event timestamps were always in UTC. The producer's new version sent timestamps in the local timezone of the datacenter (CET). Appointment times were offset by 1 hour. Six hundred appointments were scheduled for the wrong time.
✅ Fix: Document every field's format, unit, and timezone explicitly. Include format in the schema comment. Test with data at timezone boundaries. All timestamps: ISO 8601 UTC always.

Act IV
Saga & Distributed Transaction Problems

The vaccination appointment booking was a distributed transaction: Book appointment → Decrement clinic inventory → Send citizen confirmation → Send clinic notification. All four or none.

Dev had assumed Sagas were configured. They were not — not correctly.

19. Missing Saga — Partial Completion Without Rollback
Appointment booked. Inventory decremented. Confirmation sent. Clinic notification FAILED. The patient had a booking. The clinic had confirmed inventory committed. But the clinic was never notified. The patient arrived. The clinic had no record.
✅ Fix: Saga Pattern with compensating transactions. If step 4 (clinic notification) fails after retries: compensate by un-decrementing inventory, cancelling the booking, and notifying the citizen that their booking failed. Never leave the system in a partially successful state that appears successful to the user.
20. Compensating Transaction Also Failing
After clinic notification failed, the Saga triggered a compensation (cancel booking). The cancellation service was also experiencing issues. Both the forward flow and the compensation failed. The appointment existed in the system but was in an unknown state.
✅ Fix: Compensating transactions must be ultra-simple and ultra-reliable. Separate compensation events from forward-flow events. Design compensation to be retried indefinitely until success. Use a saga orchestrator (Temporal, AWS Step Functions) that handles compensation retry logic.
21. No Saga Timeout — The Eternal Pending State
A saga was waiting for an inventory reservation confirmation. The inventory service was slow. The saga had no timeout — it waited forever. After 6 hours, the inventory service responded. The clinic's time slot was now in the past. The saga confirmed a booking for a slot that had already occurred.
✅ Fix: Every saga step must have a timeout. Timeout = maximum wait → trigger compensation → abort and notify user. Never allow a saga to wait indefinitely for any single step.
22. Choreography Without Visibility — "Who Is Responsible?"
The saga was implemented as choreography: each service reacted to the previous service's event. When an appointment got stuck in a "pending" state, no one could determine which service was supposed to have acted next — or why it hadn't. Debugging required examining events across 4 topics in chronological order manually.
✅ Fix: Choreography sagas need correlation IDs propagated through every event. Use saga state tracking service that records each step's completion. For complex sagas: orchestration (central coordinator) provides clearer visibility into state.
23. Outbox Table Not Cleaned Up
The Outbox Pattern was implemented — events written atomically with DB records. The outbox relay published events and marked them as published. But the cleanup job (delete published rows) was never implemented. After 3 months, the outbox table had 40 million rows. Queries on it were taking 8 seconds.
✅ Fix: Outbox cleanup is a first-class concern. Delete or archive published rows with a background job. Run it continuously, not in large batches. Monitor outbox table size as an operational metric.

Act V
Observability & Operations Problems
24. Missing Correlation ID — "Which Confirmation Belongs to Which Appointment?"
Dev received a complaint: "My appointment confirmation says 10 AM but my slot is 2 PM." He needed to trace the event from booking → inventory → confirmation. Each event had different IDs with no common correlation ID. Manual cross-referencing of three topics took 45 minutes.
✅ Fix: Every event must carry the same correlationId from the originating action. The booking service generates correlationId. Every downstream event inherits it. Every log line includes it. Distributed traces are grouped by it. One ID traces the entire journey.
25. No Consumer Lag Monitoring
The consumer lag had been growing for 3 hours before Dev noticed it. The alert was configured on: "consumer is down" (binary). No alert was configured on: "consumer lag > 10,000 messages." By the time the alert fired, lag was at 847,000.
✅ Fix: Three-tier consumer lag alerting: Warning (lag > 1,000 = growing), Error (lag > 10,000 = investigate now), Critical (lag > 100,000 = incident). Alert on lag growth rate, not just absolute value.
26. No Event Replay Capability
A consumer bug caused 2 hours of incorrect processing. To fix it: replay the affected time range. But the consumer group's offsets had been committed past those messages. Resetting to the beginning would re-process 3 months of history.
✅ Fix: Keep Kafka retention long enough to replay any recent window (minimum 7 days for critical topics). Use separate consumer groups for replay so production offsets are never disturbed. Documented replay procedure with tested runbook.
27. Logging Event Payload Without Considering PII
Dev added detailed event logging to debug the appointment issue — including citizen name, date of birth, and health condition. These logs were shipped to the central logging system, accessible to all engineers. A compliance audit later identified a GDPR violation.
✅ Fix: Log event metadata, not payload content. Log: eventId, correlationId, topic, partition, offset, timestamp, processingDuration. Never log PII. Structured log format — machine-readable, not human narrative.
28. Alert Fatigue — 400 Alerts, No Signal
The vaccination system had 400 alert rules. By the time the crisis was in full swing, Dev's alert channel had 847 unread messages. The critical alert — "consumer lag critical: 847,000" — was buried on page 12 of the unfilterable stream.
✅ Fix: Alert on symptoms (user impact), not causes. Age-old rule: each alert must be actionable. Remove or silence alerts that cannot be acted on. Use severity tiers with separate channels. For on-call: 3 critical alerts maximum. Everything else: reviewed daily.

Act VI
Infrastructure & Configuration Problems
29. Too Few Partitions — Throughput Ceiling
The vaccination-appointments topic had 3 partitions. Maximum parallelism: 3 consumers. At peak: 450 events/second arriving. Three consumers at maximum speed could process 300/second. The topic could not be processed fast enough regardless of how many consumers were added.
✅ Fix: Partition count determines maximum consumer parallelism. Set partitions based on peak expected throughput / single consumer throughput. Note: partitions cannot be reduced once set. Plan for peak traffic × 2 when creating topics.
30. Kafka Broker Disk Full — Topic Writes Failing Silently
A Kafka broker's disk filled overnight. New writes silently failed. Producers received no error (acks=0 misconfiguration). 6 hours of vaccination appointment events were lost before the issue was discovered.
✅ Fix: Monitor Kafka broker disk utilization. Alert at 75% (warning) and 85% (critical). Retention policy per topic based on volume. acks=all ensures producers know about write failures. Never configure acks=0 for mission-critical topics.
31. Thundering Herd After Restart
The vaccination system was restarted after the crisis. All 40 consumer instances reconnected simultaneously. Each immediately began processing its partition's backlog at maximum speed. The downstream Confirmation Service received 450× its normal write rate. It crashed.
✅ Fix: Rate-limit consumers during catch-up mode. Add artificial processing delay when lag > threshold. Restart consumers in staggered waves. Protect downstream with backpressure—consumer checks downstream health before processing batch.
32. Consumer Deployed With Wrong Kafka Cluster Config
A consumer was accidentally deployed pointing to the staging Kafka cluster instead of production. It was happily consuming and committing offsets — on staging events. Production events accumulated unprocessed. No monitoring detected the mismatch.
✅ Fix: Validate Kafka cluster endpoint against expected production domain on startup. Log bootstrap.servers on every startup. Alert on unexpected consumer appeared on staging cluster consuming "production-named" topics.
33. Unmonitored Dead Letter Queue
The DLQ was correctly configured. Messages that failed were routed there. But no one was monitoring the DLQ. For 3 weeks, 840 failed appointment events accumulated — 840 citizens who never received confirmations. None were processed. Some had already missed their appointments.
✅ Fix: DLQ monitoring is mandatory. Alert immediately on any DLQ write. Weekly DLQ report summarizing count and category of failures. DLQ messages must be reviewed and resolved — not just stored and forgotten.
34. Back-Pressure Not Implemented — Consumer OOM
The consumer received events faster than it could process them. It bufferred them in memory. The buffer kept growing. After 40 minutes, the consumer ran out of memory and was OOM-killed — losing the buffered events and triggering a rebalance.
✅ Fix: Use Kafka's built-in flow control: max.poll.records limits how many events are fetched per poll. Process the batch before polling again. Never create unbounded in-memory queues that hold events fetched from Kafka.
35. Global Consumer Group Offset Reset Error
An engineer accidentally reset consumer group offsets to the earliest position across all topics — intending to reset only one topic. Every consumer began re-processing months of events. Idempotency saves partially; the volume caused a cascade.
✅ Fix: Offset reset commands must specify exact partition and offset. Multi-confirm prompts. Runbook requiring peer review for offset resets. Test offset reset procedures on a copy of the topic.
36. Event Containing a Reference, Not the Data
Events contained only appointmentId. Consumers fetched appointment details from the API on each event. During peak load, the appointment API was slow — adding 800ms to each event's processing. Consumer lag grew.
✅ Fix: Include necessary data in the event itself (event-carried state transfer). Don't design events that require a synchronous API call to process. This is called "thin events with API calls" vs "fat events" — for high throughput, prefer fat events for critical data.
37. Missing Idempotency at the Database Level
The consumer was idempotent in memory (checked a Redis cache). But the Redis cache had a TTL of 1 hour. An event processed at 8 AM and received again at 9:30 AM (due to a weird Kafka replay) was processed again — outside the Redis TTL window.
✅ Fix: Idempotency keys in the database (not cache) for events that must never be duplicated. Database has a UNIQUE constraint on (messageId). Duplicate inserts fail with a constraint error, not silently duplicate.
38. Asynchronous Failure Misinterpreted as Success
The booking API returned HTTP 202 Accepted (event published to Kafka). The mobile client treated 202 as "booking confirmed." The booking event was later rejected by the consumer due to a business rule. The client never knew. The citizen arrived for an appointment that was never confirmed.
✅ Fix: For high-stakes operations: use polling or webhooks to deliver final status. Or: synchronous processing for the critical path, async for non-critical downstream. 202 must be communicated as "pending" to users, not "done."
39. Clock Skew Between Producers
Two producers published events for the same appointment. Producer A's clock was 47 seconds ahead. Events were ordered by timestamp. The UPDATED event from Producer B (real-world time: later) appeared before the CREATED event from Producer A (timestamp: 47s ahead but real-world: earlier).
✅ Fix: Never rely on producer timestamps for event ordering. Use Kafka's broker-assigned timestamps (logAppendTime), which are consistent across partitions. Use sequence numbers or vector clocks for ordering sensitive workflows.
40. Fan-Out Storm Without Rate Control
A single "Mass Rescheduling" event triggered the system to fire 500,000 individual appointment update events simultaneously. The notification service received 500,000 events in 30 seconds — 20× its designed capacity. SMS gateway was throttled. Confirmations delayed by 6 hours.
✅ Fix: Rate-limited fan-out. Publish batch events, not individual events for mass operations. Consumer unpacks batch and processes at controlled rate. Use flow control to limit publishing rate to downstream capacity.
41. No Topic Retention Policy — Events Deleted Too Early
Default Kafka retention (7 days) was used. A consumer that had been down for 8 days (during a deployment freeze) reconnected. Its earliest unprocessed event was gone — deleted by retention. It had to restart from the earliest available event, which was after the deletion point.
✅ Fix: Set retention per topic based on maximum expected consumer downtime + margin. For critical topics: 30 days. Monitor: alert when consumer lag approaches the retention boundary before events are deleted.
42. Event Payload Too Large — Broker Rejection
A producer included a full vaccination record PDF (2MB) as a base64 field in the appointment event. Kafka's default message size limit (1MB) caused the event to be rejected. The producer returned an error. The event was lost. The developer didn't handle the RecordTooLargeException.
✅ Fix: Store large payloads in blob storage (S3). Include a reference URL in the event. Kafka events: small, fast, reference-based. Handle RecordTooLargeException explicitly — don't let it silently swallow events.
43. Concurrent Saga Instances for Same Entity
Two concurrent booking requests for the same clinic slot started two Saga instances. Both passed inventory check (availability: 1 slot). Both consumed the slot. The clinic was double-booked. Two citizens arrived for one slot.
✅ Fix: Saga per entity must be mutex. Distributed lock on the entity ID before starting the Saga. Or: use database optimistic locking at the inventory step — second Saga fails the inventory check and rolls back.
44. Event-Sourced Read Model Not Updated
The appointment status read model (for the mobile API's "my appointments" view) was populated from events. A consumer that updated the read model crashed for 30 minutes. Users saw stale appointment data — confirmed appointments showing as "pending."
✅ Fix: Read model consumers need the same reliability as any critical consumer: DLQ, alerting, monitored lag. Design read models to tolerate brief staleness with a "last updated" timestamp. Show "data as of 2 minutes ago" rather than wrong data.
45. Kafka Topic Permissions Misconfigured
The Clinic Notification Service lost ACL permissions to publish to the clinic-notifications topic after a security configuration update. It published without error to an internal buffer — for 2 hours, before the buffer filled and the service started returning 500 errors. No clinic received notifications for 2 hours.
✅ Fix: Test ACL permissions in CI/CD pipeline. Alert on failed publish attempts (authorization errors in producer logs). Monitor publish rate by service — drop to zero is an immediate alert.
46. No Multi-Region Failover for Kafka
The Kafka cluster was single-region. During a brief AWS us-east-1 issue, the entire vaccination system stopped for 40 minutes. 12,000 appointment events were lost (acks=0 on some producers). No multi-region replication had been configured.
✅ Fix: For mission-critical event backbones: use multi-region Kafka (MirrorMaker 2, Confluent Replication). Active-active or active-passive. Understand RPO (how much data can be lost) and RTO (how quickly can it recover). Vaccination system is mission-critical — equivalent to a payment system for infrastructure design.
Epilogue

Dev's Final Words — The Six Questions

At 6:18 AM, Dev closed his last incident ticket. The vaccination system was stable. The first appointments were beginning. He opened a new document and wrote:

"I will not be on call next month. The next person who sits in front of this monitoring dashboard will face every one of these problems again — not because they are bad engineers, but because event-driven systems have their own unique failure language. I want to give the next person a shortcut."

⚡ The Six Questions — Before Any Event-Driven Feature Ships

1
What happens if this message is delivered twice?
Kafka guarantees at-least-once. Duplicate processing = duplicate charges, duplicate bookings, duplicate emails. Your consumer MUST be idempotent.
2
What happens if it arrives out of order?
Network partitions, rebalances, and multiple partitions can reorder events. If your state machine requires order — use partition keys + handle out-of-order gracefully.
3
What happens if the consumer crashes halfway through processing?
Offset was not committed. Message will be redelivered. Processing must be atomic — either fully complete or not at all. Never commit offset before persistence is confirmed.
4
What happens if the downstream is slow — not down, just slow?
Slow downstream = consumer lag grows. More consumers cannot help if partition count is the limit. Detect lag early. Alert before it becomes a crisis.
5
What happens if this event is missing entirely?
Producer crash before commit, disk full, wrong ACLs, retention expired. Can your system detect a missing event? Can it recover without it?
6
What happens if I need to replay this topic for recovery?
Your consumer must be idempotent (answer 1). Your retention must be long enough. Your consumer group offset reset must be documented. Your replay must not kill downstream.

Dev printed this document, laminated it, and left it taped to the wall above the on-call workstation.

The next engineer to sit down read it on their first night and made one small change at the bottom: they added their own initials, the date, and the number of times it had saved them.

By year three, there were forty-two sets of initials.

The vaccination system has run without a major incident since that night.

500,000 citizens were vaccinated. On time. Without loss.

Because one engineer, on one last night on call, wrote everything down.
