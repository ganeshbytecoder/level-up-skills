POSTGRESQL INTERNALS — MVCC, WAL, VACUUM & INDEX TYPES
MVCC — MULTI-VERSION CONCURRENCY CONTROL: Core idea: every row has hidden system columns: xmin: Transaction ID (XID) that CREATED this row version xmax: Transaction ID that DELETED/UPDATED this row version (0 = still live) Scenario: T1 started at XID=100. T2 updates a row at XID=105. Before update: row_v1 { name:'Alice', xmin:50, xmax:0 } ← visible to T1 (xmin=50 < 100) After update: row_v1 { name:'Alice', xmin:50, xmax:105 } ← T1 still sees this (xmax=105 > 100) row_v2 { name:'Bob', xmin:105, xmax:0 } ← T1 can't see (xmin=105 > 100) T1 sees: 'Alice' (old version). T2 sees: 'Bob' (new version). NO LOCKS on reads. Readers never block writers. Writers never block readers. But: dead row versions (xmax set) accumulate over time → BLOAT → need VACUUM. WAL — WRITE-AHEAD LOG (Durability + Replication): Every change written to WAL FIRST before touching data pages. Crash recovery: replay WAL from last checkpoint → consistent state. Zero data loss. Streaming replication: standby continuously reads WAL from primary → applies → stays in sync. WAL flow: INSERT → WAL record written → wal_writer flushes to disk (fsync) → heap page updated (buffered) → checkpoint periodically syncs heap to disk Synchronous replication (synchronous_commit=on): Primary: wait for standby to confirm WAL written → then return success to client RPO=0 (zero data loss). RTO: standby promoted in <30s. Latency: +RTT to standby. Asynchronous (synchronous_commit=off): Primary: return success immediately. Standby lags up to wal_receiver_status_interval. Risk: crash + failover = up to ~500ms of data loss. INDEX TYPES — CHOOSE THE RIGHT TOOL: B-Tree (default): Range queries, equality, ORDER BY, LIKE 'prefix%' CREATE INDEX ON orders(created_at); ← range: WHERE created_at > '2024-01-01' CREATE INDEX ON orders(user_id, status); ← composite: WHERE user_id=? AND status=? Rule: leftmost column of composite MUST appear in WHERE for index to be used. GIN (Generalized Inverted Index): Array membership, JSONB @, full-text search CREATE INDEX ON products USING GIN(tags); ← WHERE tags @> '{electronics}' CREATE INDEX ON articles USING GIN(to_tsvector('english', body)); ← full-text GiST: Geometric types, PostGIS (geography), nearest-neighbour queries CREATE INDEX ON locations USING GIST(coordinates); ← WHERE ST_DWithin(coord, point, 1000) BRIN (Block Range Index): Very large append-only tables with natural order CREATE INDEX ON events USING BRIN(event_time); ← 1000× smaller than B-tree Works when data is physically ordered (e.g., timestamps of INSERT order matches value order) NOT useful for random access. Scans block ranges, much coarser than B-tree. Partial Index (major performance win): CREATE INDEX ON orders(user_id) WHERE status = 'PENDING'; ← only pending orders indexed 10M orders, only 50K pending → index is 200× smaller. Queries on pending: blazing fast. Expression Index: CREATE INDEX ON users(LOWER(email)); ← case-insensitive email lookup WHERE LOWER(email) = LOWER($1) → uses index. WHERE email = $1 → does NOT use this index (different expression). VACUUM — PREVENTING XID WRAPAROUND (The silent killer): XIDs are 32-bit integers: 4 billion max. When XID=4B → wraps to 0. Postgres: any row with XID > current_xid + 2B = "in the future" → invisible! VACUUM prevents this by FREEZING old rows (marking as "visible to all future transactions"). autovacuum_freeze_max_age = 200M (transactions). Must VACUUM within 200M XID. Warning signs: SELECT * FROM pg_stat_user_tables WHERE n_dead_tup > 1000000; Emergency: pg_database.datfrozenxid approaching current_xid by <50M = URGENT. PostgreSQL will refuse new connections and initiate emergency autovacuum to prevent wraparound. TABLE PARTITIONING (Postgres 10+ declarative): CREATE TABLE orders (id BIGINT, created_at DATE, amount NUMERIC) PARTITION BY RANGE (created_at); CREATE TABLE orders_2024_q1 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2024-04-01'); CREATE TABLE orders_2024_q2 PARTITION OF orders FOR VALUES FROM ('2024-04-01') TO ('2024-07-01'); Query: SELECT * FROM orders WHERE created_at = '2024-04-15' Planner: partition pruning → only scans orders_2024_q2. Other partitions skipped. Benefits: partition detach/drop = instant (vs DELETE which vacuums). Indexes per partition. Pitfall: queries WITHOUT partition key = FULL scan all partitions. Always include partition key.

Isolation Levels
Read Committed (default): each statement sees latest committed data. Non-repeatable reads possible. Repeatable Read: snapshot at tx start. No phantom reads in Postgres (unlike SQL standard). Serializable (SSI): true serialization via predicate locks. Highest safety, some abort+retry overhead. Use for financial transfers.
🏊
PgBouncer — Connection Pooling
PostgreSQL: each connection = OS process (~5MB RAM). 1000 connections = 5GB RAM just for processes. PgBouncer pools: Transaction mode (best): connection returned to pool after each transaction. Supports 10K clients on 100 DB connections. Session mode: connection held until client disconnects. Only 2-3× multiplexing benefit.
📊
EXPLAIN ANALYZE
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON). Key metrics: Seq Scan = missing index. actual rows ≫ estimated rows = stale statistics (run ANALYZE). Nested Loop on large tables = should be Hash Join. Buffers: shared hit = from cache (fast). Buffers: shared read = disk I/O (slow).
🔑
Deadlocks
T1 locks row A, wants row B. T2 locks row B, wants row A. Cycle = deadlock. Postgres detects after deadlock_timeout (1s default) → aborts one transaction (ERROR 40P01). Prevention: always acquire locks in consistent order (e.g., always by id ASC). Use SELECT ... FOR UPDATE NOWAIT to fail fast instead of waiting.

INNODB ARCHITECTURE — Clustered Index, Buffer Pool & Gap Locks
CLUSTERED INDEX — InnoDB's Most Important Design Decision: InnoDB: data rows stored PHYSICALLY sorted by PRIMARY KEY in a B+ tree. The primary key IS the table. Secondary indexes store PK values as "row pointers." GOOD PK: INT AUTO_INCREMENT or BIGINT. Inserts always append to end. No page splits. BAD PK: UUID (random). Inserts anywhere in tree → page splits → fragmentation → 30-50% wasted space. 50M UUID-keyed rows: table often 2× larger than AUTO_INCREMENT equivalent. Secondary index lookup: SELECT * FROM orders WHERE status = 'PENDING' (secondary index on status) 1. Scan status index → get list of primary keys 2. For EACH PK: go back to clustered index to fetch full row (bookmark lookup) If 1M rows match: 1M random lookups in clustered index = SLOW. (Index inefficient if high selectivity) Covering index (eliminate step 2): CREATE INDEX idx_status_covering ON orders(status, user_id, amount); SELECT user_id, amount FROM orders WHERE status = 'PENDING' ALL required columns in index → no clustered index lookup. "Using index" in EXPLAIN. BUFFER POOL — InnoDB's Main Cache: innodb_buffer_pool_size (default: 128MB — always too small for production) Rule: set to 70-80% of total RAM. 16GB RAM → 12GB buffer pool. What's cached: data pages (16KB blocks), index pages, change buffer, undo logs. Eviction: LRU algorithm (modified: young list + old list for scan resistance). New page enters "old sublist" (3/8 of buffer pool). Accessed again → promoted to "young sublist". Large table scan: pages stay in old sublist → don't evict hot working set. Smart. Buffer pool hit rate: SHOW STATUS LIKE 'Innodb_buffer_pool_read_requests' / 'Innodb_buffer_pool_reads' Target: > 99% hit rate. If lower → increase buffer pool or add read replicas. GAP LOCKS — Why MySQL Prevents Phantom Reads: Scenario (REPEATABLE READ isolation): T1: SELECT * FROM orders WHERE amount BETWEEN 100 AND 200 FOR UPDATE T2: INSERT INTO orders (amount=150) → BLOCKS T1 locks the GAP between 100 and 200, not just existing rows. Prevents T2 from inserting a row that would appear in T1's range on re-read. Next-key lock = record lock + gap lock before it. Deadlock cause: T1 holds gap lock on (100,200). T2 holds gap lock on (150,300). Both want each other's gaps. Fix: use READ COMMITTED isolation (no gap locks, but allows phantom reads). Or: shorter transactions. Or: ORDER BY id in all transactions. MYSQL REPLICATION — GTID MODE: Binlog: record of all changes. 3 formats: STATEMENT: logs SQL query. Compact. Risk: non-deterministic functions (NOW(), UUID()) give different results on replica. ROW: logs actual row values changed. Larger but safe. Default in MySQL 8. MIXED: statement when safe, row when not. GTID (Global Transaction Identifier): server_uuid:transaction_id (e.g., "abc-123:5") Replica: tracks which GTIDs applied. On failover: new primary knows EXACTLY which transactions replica is missing. No more: binlog file + position tracking (error-prone, complex failover). Semi-synchronous replication: Primary: wait for at least 1 replica to confirm binlog received before ack to client. Not "applied" — just received. Not truly synchronous. Prevents data loss on primary crash. Replica can be promoted with guaranteed receipt of all transactions.

SHARD KEY DESIGN + AGGREGATION PIPELINE OPTIMIZATION
SHARD KEY — THE MOST IMPORTANT MONGODB DECISION (immutable after set): BAD: _id (ObjectID) — random distribution. All writes spread perfectly but no locality. Query WHERE userId=123 → scatter-gather to ALL shards → expensive. BAD monotonic (_id timestamp, auto-increment) — all writes go to ONE shard (last chunk). "Hotspot" shard: all writes, all new data. Others idle. GOOD: {userId: 1} — all documents for a user on same shard. Single-shard queries. Good cardinality. But: "celebrity problem" — famous user has 100M docs → 1 shard overloaded. BEST: Compound hashed shard key — {userId: "hashed", date: 1} Hashed userId: distributes writes evenly (no hotspot). date component: within a userId, documents sorted by date (range queries efficient within shard). Shard key rules: High cardinality (many unique values). Low frequency (no super-frequent values). Non-monotonic. Include fields in your most common WHERE clauses → targeted queries (vs scatter-gather). AGGREGATION PIPELINE PERFORMANCE: db.orders.aggregate([ { $match: { status: "COMPLETED", created_at: { $gte: ISODate("2024-01-01") } } }, ← FIRST. Uses index. Reduces docs. { $lookup: { from: "users", localField: "userId", foreignField: "_id", as: "user" } }, { $unwind: "$user" }, { $group: { _id: "$user.country", total: { $sum: "$amount" } } }, { $sort: { total: -1 } }, { $limit: 10 } ]) Optimization rules: 1. $match FIRST — filter early, process fewer docs through pipeline. Uses indexes. 2. $project after $match — reduce document size before expensive stages. 3. $lookup = MongoDB's JOIN. Expensive. Ensure foreign field is indexed. Unsharded: single-shard lookup. Sharded: $lookup broadcasts to all shards → slow. 4. allowDiskUse: true — if $group or $sort exceed 100MB memory limit → spill to disk. EMBED vs REFERENCE: EMBED: user document contains array of addresses { _id:1, name:"Alice", addresses:[{city:"NYC",...},{city:"LA",...}] } ✓ Single read gets everything. ✓ Atomic update within one document. ✗ Document grows without bound. ✗ 16MB document size limit. ✗ Cannot query across embeddings. Use when: ONE-TO-FEW relationship. Data always accessed together. No shared access. REFERENCE: order document has userId field { _id: order123, userId: user456, amount: 99 } → lookup users collection separately ✓ Unlimited size. ✓ Independent updates. ✓ Normalized. ✗ Requires $lookup (JOIN) on read. ✗ No atomic cross-collection updates without transactions. Use when: ONE-TO-MANY or MANY-TO-MANY. Data accessed independently. Large sub-arrays.

CASSANDRA INTERNALS — Write Path, LSM Tree, Tombstones & Anti-Patterns
WRITE PATH (why Cassandra writes so fast): Client → Coordinator node → determine responsible nodes (consistent hashing ring) Coordinator sends write to N replica nodes simultaneously (based on replication factor) Each node write path: 1. Commit log (WAL): append-only. Fsync. Crash recovery guarantee. 2. Memtable: in-memory sorted structure. Fast write. 3. Return ack to coordinator (when enough replicas ack based on consistency level) Background: when memtable reaches threshold → flush to SSTable (disk, immutable) SSTables never modified. Updates = new SSTable entry with higher timestamp. Reads: check memtable + bloom filter + SSTable cache + potentially multiple SSTables. Read amplification: N SSTables → read N, merge (latest timestamp wins). LSM TREE COMPACTION: Problem: 10 SSTables for same partition key → reads check all 10. Slow. Compaction: merge multiple SSTables into one. Remove obsolete versions. Remove tombstones. Strategies: STCS (Size-Tiered): merge similar-sized SSTables. Low write amplification. High space amplification. Default. LCS (Leveled): maintained in levels. L1: 1 SSTable, L2: 10, L3: 100 SSTables. Low space amplification. Read: at most 1 SSTable per level = fast reads. High write amplification. TWCS (Time-Window): for time-series. Compact within time window only. Cold windows untouched. DATA MODELLING — QUERY-FIRST DESIGN (opposite of RDBMS): RDBMS: normalize first, query with JOINs. Cassandra: design tables for specific queries. No JOINs. No ALLOW FILTERING. Use case: "Get all messages for a conversation, sorted by time" Table design: CREATE TABLE messages ( conversation_id UUID, ← PARTITION KEY: determines node sent_at TIMESTAMP, ← CLUSTERING KEY: sort order within partition message_id UUID, sender_id UUID, content TEXT, PRIMARY KEY (conversation_id, sent_at, message_id) ) WITH CLUSTERING ORDER BY (sent_at DESC); Query: SELECT * FROM messages WHERE conversation_id = ? AND sent_at > ? LIMIT 20 ✓ Hits exactly the right partition. Reads are sorted and bounded. O(1) partition lookup. ANTI-PATTERNS: ALLOW FILTERING: scans ALL partitions (full table scan like Cassandra version of SELECT *). Always wrong in prod. Secondary index: scatter-gather query to ALL nodes. Avoid. Create separate table instead. Unbounded partition growth: messages partitioned ONLY by conversation_id. 1 chat = billions of rows = 1 HUGE partition. Fix: bucket by time: PRIMARY KEY ((conversation_id, bucket_week), sent_at) TOMBSTONES — The Production Killer: Cassandra DELETE = write a tombstone (timestamp marker). Actual data NOT deleted immediately. Tombstone = SSTable entry with no data, just a deletion timestamp. Read: Cassandra reads ALL data + tombstones, filters tombstones, returns live data. 1M tombstones in a partition → read must process 1M tombstones → ReadTimeout. gc_grace_seconds (default: 10 days): tombstone kept for 10 days to prevent "zombie" resurrection on replica sync. After gc_grace: compaction removes tombstones permanently. Problem pattern: DELETE + re-INSERT at same key rapidly → accumulate tombstones. Gaming leaderboards with frequent score updates = tombstone bomb. Fix: design updates as INSERTS to new clustering keys (append-only). Use TTL instead of DELETE. Monitoring: nodetool tpstats → check TombstoneReadAborted counter.

DATA STRUCTURE DECISION GUIDE: String: Simple values, counters, session tokens, JSON blobs, rate limit counters INCR user:1000:pageviews → atomic increment. O(1). "Followers you might also follow." Sorted Set (ZSET): Leaderboards, rate limiting with sliding windows, priority queues ZADD leaderboard 9500 "user:1000" → score=9500 ZREVRANK leaderboard "user:1000" → O(log N) rank lookup ZRANGEBYSCORE rate:user:1000 (now-60s) now → sliding window rate limiting HyperLogLog: Approximate DISTINCT count. 12KB fixed memory. Error rate: 0.81%. PFADD daily_visitors:2024-04-19 user:1000 user:1001 ... PFCOUNT daily_visitors:2024-04-19 → "~1,234,567 unique users" Perfect for: daily active users, unique impressions (exact count NOT needed). Streams: Persistent log. Append-only. Consumer groups. Like mini-Kafka. XADD events:orders * userId 1000 action "checkout" amount 99 XREADGROUP GROUP analytics consumer1 COUNT 10 STREAMS events:orders > XACK events:orders analytics messageId → acknowledge processed REDIS PERSISTENCE — AOF vs RDB: RDB (Redis Database): Point-in-time snapshot. BGSAVE → fork() child writes snapshot to disk. ✓ Small file, fast restore. ✗ Last snapshot may be 5 min old. Risk: data loss on crash. Use: backup, DR recovery. Acceptable data loss. AOF (Append-Only File): log every write command. appendfsync: always: fsync after every write. Zero data loss. 2-3× slower. everysec: fsync every 1 second. Max 1 second data loss. Best balance. no: OS decides when to flush. Fast. Up to 30s data loss. Hybrid (recommended): RDB for fast restart + AOF for durability. Best of both. AOF rewrite: compacts logs (removes redundant operations). Background process. REDIS CLUSTER — 16384 HASH SLOTS: Cluster splits keyspace into 16384 slots. Each master owns a slot range. Key routing: HASH_SLOT = CRC16(key) % 16384 Key "user:1000" → slot 3272 → assigned to Master 2 → client connects to Master 2 directly. 3 masters (M1, M2, M3), each with a replica: M1: slots 0-5460, M2: slots 5461-10922, M3: slots 10923-16383 Hash tags {}: MGET user:{1000}:profile user:{1000}:orders Both keys route to same slot (slot of "1000"). Multi-key ops work. Without {}: might be on different nodes → CROSSSLOT error. Failover: M1 dies → its replica promoted via Raft-like election (CLUSTER_NODE_TIMEOUT: 15s default) Split-brain: cluster requires majority of masters reachable. <n/2+1 masters = stop accepting writes. EVICTION POLICIES (maxmemory reached): noeviction (default): reject new writes. Bad for cache use case. allkeys-lru: evict least recently used key from ALL keys. Best general-purpose cache. volatile-lru: evict LRU from keys WITH TTL set. Preserve TTL-less keys. allkeys-lfu: evict least frequently used. Better for skewed access patterns. volatile-ttl: evict key with smallest TTL. Evict about-to-expire keys first. Recommendation: allkeys-lru for pure cache. volatile-lru if some keys must never be evicted.

INVERTED INDEX + SHARD SIZING + QUERY DSL
INVERTED INDEX (How "search for 'apple'" finds documents in 5ms): Documents indexed: Doc 1: "Apple makes great iPhones" Doc 2: "Apple iPhone battery lasts all day" Doc 3: "Samsung Galaxy vs iPhone" Inverted index built at index time (tokenize → normalize → store): apple → [Doc1, Doc2] lasts → [Doc2] makes → [Doc1] all → [Doc2] great → [Doc1] day → [Doc2] iphone → [Doc1, Doc2, Doc3] samsung → [Doc3] battery → [Doc2] galaxy → [Doc3] Query: "apple iphone battery" apple → [Doc1, Doc2] ∩ iphone → [Doc1, Doc2, Doc3] ∩ battery → [Doc2] BM25 scoring: Doc2 gets highest score (all 3 terms, term frequency, document length) Result: [Doc2, Doc1, Doc3] sorted by relevance score. Lucene segment: each segment is a mini-inverted index. Immutable once written. Merge: background merging of segments. Deleted docs: marked in .del file, filtered at query time. SHARD SIZING — The Most Common ES Mistake: Each shard = one Lucene index. Has overhead: JVM heap, file handles, housekeeping. Rule of thumb: shard size 10-50GB. Never <1GB, never >50GB. Bad: 1TB index with 1000 shards of 1GB each. 1000 threads coordinating search = slow. Bad: 1TB index with 1 shard of 1TB. Can't distribute across nodes. Can't parallelize. Good: 1TB index with 20 shards of 50GB each. Distribute 4 shards per 5 nodes. Calculate: (data_size_GB / target_shard_size_GB) = shard_count Add replicas for HA: 1 replica = double node count needed. Shards are immutable after index creation. Plan carefully. Too many small shards: each node has too many shards → GC pressure → heap exhausted. Monitor: GET /_cat/indices?v → check shard count vs docs count vs store.size QUERY DSL — FILTER vs QUERY context (critical performance difference): FILTER context: binary yes/no. Is status = 'active'? Cached. Zero relevance scoring. QUERY context: relevance scored. BM25 calculated. Not cached. B+ TREE vs LSM TREE — Core Trade-Off Behind Every Major DB Choice

B+ TREE (PostgreSQL, MySQL InnoDB, SQLite):
  Structure: balanced tree. Leaf pages: sorted data. Internal pages: routing keys.
  Read:  O(log N). Tree traversal → reach leaf page. 3-4 disk reads for 1B rows.
  Write: find leaf page → insert → rebalance if needed (split pages).
         Random I/O. Page splits → fragmentation → bloat over time.
         One disk write per record (to specific page location).
  Best for: read-heavy workloads. Low write amplification per record.
            Range queries (sorted leaf pages). Strong consistency.
  Used by: PostgreSQL, MySQL, SQLite, Oracle

LSM TREE (Cassandra, RocksDB, LevelDB, InfluxDB, HBase):
  Structure: memtable (in-memory) → L0 SSTables → L1 → L2 (ever-larger levels).
  Write:  ALWAYS sequential append. Memtable → flush to SSTable → compact.
          No random I/O. 10× higher write throughput than B-tree for same hardware.
  Read:   Check memtable + each SSTable (newest first). Bloom filters skip irrelevant SSTables.
          Write amplification: compaction rewrites data 10-30× over its lifetime.
  Best for: write-heavy workloads. Append-only time-series. High-frequency updates.
  Used by: Cassandra, RocksDB (Kafka, TiKV), InfluxDB, HBase, LevelDB

ACID PROPERTIES — Each One Matters:
  Atomicity:   Transaction commits entirely or not at all. No partial writes.
               Implemented by: WAL + rollback log (undo log in InnoDB, undo log in PostgreSQL).
  Consistency: DB moves from one valid state to another. Constraints satisfied after commit.
               Implemented by: constraint checking before commit, foreign key validation.
  Isolation:   Concurrent transactions don't interfere. Levels: RC, RR, Serializable.
               Implemented by: MVCC (PostgreSQL), locks (MySQL), SSI (Serializable Snapshot Isolation).
  Durability:  Committed transactions survive crashes.
               Implemented by: WAL + fsync. "If fsync is disabled → durability is a lie."

CAP THEOREM (for distributed databases):
  Consistency + Availability + Partition Tolerance: pick 2 (P is always happening in distributed systems).
  So really: CP (consistency + partition tolerance) vs AP (availability + partition tolerance).
  CP: PostgreSQL (leader fails → elect new one; no writes during election = unavailable but consistent)
      HBase, Zookeeper (stale reads refused)
  AP: Cassandra (always accepts writes; eventual consistency; may read stale data)
      CouchDB, DynamoDB (with eventual consistency)
  CA: Not possible in distributed system (partitions happen)
  Modern nuance: spectrum not binary. Spanner: "effectively CA" via TrueTime atomic clocks.

CONNECTION POOLING — Why It Matters:
  PostgreSQL: each connection = OS process (~5-10MB RAM, fork() overhead)
  10,000 app instances × 10 connections each = 100,000 connections × 5MB = 500GB RAM just for connections.
  PgBouncer (transaction mode): map 100,000 app connections → 100 actual DB connections.
  Idle connections still consume: pgstat, bgwriter, checkpoint coordination overhead.
  MySQL: thread-per-connection (lighter than pg processes). Still: 10K threads → context switches.
  Always: set connection pool size = num_cores × (1 + wait_time/compute_time). Typical: 10-20 per service.

PACELC + CONSISTENCY MODELS + DISTRIBUTED TRANSACTIONS + SCHEMA MIGRATIONS
PACELC THEOREM — CAP's Successor (What Production Systems Actually Use): CAP theorem: choose 2 of Consistency, Availability, Partition Tolerance. Problem: partitions are rare. What about NORMAL operation — no partition? CAP says nothing. PACELC (Daniel Abadi, 2012): If Partition (P): trade-off between Availability (A) and Consistency (C). [same as CAP] Else (E): trade-off between Latency (L) and Consistency (C). Read as: "PAC, ELC" — two separate trade-offs depending on whether partition is happening. Real systems: DynamoDB: PA / EL — highly available during partition (AP). Low latency else. Eventual consistency. Cassandra: PA / EL — same as DynamoDB. Tunable but defaults to availability + low latency. Spanner: PC / EC — consistent during partition (CP). Strong consistency always. Pays latency cost. CockroachDB: PC / EC — same as Spanner. Serializable everywhere. Latency: ~10ms (cross-region: 100ms). MySQL (single master): PC / EC — consistent, but latency spike on writes (synchronous). MongoDB (majority): PC / EC — with w:majority read+write, consistent and low latency on single region. Why this matters in interviews: "Why is DynamoDB eventually consistent by default?" → It optimizes for EL (low latency in normal operation). "Why is Spanner's write latency 10ms minimum?" → It optimizes for EC (always consistent, pays latency). "We need global consistency AND low latency. What DB?" → Impossible. PACELC says you must choose. LINEARIZABILITY vs SERIALIZABILITY — The Most Confused Concepts: These are NOT the same. 80% of senior engineers confuse them. Know the difference cold. SERIALIZABILITY (transaction isolation): Multi-transaction property. Multiple concurrent transactions appear to execute one at a time. The ORDER they appear to execute may not match real time order — just must be SOME valid serial order. Example: T1 and T2 run concurrently. Serializable = result is equivalent to T1→T2 OR T2→T1. The chosen serial order can be different from the actual clock order. "Strongest isolation level in SQL" — Serializable isolation. LINEARIZABILITY (single-operation consistency): Single-operation property. Each operation appears to take effect at some point between its start and end. THE ORDER must respect real wall-clock time. If Write completes BEFORE Read starts → Read must see the write. "Real-time ordering guarantee." Sometimes called "strong consistency" or "external consistency." STRICT SERIALIZABILITY = Serializability + Linearizability. The gold standard. Spanner provides strict serializability (via TrueTime commit wait). Most databases claiming "serializable" isolation only provide Serializability (not Linearizable reads). Practical example: Op1: Write(x=1) starts at T=0, completes at T=10ms. Op2: Read(x) starts at T=15ms. Linearizable: Op2 MUST return x=1. Write completed before Read started. Serializable only: Op2 MIGHT return x=0 if placed before Op1 in the serial order. Still valid! "Eventual Consistency" means: NEITHER. No order guarantee. Read might return stale data for arbitrary time. Interview answer: "PostgreSQL Serializable isolation = serializable. Not linearizable (reads from replica may be stale). Spanner = strictly serializable. Redis single-node = linearizable (single-threaded). Redis Cluster = not linearizable." TWO-PHASE COMMIT (2PC) vs SAGA PATTERN — Distributed Transactions: Problem: operation spans multiple databases (order-service DB + inventory-service DB + payment-service DB). You need atomicity: either ALL succeed or ALL fail. No partial state. TWO-PHASE COMMIT (2PC): Phase 1 (Prepare): Coordinator sends PREPARE to all participants. Each participant: lock resources. Write to WAL. Reply READY or ABORT. Phase 2 (Commit): If ALL replied READY → Coordinator sends COMMIT. All flush and unlock. If ANY replied ABORT → Coordinator sends ROLLBACK. All undo. Problem: Coordinator failure after Phase 1 COMMIT sent to some but not all participants: Some participants committed. Others waiting. Blocking protocol. Cannot make progress without coordinator. Recovery: coordinator restarts, re-reads log, sends COMMIT to remaining. But outage window exists. When to use 2PC: Same data store (MySQL XA, PostgreSQL distributed with pg_dist_transaction). Short transactions (OLTP). Low latency tolerance. Need synchronous guarantee. NOT for: cross-service microservices (your services shouldn't share DB connection context). SAGA PATTERN (for microservices): Sequence of local transactions. Each local transaction publishes an event → triggers next service. If step N fails → compensating transactions for steps 1..N-1 (undo actions). Order Saga example: Step 1: OrderService creates order (status=PENDING). Publishes ORDER_CREATED event. Step 2: InventoryService reserves item. Publishes RESERVATION_CONFIRMED event. Step 3: PaymentService charges card. Publishes PAYMENT_CHARGED event. Step 4: OrderService updates order (status=CONFIRMED). If PaymentService fails: Compensating: PaymentService publishes PAYMENT_FAILED → InventoryService releases reservation → OrderService marks order CANCELLED. Choreography (event-driven): each service listens to events, reacts. Simple. No coordinator. Can be hard to trace. Orchestration (coordinator): central Saga orchestrator sends commands to each service. Easier to trace. More coupling. 2PC vs Saga trade-off: 2PC: synchronous, strong consistency, blocking, tight coupling, simpler logic, same tech stack. Saga: async, eventual consistency, non-blocking, loose coupling, complex compensating logic, any tech stack. FAANG default: Saga for microservices. 2PC: only within single database or tightly coupled systems. ZERO-DOWNTIME SCHEMA MIGRATIONS AT FAANG SCALE: Problem: ALTER TABLE orders ADD COLUMN delivery_time TIMESTAMP; PostgreSQL: acquires ACCESS EXCLUSIVE LOCK. All reads/writes blocked. 100M rows = 30 minutes offline. MySQL: some ALTERs copy full table. Production dead for minutes. SOLUTION: Expand-Contract Pattern (Blue-Green for schema) Phase 1: EXPAND (backward-compatible addition) Deploy: add new column delivery_time NULLABLE. Old code writes NULLs (doesn't know about it). New code: writes delivery_time on every new row. Both old and new code work. No downtime. Online tool: gh-ost (GitHub) or pt-online-schema-change (Percona) — copy table with new schema, apply binlog changes live, atomic rename at end. Zero-lock migration. Phase 2: BACKFILL Background job: UPDATE orders SET delivery_time = shipped_at WHERE delivery_time IS NULL AND id BETWEEN ? AND ? Batched updates (1000 rows at a time). Throttled. No table lock. Runs for hours/days. Phase 3: SWITCH All code now writes delivery_time. All rows backfilled. Add NOT NULL constraint. Online: CREATE INDEX CONCURRENTLY ON orders(delivery_time); ← PostgreSQL concurrent index, no lock Phase 4: CONTRACT (backward-incompatible removal, later) Deploy code that no longer reads old column. Drop column. Zero-downtime. Rule: NEVER run ALTER TABLE that locks on a large table in production. Always: gh-ost / pt-osc / Flyway (with careful migration) / Liquibase. Most dangerous migration: adding NOT NULL column without default to a large table. Safest: always add columns as NULLABLE first, backfill, then add constraint. READ CONSISTENCY PATTERNS — What Your Application Code Must Handle: After a write to PostgreSQL primary, read from replica. Replica 200ms behind. Read sees old value. This is NOT a bug in the DB. It's expected. Your application must handle it. READ-YOUR-OWN-WRITES: Problem: User updates profile photo. Reads back their profile from replica → sees old photo. Confusing. Solution 1: After write, read from primary (not replica). Expensive but simple. Solution 2: After write, remember "I wrote at T=X" in session. If replica lag > T → read from primary. Solution 3: Sticky sessions to replica that received the write. Route reads to same node for 5s. MONOTONIC READS: Problem: User reads message at T=0 (sees 5 messages). Consistent hash(userId) → replica selection. CONSISTENT PREFIX READS: Problem: Conversation with 2 messages. Replica A has message 1. Replica B has message 2. User reads message 2 before replaying message 1 → sees answer before question. Solution: causal tokens. Write Token includes logical timestamp. Reads wait until replica catchup >= token. CAUSAL CONSISTENCY: "If I see event B, I must have seen event A (which caused B)." MongoDB (causally consistent sessions): client sends clusterTime token on each read. Consistency Spectrum
Strict Serializable (Spanner): real-time ordered + one-at-a-time. Serializable (PostgreSQL SERIALIZABLE): one-at-a-time but not real-time ordered. Linearizable (single-key): real-time ordered per key. Causal (MongoDB sessions): cause-before-effect. Eventual (Cassandra default): will converge, no guarantees.
🔄
2PC Failure Modes
Coordinator crash between Phase 1 COMMIT sent + Phase 2: blocking — some participants committed, others uncertain. Cannot proceed. Requires coordinator restart + log replay. Use 3PC or Paxos-based protocols to avoid. In practice: use Saga or accept 2PC limitations to single-DB transactions.
📐
Saga Failure Handling
Compensating transaction must be idempotent: if it fails mid-way and retries, it must produce same result. Design: use correlation IDs. Log saga state in DB. Idempotency key: PaymentService checks "did I already charge idempotencyKey=order123?" before charging. Prevents double-charge on retry.
🏗️
DB-per-Service Pattern
Each microservice owns its DB. NO direct cross-service DB queries. Only APIs. Prevents tight coupling. Challenge: cross-service queries require API composition or event-driven denormalization. CQRS: separate read model (materialized view) from write model. Read model built by consuming events from other services.
🔐
Data at Rest Security
Transparent Data Encryption (TDE): PostgreSQL pgcrypto, MySQL InnoDB TDE — encrypts data files. App doesn't change. Column-level encryption: encrypt specific sensitive columns (SSN, credit card). Only app decrypts. DB stores ciphertext. Row-level Security: PostgreSQL RLS — DB enforces tenant isolation in WHERE clause.
🌍
Data Residency & GDPR
EU users' data must stay in EU. Solution: geo-partitioning. CockroachDB: PARTITION BY LIST (region) CONFIGURE ZONE constraints='region=EU'. Data pinned to EU nodes. Spanner: similar. DynamoDB Global Tables: data replicated globally — need custom routing to ensure EU writes go to EU table. Challenge: joins across regions = cross-region latency.

⚡ SQL & DB Diagnostic Cheatsheet
Find slow queries (PG)
SELECT query, calls, mean_exec_time, total_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 20; — must enable pg_stat_statements extension.
Find blocking (PG)
SELECT pid, now()-query_start AS duration, query, wait_event_type, wait_event FROM pg_stat_activity WHERE state != 'idle' ORDER BY duration DESC; then SELECT pg_blocking_pids(blocking_pid);
Table bloat (PG)
SELECT relname, n_dead_tup, n_live_tup, ROUND(n_dead_tup::numeric/NULLIF(n_live_tup,0)*100,1) AS dead_pct FROM pg_stat_user_tables ORDER BY n_dead_tup DESC LIMIT 10;
Index usage (PG)
SELECT indexrelname, idx_scan, idx_tup_read FROM pg_stat_user_indexes WHERE schemaname='public' ORDER BY idx_scan ASC; — indexes with 0 idx_scan are unused. Consider dropping.
MySQL index usage
EXPLAIN FORMAT=JSON SELECT ...; — look for "type":"ALL" (full scan). SHOW INDEX FROM orders; — cardinality. SELECT * FROM sys.schema_unused_indexes; — unused indexes.
Redis diagnostics
INFO memory — used_memory, maxmemory, evicted_keys. INFO stats — keyspace_hits vs keyspace_misses → cache hit rate. MEMORY DOCTOR — Redis 4+ memory analysis. redis-cli --latency — latency baseline.
Cassandra nodetool
nodetool status — cluster health. nodetool tpstats — thread pool stats (dropped messages). nodetool tablestats keyspace.table — SSTable count (high = need compaction). nodetool compactionstats.
ES cluster health
GET /_cluster/health?pretty — status green/yellow/red. GET /_cat/indices?v&s=store.size:desc — largest indexes. GET /_nodes/stats/jvm — heap pressure. GET /_cluster/pending_tasks — pending operations.
